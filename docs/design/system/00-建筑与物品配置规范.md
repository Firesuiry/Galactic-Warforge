# 建筑与物品配置规范（V1）

## 1. 目标
- 为后续系统实现提供统一、可验证、可扩展的配置方案。
- 建筑采用“每个建筑一个配置文件”。
- 物品采用“每个物品一个配置文件”。
- 配置只描述静态定义，不直接存放运行时状态。

## 2. 配置目录结构

```text
config/
  defs/
    manifest.yaml
    buildings/
      collect/
        mining_machine.yaml
        advanced_mining_machine.yaml
      transport/
        conveyor_belt_mk1.yaml
      ...
    items/
      raw/
        iron_ore.yaml
      intermediate/
        iron_ingot.yaml
      fluid/
        crude_oil.yaml
      ...
```

规则：
- `buildings/` 下每个文件只允许定义 1 个建筑。
- `items/` 下每个文件只允许定义 1 个物品。
- 文件名必须等于对象 `id`：`<id>.yaml`。
- 目录按分类组织，但分类以文件内字段为准，目录仅用于管理。

## 3. 基础约定
- 格式：YAML（UTF-8，空格缩进 2）。
- 顶层版本：每个文件都必须带 `spec_version`（当前 `v1`）。
- 标识符规范：
  - `id` 只允许小写字母、数字、下划线：`^[a-z0-9_]+$`。
  - 一旦发布不可复用、不可重命名（需新建 id 并迁移引用）。
- 单位规范：
  - 时间：`tick`
  - 功率：`mw`
  - 能量：`mj`
  - 物流吞吐：`items_per_tick`
  - 液体吞吐：`units_per_tick`
  - 堆叠体积：`stack_space`

## 4. 清单文件（manifest）

`config/defs/manifest.yaml` 管理加载入口与一致性校验。

必填字段：
- `spec_version`: `v1`
- `load_order`:
  - `items_glob`: `defs/items/**/*.yaml`
  - `buildings_glob`: `defs/buildings/**/*.yaml`
- `integrity`:
  - `require_unique_ids`: `true`
  - `require_reference_resolved`: `true`
  - `strict_unknown_fields`: `true`
- `runtime_policy`:
  - `hot_reload`: `false`（默认）
  - `fail_fast_on_boot`: `true`

## 5. 建筑配置规则

### 5.1 建筑文件最小结构

```yaml
spec_version: v1
kind: building
id: mining_machine
name: 采矿机
category: collect
subcategory: mining
```

### 5.2 建筑完整字段（V1）

```yaml
spec_version: v1
kind: building
id: mining_machine
name: 采矿机
category: collect
subcategory: mining
tech_tier: 1

footprint:
  width: 2
  height: 2
  rotatable: true

placement:
  allowed_terrain: [buildable_land]
  require_resource_node: true
  resource_node_types: [ore_vein]
  near_equator_only: false
  orbit_only: false
  gas_giant_only: false

build:
  cost:
    - item_id: iron_ingot
      count: 4
  build_ticks: 60
  dismantle_refund_ratio: 1.0

unlock:
  required_tech: [electromagnetism]
  required_buildings: []
  required_items: []

power:
  mode: consumer              # consumer | producer | converter | none
  idle_mw: 0.2
  work_mw: 0.8
  peak_mw: 1.0
  fuel_types: []              # 仅 mode=producer/converter 时可用
  generation_mw: 0

io:
  input_ports: []
  output_ports:
    - port: out_1
      accepts: [ore_item_group]
      throughput_items_per_tick: 1.0
      buffer_capacity: 50
      priority: normal         # high | normal | low
  fluid_input_ports: []
  fluid_output_ports: []

operation:
  modes: [default]
  default_mode: default
  cycle_ticks: 60
  recipe_pool: []              # 生产建筑填 recipe_id 列表
  supports_proliferation: false
  supports_overclock: false

combat:
  enabled: false
  hp: 0
  shield: 0
  resistances: []
  attack: null

visibility:
  detectable_level: 1
  scout_radius: 0

events:
  emits:
    - building_started
    - building_stopped
    - building_blocked
    - building_no_power

upgrade:
  next_ids: [advanced_mining_machine]
  inherit_runtime_buffer: true
  upgrade_ticks: 120
```

### 5.3 建筑字段约束
- `kind` 必须为 `building`。
- `id` 全局唯一，且与文件名一致。
- `footprint.width/height` 必须大于 0。
- `placement.require_resource_node=true` 时，`resource_node_types` 不可为空。
- `power.mode=producer` 时，`generation_mw` 必须大于 0。
- `io` 中每个端口名在本建筑内唯一。
- `upgrade.next_ids` 中每个目标建筑必须存在。
- `build.cost[].item_id` 必须引用已存在物品。

## 6. 物品配置规则

### 6.1 物品文件最小结构

```yaml
spec_version: v1
kind: item
id: iron_ore
name: 铁矿石
form: solid
```

### 6.2 物品完整字段（V1）

```yaml
spec_version: v1
kind: item
id: iron_ore
name: 铁矿石
category: raw
form: solid                    # solid | liquid | gas
stack_space: 1

storage:
  stack_size: 100
  can_store_in_depot: true
  can_store_in_tank: false
  can_store_in_station: true

transport:
  belt_transportable: true
  sorter_transportable: true
  station_transportable: true
  drone_transportable: true
  vessel_transportable: true

fuel:
  is_fuel: false
  fuel_value_mj: 0
  fuel_type: none              # none | chemical | hydrogen | antimatter

combat:
  is_ammo: false
  ammo_type: none
  base_damage: 0

production:
  produced_by: []              # recipe_id 列表
  consumed_by: []              # recipe_id 列表

tags: [ore]
```

### 6.3 物品字段约束
- `kind` 必须为 `item`。
- `id` 全局唯一，且与文件名一致。
- `form=solid` 时 `can_store_in_tank` 必须为 `false`。
- `form=liquid/gas` 时 `belt_transportable` 默认应为 `false`（特殊物品需显式说明）。
- `is_fuel=true` 时 `fuel_value_mj` 必须大于 0。
- `is_ammo=true` 时 `ammo_type` 不可为 `none`。

## 7. 跨文件引用规则
- 建筑 `build.cost[].item_id` 必须引用物品 `id`。
- 建筑端口 `accepts` 可引用：
  - 具体 `item_id`
  - 物品组 `item_group`（组定义可在后续 `groups/` 扩展）
- 配方（后续独立 schema）引用建筑时，用 `building.id`。
- 引用解析失败时，系统启动失败（`fail_fast_on_boot=true`）。

## 8. 加载与校验流程
- 启动阶段按 `manifest.load_order` 加载：先物品，后建筑。
- 每个文件执行 schema 校验与语义校验。
- 完成全局校验：
  - id 唯一
  - 引用闭合
  - 升级链无非法环（允许同级分叉，不允许自循环）
- 校验通过后生成只读内存索引：
  - `building_by_id`
  - `item_by_id`
  - `building_by_category`
  - `item_by_tag`

## 9. 版本演进规则
- `spec_version` 不向后兼容时，升级为 `v2` 并保留旧加载器。
- 字段演进优先追加，不直接改语义；删除字段需走迁移期。
- 重大变更需要提供迁移脚本（从旧配置自动转换）。

## 10. 后续可扩展文件（预留）
- `config/defs/recipes/**/*.yaml`：配方定义。
- `config/defs/techs/**/*.yaml`：科技定义。
- `config/defs/enemies/**/*.yaml`：敌方单位定义。
- `config/defs/groups/**/*.yaml`：物品组与建筑组定义。

## 11. 实现建议
- 服务端实现顺序：
  1. `manifest` + 物品加载器
  2. 建筑加载器与引用校验器
  3. 配置热更新开关（默认关闭）
  4. 配置变更 diff 与回放一致性检测
- 测试最小集：
  - 文件名与 `id` 不一致应报错
  - 建筑引用不存在物品应报错
  - 端口字段非法应报错
  - 升级链循环应报错
  - 合法最小配置应可启动
