# T090 戴森球计划剩余建筑、科技树与玩法闭环收口

## 问题背景

- 2026-04-03 对当前工作区代码做了两套隔离实测：
  - 默认开局：`config-dev` 派生配置，端口 `18101`
  - 戴森中后期验证局：`config-midgame` 派生配置 + `map-midgame.yaml`，端口 `18102`
- 本任务只关心“当前项目宣称对标《戴森球计划》的建筑、科技树与玩法闭环”是否真正可玩。
- 以下明确设计差异不计入缺陷范围：
  - 上帝视角 + 执行体，而不是原版机甲直操
  - 多人 / 阵营对抗服务端
  - API 驱动、无渲染
  - 行星为 2D 平面网格

## 本轮已确认可用的部分

- `solar_sail_orbit` / `ray_receiver` / `vertical_launching` / `gas_giants` 相关主线科技、物品与配方已经进入 `/catalog`
- `transfer` 已能把背包物品装入建筑本地存储
- `em_rail_ejector` 可通过 `transfer + launch_solar_sail` 公开命令成功发射太阳帆
- `vertical_launching_silo` 可通过 `transfer + launch_rocket` 公开命令成功发射火箭
- `build_dyson_node` / `build_dyson_frame` / `build_dyson_shell` 可执行
- `ray_receiver` 在有太阳帆 / 戴森结构输出时可产出 `critical_photon`

这些能力已经不属于“完全没做”，后续任务不要再把它们按旧问题重复记录为缺失。

## 仍未实现 / 新发现的问题

### 问题 1：`orbital_collector` 能建出来，但当前实际不能运行采集

#### 复现

1. 启动 `18102` 隔离 midgame 局并登录 `p1`
2. 在气态行星 `planet-1-2` 上执行：
   - `move u-2 4 2`
   - `build 6 2 orbital_collector`
   - 补建风机与电塔：
     - `build 1 0 wind_turbine`
     - `build 2 0 wind_turbine`
     - `build 3 1 wind_turbine`
     - `build 4 1 wind_turbine`
     - `build 5 1 wind_turbine`
     - `build 4 3 tesla_tower`
     - `build 7 1 tesla_tower`
3. 查询：
   - `inspect planet-1-2 building b-20`

#### 实际现象

- `orbital_collector` 建造成功，但长期处于：
  - `state = no_power`
  - `state_reason = power_out_of_range`
- `server/internal/model/building_runtime.go` 中 `orbital_collector` 的运行时定义只有：
  - `EnergyConsume: 4`
  - `Energy` / `Orbital` 模块
  - **没有任何 `connection_points`**
- `server/internal/gamecore/orbital_collector_settlement.go` 只会在 `building.Runtime.State == running` 时产出 `hydrogen/deuterium`
- 实测下该建筑始终进不了 `running`，因此采集闭环没有真正跑起来

#### 影响

- `gas_giants` 科技虽然可研究 / 预解锁，`orbital_collector` 也能摆出来，但气态行星采集玩法没有真正闭环
- 星际物流链里最关键的“气态行星 -> 氢 / 氘”来源仍不可用

### 问题 2：一批 DSP 建筑仍停留在“有定义但不可建造”状态，未进入真实科技树与主线玩法

#### 已确认当前不可建造的建筑

- `advanced_mining_machine`
- `pile_sorter`
- `recomposing_assembler`
- `energy_exchanger`
- `jammer_tower`
- `sr_plasma_turret`
- `planetary_shield_generator`
- `self_evolution_lab`

#### 复现

在 midgame 局执行：

- `build 0 5 recomposing_assembler`
- `build 1 5 advanced_mining_machine`
- `build 2 5 energy_exchanger`

#### 实际现象

- 返回：
  - `VALIDATION_FAILED: building type not buildable: recomposing_assembler`
  - `VALIDATION_FAILED: building type not buildable: advanced_mining_machine`
  - `VALIDATION_FAILED: building type not buildable: energy_exchanger`
- `/catalog` 中这些建筑的 `buildable` 仍为 `false`
- 其中多数建筑当前没有任何科技解锁入口
- `self_evolution_lab` 虽然在隐藏科技 `dark_fog_matrix` 上有别名映射，但建筑本身仍然 `buildable=false`

#### 影响

- 参考文档中已经列出的 DSP 建筑并没有真正进入玩家可用建筑集
- 当前科技树覆盖与建筑覆盖仍不一致，玩家无法通过研究路线进入这些建筑玩法
- `docs/player/玩法指南.md` 5.4 里列出的这批“有定义，但不算主线可玩”的建筑，实际上多数仍属于“建不出来”

### 问题 3：科技树研究仍然是抽象研究点推进，没有矩阵实物消耗闭环

#### 复现

1. 启动默认开局隔离局 `18101`
2. 登录 `p1`
3. 在没有矩阵产线、没有矩阵库存展示入口的开局状态下直接执行：
   - `start_research electromagnetism`
4. 等待若干 tick

#### 实际现象

- `start_research electromagnetism` 直接成功排队
- 数秒后收到：
  - `research_completed ... tech_id=electromagnetism`
- `server/internal/gamecore/research.go` 当前明确写着：
  - “Cost is in matrix items, but we track as research points”
  - `calculateTechCost` 只把矩阵数量换算成点数
  - 研究推进过程中没有检查 / 扣除玩家矩阵物品

#### 影响

- 科技树虽然“可点完”，但不是 DSP 式的矩阵生产 -> 投入 -> 解锁闭环
- 中后期矩阵产线和科研物流的重要性被大幅削弱
- 当前“科技树已实现”只能算到定义与前置关系层，不能算到完整玩法层

### 问题 4：多星球经营与星际物流仍被 `active planet` 限死，没有真正形成 DSP 式多行星玩法

#### 复现

1. 启动 midgame 局 `18102`，当前 `active_planet_id = planet-1-2`
2. 查询：
   - `GET /world/planets/planet-1-1/runtime`
   - `GET /world/planets/planet-1-1/networks`

#### 实际现象

- 返回结果中：
  - `available = false`
  - `active_planet_id = planet-1-2`
- 代码层：
  - `server/internal/query/runtime.go`
  - `server/internal/query/networks.go`
  - 都在 `ws.PlanetID != planetID` 时直接返回不可用视图
- 当前没有公开玩家命令让你切换 `active planet`

#### 影响

- 即使已经有 `interstellar_logistics`、`gas_giants`、`orbital_collector`、星际站等定义
- 玩家仍然无法像 DSP 那样长期经营多颗行星，并把气态行星、矿星、主工厂星串成稳定网络
- 现有“星际物流”更接近单 active planet 下的收敛能力，不是完整多行星玩法

### 问题 5：`ray_receiver` 缺少公开模式切换入口，玩家无法切到纯发电 / 纯光子模式

#### 实际现象

- `server/internal/model/ray_receiver.go` 已支持：
  - `power`
  - `photon`
  - `hybrid`
- 但当前公开 API / CLI / 玩家指南中没有任何“切换射线接收站模式”的命令或入口
- 实测建出的 `ray_receiver` 默认以 `hybrid` 运行

#### 影响

- 当前玩家只能用默认模式，无法按 DSP 玩法在“直接供电”和“临界光子产出”之间切换
- `ray_receiver` 只算半实现：底层模型有了，公开玩法入口还没有

### 问题 6：`client-cli` 连续粘贴多条命令时会并发派发，导致回包顺序和 `request_id` 对不上输入顺序

#### 复现

在 midgame 局连续输入：

- `transfer b-21 solar_sail 2`
- `build_dyson_node sys-1 0 12 22 --orbit-radius 1.2`
- `launch_solar_sail b-21 --count 1`

#### 实际现象

- `ACCEPTED request_id=...` 的展示顺序与后续 `command_result.request_id` 的对应关系会错位
- 本轮实测中出现：
  - 输入第 1 条命令得到的 `ACCEPTED request_id`，却对应第 2 条命令的 `command_result`
  - 输入第 2 条命令得到的 `ACCEPTED request_id`，却对应第 1 条命令的 `command_result`
- `client-cli/src/repl.ts` 当前对 `line` 事件直接 `async` 处理，没有串行队列；连续粘贴时多条命令会并发发出

#### 影响

- 人工逐条敲命令问题不明显，但粘贴批量命令、脚本驱动或 AI 驱动试玩时会严重干扰定位
- 会把服务端事件与用户输入顺序混淆，影响回归测试可靠性

## 改动要求

### 1. 补齐 `orbital_collector` 真实可运行语义

- 统一 `orbital_collector` 的供能模型，二选一并落到实现与文档：
  - 方案 A：补 `connection_points` / 电网覆盖规则，让它像当前实现描述一样吃电运行
  - 方案 B：改成不依赖电网，而是按气态行星采集器自己的燃料 / 特殊规则运行
- 无论采用哪种方案，都必须让它在 midgame 气态行星上进入 `running`
- 增加端到端验证，确认库存里能持续增加 `hydrogen` / `deuterium`

### 2. 把当前“定义存在但不可建造”的 DSP 建筑真正并入科技树与建筑主线

- 为下列建筑补齐可落地状态：
  - `advanced_mining_machine`
  - `pile_sorter`
  - `recomposing_assembler`
  - `energy_exchanger`
  - `jammer_tower`
  - `sr_plasma_turret`
  - `planetary_shield_generator`
  - `self_evolution_lab`
- 对每个建筑明确：
  - 对应科技
  - `/catalog` 的 `buildable`
  - 必需运行时模块
  - CLI / API 是否已有公开入口
- 如果某建筑短期仍不做，必须把参考文档和玩家指南从“已覆盖”改成“未实现”，不能继续保持半真半假的定义态

### 3. 把研究系统补到“矩阵实物驱动”而不是“抽象点数驱动”

- 研究开始前要校验所需矩阵物品来源
- 研究推进时要有明确的矩阵消耗 / 扣减规则
- `matrix_lab` 与科研速度的关系要进入真实物料闭环
- 默认开局下，玩家不能在零矩阵投入时直接完成中后期科技
- 同步更新：
  - `docs/player/玩法指南.md`
  - `docs/dev/服务端API.md`
  - `docs/dev/客户端CLI.md`（若新增科研相关查询 / 命令）

### 4. 打通多星球经营与真正可用的星际物流玩法

- 至少要提供一条公开、可测试的多星球路线：
  - 切换 / 指定 active planet
  - 或让非 active planet 也能查询并经营
- 需要让玩家可以在 rocky 星球与 gas giant 之间建立真实的资源链
- midgame 场景中要能验证“气态行星采集 -> 星际物流 -> 主工厂星使用”的最小闭环

### 5. 为 `ray_receiver` 增加公开模式控制入口

- 新增 API / CLI / Web 至少一种玩家可见入口，用来切换：
  - `power`
  - `photon`
  - `hybrid`
- 文档写清模式差异、默认值和验证方法

### 6. 修复 `client-cli` 的批量命令串行语义

- `client-cli` 对连续输入的命令必须按输入顺序串行派发
- 粘贴多条命令时，`ACCEPTED request_id`、`command_result` 与输入行应保持一一对应
- 需要补自动化测试覆盖这类输入

## 需要同步更新的文档

- `docs/player/玩法指南.md`
- `docs/dev/服务端API.md`
- `docs/dev/客户端CLI.md`
- 如涉及 midgame 回归路线，也要同步：
  - `docs/player/上手与验证.md`

## 验收标准

1. 使用仓库内隔离配置启动默认局与 midgame 局，不允许手改存档注入状态。
2. `orbital_collector` 在气态行星上能稳定进入 `running`，并能通过查询看到 `hydrogen/deuterium` 库存增长。
3. 上述 8 个当前不可建造建筑，要么：
   - 已能通过研究后 `build` 成功建造并运行；
   - 要么被明确从参考 / 玩家文档中移出“已实现覆盖”范围。
4. 默认新局下，没有矩阵实物时不能凭空完成 `electromagnetism`；补齐矩阵来源后才能完成研究。
5. midgame 局里可以真实建立至少一条跨两颗行星的资源流，非 active planet 不再长期是 `runtime/networks unavailable`。
6. `ray_receiver` 能通过公开命令切到指定模式，并能验证模式差异。
7. `client-cli` 连续粘贴多条命令时，事件与 `request_id` 的对应关系稳定、可测试。
