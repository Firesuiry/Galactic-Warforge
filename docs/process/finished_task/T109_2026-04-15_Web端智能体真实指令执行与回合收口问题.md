## 当前问题列表

### 1. 建造类 turn 已真实执行动作，但最终仍以失败结束，玩家拿不到结果

- 复现步骤：
  1. 打开 `http://127.0.0.1:5173/agents`
  2. 新建成员 `web回归总管-1776267361095`
  3. 绑定 `builtin-minimax-api`
  4. 保存权限：
     - `planet-1-1`
     - `observe, build, research, management`
     - 勾选 `允许创建智能体`
  5. 发起私聊
  6. 发送：
     `在 planet-1-1 上真实执行以下动作，不要只给计划：如果当前 active planet 不是 planet-1-1，先切换到它；然后在坐标 3,2 建造 wind_turbine，在坐标 2,3 建造 matrix_lab。完成后只回复一行中文，明确写出两个建筑各自的 building id。`
- 实际现象：
  - Web 中该请求对应 turn 最终为：
    - `turnId = fa1603a3-ff78-4eb1-8331-d26c53fa503e`
    - `status = failed`
    - `outcomeKind = blocked`
    - `errorCode = loop_exhausted`
    - `rawErrorMessage = agent loop exceeded maxSteps`
  - 页面展示的是失败，而不是最终建造结果。
  - 但 authoritative 世界状态确认动作已经真的执行：
    - `wind_turbine` 已建成：`b-27 @ (3,2)`
    - `matrix_lab` 已建成：`b-28 @ (2,3)`
  - authoritative `audit` 还确认本轮实际发出了这些命令：
    - `switch_active_planet`
    - `build wind_turbine`
    - `build matrix_lab`
    - 随后又重复多次 `scan_planet`
- 影响：
  - 玩家看到的是“失败”，但世界实际上已经被改动。
  - 这会让 `/agents` 对建造类工作变成高风险入口：玩家无法判断到底该不该重试，也拿不到建筑 ID。
  - turn 收口语义与 authoritative 世界状态不一致。
- 改动需求：
  - 对建造类任务，如果关键动作已经执行成功，runtime 必须继续完成结果收口，至少返回：
    - 已建成的建筑类型
    - 建筑 ID
    - 建筑坐标
  - 不能因为后续补充观察或总结失败，就把整个 turn 标成纯失败。
  - 若动作已执行但最终总结未完成，页面与 turn 状态必须显式区分：
    - “动作已执行”
    - “结果收口失败”
  - 补一条真实浏览器回归：
    - Web 创建成员
    - Web 私聊委派建造
    - 页面最终能显示真实建筑 ID
    - authoritative 世界状态与 turn 结论一致

### 2. 科研类 turn 仍停在“只规划不执行”，不会真正执行 transfer/start_research

- 复现步骤：
  1. 复用同一个成员与私聊会话
  2. 在上一问题的建造委派后，authoritative 世界中已存在：
     - `matrix_lab = b-28`
  3. 发送：
     `在 planet-1-1 上真实执行以下动作，不要只给计划：把 10 个 electromagnetic_matrix 装入 b-28，然后启动 electromagnetism 研究。完成后只回复实际结果；如果失败，明确失败原因。`
- 实际现象：
  - Web 中该请求对应 turn 为：
    - `turnId = 5ac4d9aa-5068-4499-b95a-455b49cf953a`
    - `status = failed`
    - `repairCount = 1`
    - `errorCode = provider_incomplete_execution`
    - `rawErrorMessage = provider_incomplete_execution: 这轮只有规划，没有执行所需动作`
  - action summary 只停在：
    - `game.command / 待执行 / 扫描行星 planet-1-1`
  - authoritative 世界状态确认：
    - 没有开始 `electromagnetism`
    - 没有完成 `electromagnetism`
    - 没有看到 `transfer` 与 `start_research` 的落地结果
- 影响：
  - Web 虽然允许玩家给 agent 下科研任务，但当前默认新局里的真实科研闭环并没有打通。
  - 这会让“让 agent 帮玩家推进起步科研”成为假能力。
- 改动需求：
  - research 类 turn 不能停在 scan/plan 阶段，必须真正推进到：
    - `transfer_item`
    - `start_research`
    - 最终研究结果或真实失败原因
  - repair 逻辑在识别到“这轮需要执行动作”时，必须继续逼近真实动作，而不是接受规划句。
  - turn 失败时需要明确区分：
    - 只是缺参数
    - 真的权限不足
    - provider 只规划不执行
    - runtime repair 后仍未落动作
  - 补真实浏览器回归：
    - Web 创建成员
    - Web 授予 `research + management`
    - Web 私聊下达 `transfer + start_research`
    - authoritative 看到研究开始或给出真实可操作失败

### 3. 创建下级智能体的 turn 仍停在“声明要创建”，并没有真的落库

- 复现步骤：
  1. 复用同一个成员与私聊会话
  2. 发送：
     `创建一个名为 胡景-1776267711665 的新智能体，只给他 planet-1-1 的 observe,build 权限，并把他设成你的可调度成员。完成后只回复新成员 id、权限和你是否已能调度他；不要只给计划。`
- 实际现象：
  - Web 中该请求对应 turn 为：
    - `turnId = 92f7f0ab-96e4-44c8-a17c-495261c4be36`
    - `status = failed`
    - `repairCount = 1`
    - `errorCode = provider_incomplete_execution`
  - action summary 只显示：
    - `agent.create / 待执行 / 创建智能体 胡景-1776267711665`
  - authoritative `/agents` 列表确认：
    - 成员总数没有增加，仍是 `1`
    - 新成员并未创建成功
  - authoritative 父成员状态确认：
    - `managedAgentIds = []`
    - 没有新增可调度关系
- 影响：
  - “让总管类成员创建下级成员并继续委派”这条核心协作链仍未真正可用。
  - 页面虽然能发出这类请求，但当前结果仍然只是 planner 文案，不是实际系统动作。
- 改动需求：
  - `agent.create` 类 turn 必须以真实落库结果为准，而不是以 planner 预告句为准。
  - 创建成功后至少要同步给出：
    - 新 agent ID
    - role
    - policy
    - 是否已写入 `managedAgentIds`
  - 若失败，页面必须给出真实失败原因，而不是泛化成“只规划不执行”。
  - 补真实浏览器回归：
    - Web 创建上级成员
    - Web 授予 `canCreateAgents`
    - Web 私聊让其创建下级成员
    - authoritative `/agents` 中出现新成员，且权限与调度关系正确

## 改动目标

- 保持当前 `/agents` 中已可用的成员创建、权限保存、私聊入口不回退。
- 把 Web 智能体工作台从“能发请求”修到“真实动作能执行且能正确收口”。
- 让建造、科研、创建下级成员三类高价值任务都以 authoritative 结果为准，而不是停在 planner 状态。

## 验收标准

1. Web 中新建成员、保存权限、发起私聊继续保持可用。
2. 建造类任务如果已真实落库，不会再在页面上只表现为失败；玩家能看到建筑 ID 与真实结果。
3. 科研类任务能真实执行 `transfer_item + start_research`，不能再停在 scan/plan。
4. 创建下级成员任务能真实创建出新 agent，并正确落库权限与调度关系。
5. `/agents` 页面中的 turn 状态、动作摘要、最终回复与 authoritative 世界状态保持一致。
