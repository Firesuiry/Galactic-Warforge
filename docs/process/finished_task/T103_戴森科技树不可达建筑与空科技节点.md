## 当前问题 1：`automatic_piler` 与 `satellite_substation` 仍未接入真实可达科技树

### 复现

1. 启动默认新局 `18310`
2. 使用从未操作过的 `p2 / key_player_2` 登录
3. 确认当前科技状态只有：
   - `dyson_sphere_program`
4. 在 `planet-1-1` 执行：
   - `build 1994 1995 automatic_piler`
   - `build 1993 1994 satellite_substation`

### 实际现象

- 两条命令都被 authoritative 拒绝：
  - `building type automatic_piler requires research to unlock`
  - `building type satellite_substation requires research to unlock`
- 但当前 `/catalog.buildings[]` 又同时把这两个建筑暴露为：
  - `buildable = true`
- 本轮对 `/catalog.techs[]` 反查，没有任何科技节点公开解锁：
  - `automatic_piler`
  - `satellite_substation`
- `docs/player/玩法指南.md` 仍把这两个建筑列在当前可用建筑集中：
  - 阶段 G 里把 `automatic_piler` 列为“当前已经能摆出来的相关基础设施”
  - 建筑总表里把 `satellite_substation` 列入“发电与电网”
- `server/internal/gamecore/rules.go` 当前建造校验明确只认 `TechUnlockBuilding`
- `server/internal/model/tech.go` 当前没有任何公开科技在规范化后给出：
  - `automatic_piler`
  - `satellite_substation`
  的 `TechUnlockBuilding`

### 影响

- 这两个 DSP 建筑现在不是“可玩但需要另行研究”，而是玩家看得到、CLI 也支持 `build`，但实际上没有可达解锁路径
- 玩家文档、`/catalog` 与 authoritative 行为不一致，会把试玩者和后续回归测试引到假入口
- 物流分支与电网分支各自还保留至少一个未闭合的建筑入口

## 当前问题 2：当前公开科技树里仍有一批“空科技节点”

### 现象

- 本轮直接查询 `/catalog.techs[]`，发现以下科技节点同时满足：
  - `unlocks = []`
  - `effects = []`
  - 不是可重复升级科技
- 当前确认到的空节点有：
  - `casimir_crystal`
  - `combustible_unit`
  - `crystal_explosive`
  - `crystal_shell`
  - `crystal_smelting`
  - `engine`
  - `high_strength_glass`
  - `particle_control`
  - `polymer_chemical`
  - `proliferator_mk2`
  - `proliferator_mk3`
  - `reformed_refinement`
  - `steel_smelting`
  - `super_magnetic`
  - `supersonic_missile`
  - `thruster`
  - `titanium_ammo`
  - `wave_interference`
  - `xray_cracking`

### 影响

- 这些科技节点虽然还挂在玩家可见科技树里，但当前不会给出任何建筑、配方或数值收益
- 从玩家视角看，科技树里仍存在一批“名字和前置都在，实际研究后没有玩法变化”的假节点
- 这说明“戴森球计划相关科技树已实现”的结论还不能完全成立；至少配方/奖励层仍有明显断点

## 改动要求

### 1. 收口 `automatic_piler` 与 `satellite_substation` 的科技可达性

- 二选一，但必须前后一致：
  - 方案 A：把它们接回真实科技树
    - `automatic_piler` 补到合适物流科技的 `TechUnlockBuilding`
    - `satellite_substation` 补到合适电力科技的 `TechUnlockBuilding`
  - 方案 B：若短期不做，立即把它们从当前玩家可见能力集中移除
    - `/catalog.buildings[].buildable` 不能继续对玩家宣称可建
    - `docs/player/玩法指南.md`
    - `docs/dev/客户端CLI.md`
    - `docs/dev/服务端API.md`
    必须同步改口
- 无论采用哪种方案，都要保证 `/catalog`、文档与 authoritative 建造校验一致

### 2. 清理当前公开科技树里的空节点

- 对上面列出的空科技逐个收口：
  - 要么补齐实际 `unlock` / `effect`
  - 要么在当前版本隐藏，不再出现在玩家可见科技树与玩家文档里
- 如果这些科技对应的是尚未实现的 recipe / item / ammo / branch，就不能继续作为“已开放科技树节点”公开暴露

### 3. 增加防回归验证

- 增加测试覆盖以下约束：
  - 默认新局玩家在未完成对应科技前，不能建造 `automatic_piler` / `satellite_substation`
  - 完成对应科技后，两个建筑都能真实建造成功
  - `/catalog` 不再暴露“`buildable=true` 但没有任何可达科技解锁路径”的玩家建筑
  - `/catalog.techs[]` 不再暴露“无 unlock / 无 effect / 非可重复升级”的空科技节点

## 验收标准

1. 玩家能从科技树、`/catalog`、CLI 帮助和文档中一致地看出 `automatic_piler` 与 `satellite_substation` 的真实获取路径。
2. 默认新局里，未解锁前建造会失败；解锁后建造会成功。
3. 当前玩家可见科技树里，不再存在无任何实际产出或效果的空科技节点。
