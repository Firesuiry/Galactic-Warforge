## 当前问题：`/state/stats` 的生产统计把没有实际产出的建筑也记入 `total_output`

### 复现

1. 启动官方 midgame 场景，补一套稳定电网。
2. 在 `planet-1-2` 上实际建出并保持供电：
   - `recomposing_assembler (b-51)`
   - `self_evolution_lab (b-52)`
   - `vertical_launching_silo (b-35)`
3. 不给这些建筑补任何真实生产输入：
   - `b-51` 不设置 `recipe_id`
   - `b-52` 不设置 `recipe_id`
   - `b-35` 虽然默认挂了 `small_carrier_rocket`，但始终 `input_shortage = true`
4. 查询：
   - `GET /state/stats`
   - `GET /world/planets/planet-1-2/inspect?entity_kind=building&entity_id=b-51`
   - `GET /world/planets/planet-1-2/inspect?entity_kind=building&entity_id=b-52`
   - `GET /world/planets/planet-1-2/inspect?entity_kind=building&entity_id=b-35`

### 实际现象

- `/state/stats` 返回：
  - `production_stats.total_output = 6`
  - `production_stats.by_building_type.recomposing_assembler = 2`
  - `production_stats.by_building_type.self_evolution_lab = 3`
  - `production_stats.by_building_type.vertical_launching_silo = 1`
  - 但 `production_stats.by_item = {}`
- `b-51 (recomposing_assembler)` 的 `inspect` 显示：
  - `production = { "mode": "speed" }`
  - 没有 `recipe_id`
  - `production_monitor.last_stats.input_shortage = true`
  - `production_monitor.last_stats.efficiency = 0`
- `b-52 (self_evolution_lab)` 的 `inspect` 显示：
  - `production = { "mode": "speed" }`
  - 没有 `recipe_id`
  - `production_monitor.last_stats.input_shortage = true`
  - `production_monitor.last_stats.efficiency = 0`
- `b-35 (vertical_launching_silo)` 的 `inspect` 显示：
  - `production.recipe_id = "small_carrier_rocket"`
  - `production_monitor.last_stats.input_shortage = true`
  - `production_monitor.last_stats.efficiency = 0`
- 也就是说，这 3 座建筑都没有真实产出，但统计仍然把它们的理论 `throughput` 当成了玩家“总产出”。

### 影响

- `summary / stats / Web 总览` 给出的局势判断会失真。
- 当前 midgame 下，玩家会看到“总产出 = 6”，但实际上没有任何对应物品产出，也没有 `by_item` 统计支撑。
- 这会误导：
  - 玩家对当前产线是否真的在工作中的判断
  - AI/自动化对局势和瓶颈的分析
  - 后续 DSP 中后期产线闭环的回归测试

### 研判

- 这是根据当前实现作出的直接推断：
  - `server/internal/gamecore/stats_settlement.go` 目前只要建筑带 `Runtime.Functions.Production`，就直接把 `Throughput` 加进 `production_stats.total_output`
  - 结算时没有校验建筑是否真的有 `recipe_id`、是否发生真实产出、是否只是 `input_shortage`
- 这不是《戴森球计划》设定差异，而是统计口径错误。

### 改动要求

- 修正 `updateProductionStats` 的统计口径，不要再把“只有生产函数定义、但没有实际产出”的建筑计入 `total_output`。
- 至少覆盖以下情况：
  - 没有 `recipe_id` 的生产建筑不能计入产出
  - `input_shortage = true` 且本 tick 没有真实产物生成时不能计入产出
  - `vertical_launching_silo` 这类默认挂配方但空转的建筑，不能仅凭 `throughput=1` 就算有产出
- 统计应优先来自真实产出事件 / 实际出货量 / 本 tick 实际完成的 recipe，而不是静态能力值。
- 同步检查以下展示面是否都改成同一口径：
  - `GET /state/stats`
  - CLI `stats`
  - `client-web` 总览页中的局势卡片与后续可能接入的生产概览
- 增加回归测试，至少覆盖：
  - `recomposing_assembler` 无配方时 `total_output = 0`
  - `self_evolution_lab` 无配方时 `total_output = 0`
  - `vertical_launching_silo` 在 `input_shortage = true` 且没有实际火箭产出时不计入 `total_output`
  - 真正产生产物时，`total_output / by_building_type / by_item` 三者同步增长

## 验收标准

1. 官方 midgame 场景下，如果生产建筑没有真实产出：
   - `production_stats.total_output = 0`
   - `production_stats.by_building_type` 不应出现对应建筑的虚假产出
   - `production_stats.by_item` 保持为空
2. 当生产建筑实际完成配方并产生产物时：
   - `total_output`
   - `by_building_type`
   - `by_item`
   必须基于同一份真实产出事实同步增长。
3. `client-web` 或其他消费 `stats` 的页面，不再把空转建筑显示成“正在提供产出”的局势来源。
