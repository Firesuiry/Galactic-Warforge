## 当前问题 1：`ray_receiver` 的 `power` 模式没有把已有戴森能量转成可见的电网收益

### 复现

1. 用官方 midgame 场景启动新局，并在 `planet-1-2` 建出以下链路：
   - `ray_receiver`
   - `vertical_launching_silo`
   - `em_rail_ejector`
2. 执行：
   - `transfer b-48 solar_sail 3`
   - `transfer b-46 small_carrier_rocket 1`
   - `set_ray_receiver_mode b-50 power`
   - `build_dyson_node sys-1 0 10 20 --orbit-radius 1.2`
   - `build_dyson_node sys-1 0 -10 -20 --orbit-radius 1.2`
   - `build_dyson_frame sys-1 0 p1-node-l0-latp1000-lonp2000 p1-node-l0-latm1000-lonm2000`
   - `build_dyson_shell sys-1 0 -15 15 0.4`
   - `launch_solar_sail b-48 --count 1`
   - `launch_rocket b-46 sys-1 --layer 0 --count 1`
3. 等待数秒后查询：
   - `GET /events/snapshot?event_types=entity_updated&limit=200`
   - `GET /state/summary`
   - `GET /state/stats`
   - `GET /world/planets/planet-1-2/inspect?entity_kind=building&entity_id=b-50`

### 实际现象

- 审计日志确认所有戴森脚手架和发射命令都已成功执行。
- `entity_updated` 事件里已经持续出现：
  - `entity_type = dyson_sphere`
  - `total_energy = 612`
- `b-50` 的 `ray_receiver` 明确处于：
  - `state = running`
  - `mode = power`
- 但玩家可见收益没有同步出现：
  - `summary.players.p1.resources.energy` 在等待后仍停在 `9909`
  - `stats.energy_stats.generation` 仍停在 `126`
  - `stats.energy_stats.consumption` 仍停在 `96`
- 也就是说，已经存在的戴森能量并没有通过 `power` 模式反映成玩家能观察到的供电回灌。

### 影响

- 当前“太阳帆 / 戴森结构 -> 射线接收站 `power` 模式 -> 电网收益”这条 DSP 核心闭环对玩家来说仍然是断的。
- 文档里关于 `ray_receiver power/hybrid` 已接入真实能源结算、能反映到能源统计的说法，和实际试玩结果不一致。
- 这会直接误导后续对戴森中后期能源玩法是否完成的判断。

### 改动要求

- 修正 `ray_receiver` 在 `power` 模式下的实际结算，使已有太阳帆/戴森结构能量真实转成玩家可见收益。
- 至少保证以下 3 个观察面一致：
  - `summary.players[pid].resources.energy`
  - `state/stats.energy_stats.generation`
  - `world/planets/{planet_id}/networks.power_networks[].supply`
- 增加端到端验证，覆盖：
  - 已有 `dyson_sphere total_energy > 0`
  - `ray_receiver mode = power`
  - 发射成功后若等待若干 tick，供电统计会显著高于纯风机/基础发电时的值

## 当前问题 2：官方 midgame 验证场景仍不能直接覆盖 `advanced_mining_machine` / `pile_sorter` / `recomposing_assembler`

### 复现

在同一局里把执行体移动到 `(7,4)` 后执行：

- `build 8 6 recomposing_assembler`
- `build 8 7 pile_sorter`
- `build 10 7 advanced_mining_machine`

### 实际现象

- 3 条命令都不是“建筑不存在”或“运行时没接线”，而是直接被科技门禁拒绝：
  - `building type recomposing_assembler requires research to unlock`
  - `building type pile_sorter requires research to unlock`
  - `building type advanced_mining_machine requires research to unlock`
- 对照实现可知它们的科技入口分别是：
  - `recomposing_assembler -> annihilation`
  - `pile_sorter -> integrated_logistics`
  - `advanced_mining_machine -> photon_mining`
- 但当前 `config-midgame.yaml` 并没有预置这些科技，因此官方 midgame 路线仍无法直接深度试玩这 3 个 DSP 建筑。

### 影响

- 现在仓库里虽然已经把这 3 个建筑纳入“当前存在的 DSP 建筑集合”，但官方中后期验证场景仍无法一把覆盖。
- 用户如果按当前文档理解“官方 midgame 可以验证当前戴森中后期剩余建筑”，会在这里被卡住。
- 这会让“已实现”和“当前场景可直接验证”混在一起，影响回归结论。

### 改动要求

- 二选一，但必须明确收口：
  - 方案 A：把 `annihilation`、`integrated_logistics`、`photon_mining` 加入官方 `config-midgame.yaml` 预置科技，让这 3 个建筑能在官方验证场景里直接试玩。
  - 方案 B：保留当前科技门禁，但同步修改：
    - `docs/player/玩法指南.md`
    - `docs/player/上手与验证.md`
    - `docs/dev/客户端CLI.md`
    - `docs/dev/服务端API.md`
    明确说明这 3 个建筑“实现已存在，但不属于官方 midgame 的即开即验范围”。

## 验收标准

1. 官方 midgame 场景下，戴森脚手架、太阳帆、火箭和 `ray_receiver power` 模式全部成功执行后，玩家可以从统计或资源面看到明确的电力收益回灌。
2. `advanced_mining_machine`、`pile_sorter`、`recomposing_assembler` 的“已实现但当前场景不可直接验证”状态必须被收口为：
   - 要么官方 midgame 直接可玩
   - 要么所有玩家侧文档都明确标注为“需要另行研究，不在官方 midgame 即开即验范围内”
