## 任务背景

本轮按用户要求进行了真实深度试玩，主入口使用 `client-web`，并用 `client-cli` 与直接 HTTP 查询做 authoritative 交叉核验。

重点不是重写理想设计，而是回答两件事：

1. 当前戴森球计划相关建筑、科技树、玩法到底有没有实现。
2. Web 端现在是否已经“好用到适合长期游玩”。

- **功能实现层面**：默认新局和官方 midgame 的戴森主链已经基本打通。
- **Web 体验层面**：桌面端已可玩，但中后期操作和反馈仍不够优雅，仍有明确的交互改造空间。

## 当前问题列表

### 1. 建造与移动仍过度依赖手填坐标，Web 缺少足够强的地图联动与距离辅助

- 真实现象：
  - 默认新局第一次在可视区域附近尝试 `build (4,3) wind_turbine`，authoritative 直接返回：
    - `INVALID_TARGET: target tile is not buildable`
  - midgame 中尝试把执行体从 `(1,1)` 直接移动到 `(7,4)`，authoritative 返回：
    - `OUT_OF_RANGE: move distance 9 exceeds unit move range 4`
  - midgame 中尝试直接在 `(9,2)` 建第二座 `tesla_tower`，authoritative 返回：
    - `OUT_OF_RANGE: executor out of range: 9 > 6`
  - 最终必须人工拆成多段移动：
    - `(1,1) -> (5,1) -> (7,3) -> (7,4)`
  - 当前 Web 虽然能做这些操作，但主要仍靠玩家自己输入坐标、自己记距离、自己反复切工作流。
- 影响：
  - 功能上可玩，但交互成本高。
  - 默认新局在大地图小视窗下不够直觉，中后期多建筑验证也容易反复踩 `INVALID_TARGET / OUT_OF_RANGE`。
  - “主要用 Web 游玩”已经成立，但离“优雅、顺手”还有明显距离。
- 改动要求：
  - 行星地图需要把点击/选中结果直接联动到建造与移动表单，不要继续以纯手填坐标为主。
  - 建造前检查需要更直接地显示目标格的地形、是否可建、是否被资源点/建筑占用、是否超出执行体操作范围。
  - 当目标超出移动或操作范围时，工作流应提供更强的一步式辅助：
    - 自动回填建议落点
    - 明确提示还差多少距离
    - 能用一次操作串起多段移动，或至少给出一组推荐中间点

### 2. 射线接收站模式面板仍直接暴露未解锁的 `photon` 选项，科技门禁没有前置提示

- 真实现象：
  - 官方 midgame 故意未预置 `dirac_inversion`
  - 但 Web 射线接收站模式下拉框仍直接显示：
    - `power`
    - `photon`
    - `hybrid`
  - 玩家在 Web 中选择 `photon` 后，命令先 accepted，随后 authoritative 失败：
    - `VALIDATION_FAILED: photon mode requires dirac_inversion`
- 影响：
  - 这不是实现缺失，而是 UI 明知当前不可用却仍把动作完整暴露给玩家。
  - 玩家只有在失败后才知道当前是科技门禁，而不是在表单里提前得到解释。
  - 这会降低科技树与操作面板之间的一致性。
- 改动要求：
  - 当玩家未完成 `dirac_inversion` 时，Web 应禁用 `photon` 选项，或至少明确标注“未解锁”。
  - 模式区需要直接展示缺失科技，不要把门禁判断完全留到 authoritative 失败后。
  - `power / hybrid / photon` 三种模式的显示内容应和当前科技树状态保持一致。

### 3. 告警与结果区仍混杂原始后台口径，玩家视角的信息层级不够统一

- 真实现象：
  - Web 桌面页虽然已经能显示最近 authoritative 结果，但告警板仍长期出现原始后台文案，例如：
    - `power_shortage`
    - `building b-44 power shortage`
    - `throughput drop detected`
  - 这些告警会和更高价值的操作结果同时占据主视觉区域。
  - 对普通玩家来说，建筑类型、建筑名、坐标、问题原因并没有被统一翻译到同一套玩家口径。
- 影响：
  - 当前页面已经不再是“完全不可用”，但仍明显带有开发调试台味道。
  - 玩家需要自己把 `b-44`、`throughput drop`、`under_power` 翻译成游戏语义，阅读负担偏高。
  - 在中后期建筑较多时，关键信息容易被低层级告警稀释。
- 改动要求：
  - 告警板必须把告警类型、建筑 ID、状态原因统一翻译成玩家视角文案。
  - 同时展示：
    - 建筑名称
    - 位置
    - 问题类型
    - 建议动作
  - 最近 authoritative 命令结果应优先于低价值 throughput/power 噪声，形成更清晰的信息层级。

## 改动目标

- 保持当前默认新局与 midgame 戴森主链不回退。
- 把 Web 从“功能已接通”进一步收口到“中后期玩家也能顺手操作”。
- 让科技门禁、坐标操作、结果反馈这三类高频交互不再依赖试错。

## 验收标准

1. 玩家可通过地图选点直接驱动建造/移动表单，且在提交前就能看到可建性与距离校验结果。
2. 对超范围移动或建造，页面能给出明确可执行的引导，而不是只在 authoritative 阶段报错。
3. 未解锁 `dirac_inversion` 时，射线接收站模式面板不会再把 `photon` 当成普通可用选项暴露。
4. 告警板与结果区不再直接显示 `power_shortage`、`throughput drop detected`、裸 `building id` 这类后台原词；玩家能直接看到统一后的中文语义。
5. 默认新局和 midgame 的既有闭环：
   - 默认新局 Web 起步科研
   - midgame 中后期建筑建造
   - 戴森 node/frame/shell
   - 太阳帆/火箭发射
   - `ray_receiver hybrid`
   不发生回退。

### 实现结果

- 建造与移动工作流补上了地图联动后的前置诊断：
  - 建造前直接显示目标地形、是否可建、资源点/建筑占用情况。
  - 当执行体操作距离不足时，直接给出建议落点、剩余差距与推荐中间点，并把第一跳自动回填到移动表单。
  - 移动面板新增距离/移动范围预检，对超范围目标显示推荐路线并支持一键填入第一跳。
- 射线接收站模式表单接入科技树状态：
  - 未完成 `dirac_inversion` 时，`photon` 选项改为禁用并标明“未解锁”。
  - 模式区直接展示缺失科技提示，不再让玩家先提交再吃 authoritative 失败。
- 活动流与告警面板改成玩家视角文案：
  - 告警面板统一显示建筑名称、坐标、问题类型与建议动作。
  - 默认关键反馈时间线不再把 `production_alert` 和命令结果混在一起，命令结果优先级更清晰。
  - `throughput_drop`、`power_shortage`、裸建筑 ID 等后台原词不再直接暴露给玩家。
