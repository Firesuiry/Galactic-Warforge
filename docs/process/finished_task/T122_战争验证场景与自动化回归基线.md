# T122 战争验证场景与自动化回归基线

## 背景

战争系统是跨：

- 服务端
- CLI
- Web
- Agent

的长链路能力。如果没有稳定验证场景和自动化回归，后续每个任务都很容易各自为政，最终只能靠手工推演。

当前仓库已有：

- `config-dev.yaml`
- `config-midgame.yaml`
- Playwright / CLI / Go 测试基础

但没有一套面向新战争系统的官方验证场景和回归基线。

## 设计参考

- `docs/dev/战争组件设计.md`
  - 第 19 章“分阶段落地路线”
  - 第 21 章“示例蓝图”

## 需求细节

1. 必须有一套官方战争验证局，能直接验证蓝图、量产、舰队、补给、战报、登陆。
2. 必须有自动化回归，覆盖：
   - 服务端
   - CLI
   - Web
3. 场景必须是 authoritative 配置，不靠手工临时摆局。

- `T121_client_web战争蓝图军工战区与战报工作台.md`

- `client-web/tests/*`
- `server/internal/gamecore/*test.go`

## 改动需求

1. 新增至少一套官方战争验证场景，建议覆盖：
   - 已解锁中阶战争科技
   - 预置至少一批公开蓝图
   - 可直接验证舰队、补给、登陆和地面任务群
2. 自动化回归至少包含：
   - Go 测试：蓝图、补给、战报、登陆、制轨权
   - CLI 测试：最小战争操作闭环
   - Playwright 测试：Web 工作台核心流程
3. 验证场景与文档口径必须一致，不能再出现“文档说可验证、实际空白”的状态。

## 不在本任务范围

- 不要求完成 AI 军事自治
- 不要求完成最终文档收口

## 验收标准

1. 仓库存在稳定、可复现的官方战争验证场景。
2. Go / CLI / Web 三类自动化回归覆盖战争系统主链路。
3. 新场景可以被后续任务和试玩直接复用，不需要每次手工起局补环境。

- 完成时间：2026-04-20
