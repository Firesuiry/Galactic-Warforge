## 当前问题列表

### 1. 戴森页“新建节点后继续建框架”的候选节点不会即时刷新，Web 工作流被迫中断

- 真实现象：
  - midgame Web 已成功提交 `build_dyson_node`
  - authoritative 已新增新节点 `p1-node-l0-latm1000-lonm2000`
  - 但不刷新页面时，后续 `build_dyson_frame` 表单里的“节点 B”选择器仍看不到新节点
  - 手动刷新后重新进入 `/planet/planet-1-2`，新节点才会出现在候选列表中，随后 frame/shell/sail/rocket 才能继续走通
- 影响：
  - 戴森球玩法本体已经实现，但 Web 把一个连续的“建节点 -> 建框架”操作链切成了“必须刷新页面”的半断流
  - 这会直接降低中后期玩法的可操作性与 UI 优雅度
- 改动要求：
  - `build_dyson_node/frame/shell` 成功后，立即失效并刷新依赖它们的系统运行态、节点列表与表单候选数据
  - 表单不应依赖旧快照；新增节点后，同页内应立刻可被 frame/shell 选择器使用
  - 为这条链补一条真实浏览器回归：同页新建 node 后无需刷新即可继续建 frame

### 2. 智能体 observe 任务会被判定为 succeeded，但没有交付最终结果

- 真实现象：
  - midgame 成员：`李斯`
  - 会话：`7485c2e3-6c78-4e6b-a06c-78a871d75f81`
  - 请求：`先观察 planet-1-2 当前戴森局势，只回复一句总结。`
  - authoritative turn：`981b9b87-b416-4060-a1a9-a2b8b7e97ab4`
  - turn 状态：`succeeded`
  - `outcomeKind = observed`
  - 最终回复内容却是：`正在等待 planet-1-2 扫描结果，目前暂无返回。`
- 影响：
  - 对玩家来说，这不是“成功完成观察”，而是“动作提交了，但结果没回来”
  - 前端显示成功会制造假阳性，降低智能体工作台的可信度
- 改动要求：
  - observe 类 turn 只有在返回面向用户的最终总结后才能判定 `succeeded`
  - 如果当前轮只完成了中间动作提交，runtime 必须继续收口，而不是提前结束
  - 页面上应明确区分“已提交观察动作”和“已交付最终观察结论”

### 3. 智能体建造与创建下级成员仍会停在“只有规划，没有执行”

- 真实现象：
  - midgame 成员：`李斯`
  - 会话：`7485c2e3-6c78-4e6b-a06c-78a871d75f81`
  - 建造请求：`在当前 active planet 上找一个可建位置，建造一个 signal_tower，并告诉我建筑 id。`
  - authoritative turn：`79366886-d1b8-42c8-b9ae-b4af81b5aa8c`
  - 结果：`failed`
  - `errorCode = provider_incomplete_execution`
  - `errorMessage = 这轮只有规划，没有执行所需动作。`
  - 创建下级请求：`创建一个名为胡景的新智能体，只给他 planet-1-2 的 observe,build 权限，并把他设为你的可调度成员。`
  - authoritative turn：`c4a5af09-ba20-48a6-a307-84964ace44fb`
  - 结果同样为 `provider_incomplete_execution`
  - `/agents` authoritative 列表中最终没有创建出 `胡景`
- 影响：
  - “让智能体建造”和“让智能体创建下级成员”都是工作台的核心场景，目前都不可靠
  - 当前页面虽然能让玩家发起请求，但执行链路仍经常停留在 planner 阶段
- 改动要求：
  - provider repair / runtime completion 逻辑需要继续收口，不能把“只有规划”当作可接受终态
  - 对 `game.command` 与 `agent.create` 这类需要落库的动作，turn 必须以实际执行结果为准
  - 对这两类失败补充真实浏览器回归，确保至少各有一条成功样例

### 4. 智能体科研任务不会真正执行 transfer/start_research，而是循环 scan_planet 直至耗尽步数

- 真实现象：
  - 默认新局通过 Web 新建成员：`科研官默认局-1776173176611`
  - 该成员的权限保存已成功：
    - `planetIds = ["planet-1-1"]`
    - `commandCategories = ["observe","research","management"]`
  - 私聊会话：`40876535-d5f9-4458-b0c2-0b4808683837`
  - 玩家请求：`在 planet-1-1 把 10 个 electromagnetic_matrix 装入 b-27，然后启动 basic_logistics_system 研究。完成后只回复实际结果；如果失败，明确失败原因。`
  - authoritative turn：`77b86f1e-8b71-4add-bf8f-afef3c3d42af`
  - Web 20 秒后仍只显示规划中的第一步：切换 active planet
  - CLI/SSE authoritative 继续确认该 turn 在后台重复发出 `scan_planet`，例如：
    - `request_id=f2960463-7a4f-4d41-9f79-5c0a3dc70da6`
    - `request_id=b262a68c-de45-41a8-a8bd-efaa3c009941`
    - `request_id=c6d700eb-331d-4121-b84a-a839b72d5581`
    - `request_id=142f55ff-0cc3-4b27-84ec-f509b304eba8`
    - `request_id=304f798c-3eaf-4911-b78b-925f263f4e2b`
  - 最终 turn 失败：
    - `status = failed`
    - `outcomeKind = blocked`
    - `errorCode = permission_denied`
    - `errorMessage = 当前智能体权限不足，无法执行该操作。`
    - `rawErrorMessage = agent loop exceeded maxSteps`
  - 同时 authoritative 世界状态确认：
    - 玩家背包仍有 `40 electromagnetic_matrix`
    - `basic_logistics_system` 没有开始也没有完成
- 影响：
  - 当前“让智能体做科研”的链路并未实现闭环，实际停留在反复观察而非执行
  - 错误口径也不真实：表层是 `permission_denied`，底层实际是 `agent loop exceeded maxSteps`
- 改动要求：
  - research 委派需要确保在观测到足够上下文后，真正执行 `transfer_item + start_research`
  - runtime 不能把“循环观察直至耗尽步数”映射成误导性的 `permission_denied`
  - turn 失败时应优先保留真实失败原因，并明确告诉玩家到底是参数不足、权限不足，还是 agent loop 卡死
  - 为默认新局补一条真实浏览器回归：新建 agent -> 授权 -> 私聊委派科研 -> authoritative 完成研究或给出真实可操作失败

## 改动目标

- 保持当前已确认可玩的默认新局与 midgame 戴森主链不回退
- 把 Web 戴森工作流从“必须手动刷新才能继续”修到同页连续可操作
- 把智能体工作台从“能发请求”修到“observe/build/create/research 都能正确收口”

## 验收标准

1. midgame Web 在同一页里执行 `build_dyson_node` 后，无需刷新即可继续选择新节点执行 `build_dyson_frame`。
2. `/agents` 中 observe 请求只有在交付最终结论后才会显示成功。
3. `/agents` 中让成员建造建筑时，能真实落地至少一条成功建造样例，并返回建筑 id。
4. `/agents` 中让成员创建下级智能体时，能真实创建出新 agent，且权限与可调度关系正确落库。
5. `/agents` 中默认新局科研委派能真实完成 `transfer_item + start_research`，不能再停在重复 `scan_planet` 直到 `maxSteps` 耗尽。
6. 任何 agent 失败都要尽量暴露真实 failure cause，不能再用误导性的表层错误覆盖底层原因。
