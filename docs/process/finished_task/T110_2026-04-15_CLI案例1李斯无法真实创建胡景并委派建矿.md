# T110 2026-04-15 CLI案例1李斯无法真实创建胡景并委派建矿

## 任务背景

按用户提供的案例 1 做真实试玩验证，目标是确认以下链路是否真的可用：

1. 创建丞相智能体李斯。
2. 赋予李斯全部可配置权限。
3. 让李斯创建工部尚书胡景，并只赋予其建筑权限。
4. 让李斯下令新建一个矿场，并委派胡景执行。
5. 胡景真实建出矿场。

本轮没有使用 mock，也没有只看单测，而是直接联调：

- `server`: `http://127.0.0.1:18081`
- `agent-gateway`: `http://127.0.0.1:18181`
- `client-cli`
- 玩家：`p1 / key_player_1`
- Provider：`builtin-minimax-api`

## 复现步骤

### 1. 创建李斯

```bash
agent_create 李斯 --id agent-lisi --provider builtin-minimax-api --role director
```

实际结果：

- 创建成功：`agent-lisi`
- `role = director`

### 2. 赋予李斯全部当前可配置权限

```bash
agent_update agent-lisi \
  --can-create-agents true \
  --can-create-channel true \
  --can-manage-members true \
  --can-invite-by-planet true \
  --can-create-schedules true \
  --command-categories observe,build,combat,research,management \
  --planet-ids planet-1-1
```

实际结果：

- 更新成功
- authoritative `/agents/agent-lisi` 确认：
  - `canCreateAgents = true`
  - `canCreateChannel = true`
  - `canManageMembers = true`
  - `canInviteByPlanet = true`
  - `canCreateSchedules = true`
  - `commandCategories = ["observe","build","combat","research","management"]`
  - `planetIds = ["planet-1-1"]`

### 3. 指令李斯创建胡景

```bash
agent_message agent-lisi 创建一个新的智能体，工部尚书胡景，赋予其建筑权限
```

预期结果：

- `/agents` 中新增胡景
- 胡景只具备建筑权限
- 李斯的 `managedAgentIds` 中出现胡景

实际结果：

- `agent-lisi.status = error`
- `/agents` 中仍只有李斯一个成员
- `managedAgentIds = []`
- `agent-lisi` 的 thread 只留下了一句助手回复：

```text
正在创建工部尚书胡景，赋予其建筑权限。
```

- thread 中没有任何 `toolCalls`
- thread 中没有任何 `executionLogs`

也就是说，这一步只产生了“正在创建”的文案，没有真实创建动作落库。

### 4. 在第 3 步失败后继续按案例发送“新建一个矿场”

```bash
agent_message agent-lisi 新建一个矿场
```

预期结果：

- 李斯通知胡景执行建造
- 胡景真实建出一个 `mining_machine`

实际结果：

- `agent-lisi.status` 再次变成 `error`
- `planet-1-1` 的 authoritative scene 里仍只有初始基地：

```text
b-1 battlefield_analysis_base @ (3,3)
```

- 没有新增 `mining_machine`
- 李斯 thread 新增了两段纯文本回复：

```text
请问要在哪颗行星的哪个坐标建造矿场？请提供行星 ID 以及 x、y 坐标。
请问要在哪颗行星建造矿场？提供行星 ID 和坐标后，我将指令胡景执行建造。当前先扫描一下当前激活的行星查看情况。
```

- 但 thread 中仍然没有 `toolCalls`
- 也没有任何扫描或建造动作真的发生

## 问题判定

本案例在第 3 步就已经失败，第 4 步和第 5 步因此无法按预期成立。

更具体地说，这里至少有三层问题：

1. `POST /agents/:agentId/messages` 这条直接 thread 入口在 `agent.create` 场景下没有真实执行，只留下规划性文案。
2. 同一入口在“新建一个矿场”场景下也没有真实执行观察或建造动作，却在文案里声称“将指令胡景执行建造”“当前先扫描一下当前激活的行星查看情况”。
3. 当 agent 最终处于 `status=error` 时，公开 thread 里没有错误码、没有执行日志、没有动作摘要，排查信息不足。

## 为什么这是有效缺陷

`docs/dev/客户端CLI.md` 当前明确把下面这条路径写成了 **Case1 最小回归路径**：

```bash
agent_create 李斯 --id agent-lisi --provider provider-case1 --role director --can-create-agents true --command-categories observe,build,combat,research,management --planet-ids planet-1-1
agent_message agent-lisi 创建胡景，并赋予其建筑权限
agent_list
agent_message agent-lisi 新建一个矿场
agent_thread agent-lisi
```

也就是说，当前文档已经把“`agent_message agent-lisi 新建一个矿场` 可以跑通案例 1”当成公开能力描述；但本轮真实验证表明，这条链路并没有真正可用。

## 影响

- 案例 1 目前不能真实完成。
- `client-cli` 中关于 Case1 的公开回归路径与实际实现不一致。
- 玩家或测试者会看到“正在创建”“我将指令胡景执行建造”这类表述，但系统没有任何真实动作落库，容易误判当前状态。

## 改动目标

1. 让 `agent_message` 直接 thread 入口真正支持 `agent.create` 落库，而不是只停在助手文案。
2. 让“新建一个矿场”这类已在文档中声明支持的最小案例，能至少完成：
   - 必要观察
   - 胡景创建
   - 委派
   - `mining_machine` 实际落地
3. 当 turn 失败时，`agent_thread` 至少要能公开：
   - 错误码
   - 失败原因
   - 已执行动作数
   - 若文案声称已扫描/已委派，则必须能看到对应动作证据

## 验收标准

1. 用本任务里的同一组命令，能真实创建出胡景。
2. 胡景创建后，`managedAgentIds` 会正确写回李斯。
3. 再执行 `agent_message agent-lisi 新建一个矿场` 后，`planet-1-1` scene 中会出现新的 `mining_machine`。
4. 若仍失败，thread 中必须公开可操作的真实失败原因，而不是只有规划性中文回复。

## 完成情况

- 已修复 `POST /agents/:agentId/messages` 的直连 thread 执行链路：`agent.create / agent.update / conversation.ensure_dm / conversation.send_message` 的 tool 结果现在会持久化到 thread，后续消息可直接复用真实成员 ID 与委派证据。
- 已为直连 thread 补齐最近一次 turn 摘要：`status`、`outcomeKind`、`executedActionCount`、`repairCount`、`errorCode/errorMessage/rawErrorMessage`、`finalMessage`。
- 已修复 `classifyTurnIntent` 对“创建胡景，并赋予其建筑权限”这类自然中文的识别，避免把成员创建误判成普通回复而跳过 repair。
- 已更新 `client-cli` 的 `agent_thread` 输出与开发文档，CLI 现在会直接打印最近一次 turn 状态、失败码、执行动作数以及 thread 中真实 tool 证据。
