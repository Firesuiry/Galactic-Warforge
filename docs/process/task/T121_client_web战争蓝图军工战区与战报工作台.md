# T121 client-web 战争蓝图、军工、战区与战报工作台

## 背景

战争系统如果只存在服务端和 CLI，仍然无法满足本项目“主要用 Web 游玩和验证”的要求。设计文档已经明确 `client-web` 至少要新增：

1. 蓝图工作台
2. 军工总览
3. 战区面板
4. 战报与情报面板

这不是单纯补几个按钮，而是要把战争系统的操作入口组织成可理解、可发现、可验证的工作流。

## 设计参考

- `docs/dev/战争组件设计.md`
  - 第 17 章“Web 交互建议”
- `docs/superpowers/specs/2026-03-26-client-web-grand-strategy-redesign-design.md`

## 需求细节

1. Web 端必须能覆盖战争系统核心工作流，而不是只做查询。
2. 蓝图、军工、战区、战报必须是长期工作台，而不是散落在各页面的小表单。
3. 操作设计要继续遵循项目当前大战略 UI 方向，而不是退回研发调试面板。

## 改动需求

1. 新增或重构以下工作台：
   - 蓝图工作台：底盘、组件、约束校验、角色预估
   - 军工总览：产线、组件库存、翻修、部署状态
   - 战区面板：任务群、姿态、目标区域、制轨 / 封锁状态
   - 战报与情报面板：接触报告、战损、拦截统计、补给异常
2. Web 工作流至少支持：
   - 创建并校验蓝图
   - 查看量产和翻修进度
   - 调整任务群姿态
   - 查看 contacts / supply / battle_report
   - 发起或跟踪登陆 / 封锁
3. 浏览器交互需要保持移动端和桌面端的基本可操作性，且不能破坏现有大战略骨架。
4. 重要错误必须可解释：
   - 蓝图非法原因
   - 无法部署原因
   - 无法登陆原因
   - 断补给原因

## 不在本任务范围

- 不要求完成 AI agent 自治
- 不要求完成最终玩家文档
- 不要求把 Web 做成全即时手操 RTS

## 验收标准

1. Web 端存在稳定的战争工作台，而不是只能靠 CLI 或原始 API 操作。
2. 玩家可在浏览器完成战争系统核心工作流的主要步骤。
3. 重要错误和约束具备可操作的解释，不需要读源码。
4. 桌面端和移动端至少都能完成最小战争操作闭环。
