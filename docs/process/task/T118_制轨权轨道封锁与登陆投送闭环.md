# T118 制轨权、轨道封锁与登陆投送闭环

## 背景

战争设计文档要求恒星系层不能只停在“舰队能攻击某个行星目标”，还必须承担：

- 制轨权
- 轨道封锁
- 运输线拦截
- 登陆窗口开启
- 滩头阵地建立

如果没有这一步，三层地图仍然是割裂的：

- 恒星系层只是查看页面
- 行星层只能本地打地面战
- 舰队无法真正改变地面战局

## 设计参考

- `docs/dev/战争组件设计.md`
  - 第 13 章“制空、制轨与登陆”
  - 第 5 章“战争层级模型”

## 需求细节

1. 恒星系层必须存在 authoritative 的 `orbital_superiority` / `blockade_state`。
2. 舰队必须能拦截：
   - 补给线
   - 运输线
   - 登陆投送
3. 登陆必须是独立流程，而不是把 `fleet_attack` 目标指到星球就算完成。

## 改动需求

1. 在恒星系层引入 authoritative 的制轨权状态，至少体现：
   - 当前优势方
   - 争夺强度
   - 最近变化原因
2. 引入轨道封锁与拦截逻辑，至少覆盖：
   - 运输舰 / 补给舰被拦截
   - 封锁导致行星补给受损
   - 封锁与制轨权联动
3. 新增登陆投送流程，至少拆成：
   - 侦察登陆点
   - 打开登陆窗口
   - 投送前锋
   - 建立滩头阵地
   - 后续重装与补给接续
4. 登陆必须受以下条件约束：
   - 运输能力
   - 局部制轨
   - 登陆点安全
   - 初始补给充足
5. Query 层至少可查询：
   - 某恒星系当前制轨权
   - 某行星是否被封锁
   - 某次登陆投送的阶段和结果

## 不在本任务范围

- 不要求完成地面前线推进 UI
- 不要求完成 Web 战区工作台
- 不要求完成 AI 总参谋

## 验收标准

1. 恒星系层存在可查询、可测试的制轨权与封锁状态。
2. 封锁会真实影响运输、补给或投送，而不是只显示一个标签。
3. 登陆具备独立流程与失败条件，不能退化成“攻击星球 = 已登陆”。
4. 恒星系层结果能真实改变后续行星层战局入口。
