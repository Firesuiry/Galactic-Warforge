# T113 战争蓝图量产、改型翻修与部署枢纽闭环

## 背景

蓝图目录和校验落地后，下一步必须让蓝图真正进入工业体系。当前仓库的战争单位仍以：

- `deploy_squad`
- `commission_fleet`

为主，但它们依赖固定载荷 item，无法表达：

- 蓝图量产
- 改型切换
- 旧单位翻修
- 不同部署枢纽的产能与约束

战争系统如果不能和工厂、船坞、部署枢纽、背包 / 建筑库存连接起来，就仍然只是“更复杂的单位表”。

## 需求细节

1. 蓝图必须可被量产。
2. 已有单位必须支持受控翻修 / 改型升级。
3. 部署枢纽必须从“固定生成某几种单位”演进成“消耗蓝图产物并生成 runtime 单位”。
4. 现有 `prototype`、`precision_drone`、`corvette`、`destroyer` 路线要迁移到新闭环，不允许新旧两套部署方式长期并存。

## 改动需求

1. 建立蓝图量产链路，至少区分：
   - 组件生产
   - 总装
   - 部署
2. 至少支持以下 authoritative 能力：
   - 以蓝图为目标排产
   - 查询生产进度
   - 完成后获得可部署的军备产物
   - 通过部署枢纽把产物转成 runtime 单位
3. 翻修 / 改型必须支持：
   - 已部署单位返回维修 / 翻修设施
   - 切换到允许的目标蓝图
   - 消耗额外材料与时间
   - 查询翻修进度和结果
4. 量产效率要能表达：
   - 同蓝图连续量产收益
   - 切换产线的重整成本
5. 部署枢纽至少要明确：
   - 可部署域（地面 / 轨道 / 太空）
   - 需要的库存 / 载荷
   - 需要的供电 / 状态
   - 生成后的 runtime 身份与 blueprint 关联
6. 现有固定载荷闭环要收口：
   - 不能再出现旧命令只认识固定 item，而新 runtime 只认识 blueprint 的双轨状态

## 不在本任务范围

- 不要求完成战区和姿态
- 不要求完成补给惩罚
- 不要求完成 Web 军工工作台

## 验收标准

1. 玩家可基于已定型蓝图排产、完成总装并部署到 runtime。
2. 生产结果与 runtime 单位可稳定追溯到 `blueprint_id`。
3. 已部署单位可进入翻修 / 改型流程，并在完成后以目标蓝图返回。
4. 现有公开战争单位不再依赖旧式固定产物部署逻辑长期并存。
5. 保存 / 读档 / 回放后，量产、翻修、部署状态保持一致。
