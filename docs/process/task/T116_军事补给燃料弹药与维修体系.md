# T116 军事补给、燃料、弹药与维修体系

## 背景

当前战争单位的持续作战能力仍然过于抽象，缺少：

- 弹药
- 导弹库存
- 燃料
- 维修件
- 护盾补充材料
- 前线维修

设计文档明确要求战争系统不能只比拼纸面 DPS，而要受到：

- 补给网络
- 维护成本
- 维修能力
- 持续消耗

如果这层没有落地，蓝图设计器和高阶舰队很容易退化成“战斗数值越大越强”的单轴系统。

## 设计参考

- `docs/dev/战争组件设计.md`
  - 第 11 章“补给、燃料与维修”
  - 第 20 章“平衡基线”

## 需求细节

1. 战争单位必须消耗军事补给，而不是无限持续战斗。
2. 断补给时，单位应逐级退化，而不是瞬间失效。
3. 维修必须分层：
   - 战场应急修复
   - 前线维修站
   - 后方完整翻修

## 改动需求

1. 至少引入以下战争补给维度：
   - `ammo`
   - `missiles`
   - `fuel`
   - `spare_parts`
   - `shield_cells`
   - `repair_drones`
2. 明确补给来源：
   - 行星物流站
   - 星际物流站
   - 轨道补给港
   - 补给舰
   - 前线补给车 / 空投
3. 明确断补给退化顺序，至少体现：
   - 弹药短缺
   - 护盾恢复变慢
   - 维修停止
   - 机动下降
   - 凝聚度下降 / 被迫撤退
4. 明确维修层级的能力边界：
   - 各层级能恢复什么
   - 需要什么消耗
   - 多久可完成
5. Query 层至少支持：
   - 任务群 / 舰队补给状态
   - 某补给节点当前军需库存
   - 某单位当前维修状态

## 不在本任务范围

- 不要求完成导弹 / 点防战报
- 不要求完成 Web 军工总览
- 不要求完成 AI 后勤参谋

## 验收标准

1. 战争单位存在真实补给与维修消耗，而不是无限续航。
2. 断补给会导致多阶段退化，不是单一布尔停机。
3. 前线维修与后方翻修具备清晰边界和真实时间 / 物料成本。
4. Query 层可稳定观察补给与维修状态。
