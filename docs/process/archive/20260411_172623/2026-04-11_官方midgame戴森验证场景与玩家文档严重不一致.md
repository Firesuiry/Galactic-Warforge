# 2026-04-11 官方 midgame 戴森验证场景与玩家文档严重不一致

## 背景

- 试玩日期：2026-04-11
- 试玩方式：`client-web` 为主，`client-cli` 与直接 API 查询辅助核对
- 环境：
  - 新局服务端：`http://127.0.0.1:18081`
  - midgame 服务端：`http://127.0.0.1:18082`
  - Web：`http://127.0.0.1:4174`
- 使用配置：
  - `server/config-midgame.yaml`
  - `server/map-midgame.yaml`

当前玩家文档和联调文档都把官方 midgame 描述成“已预置戴森与中后期建筑、可直接做戴森回归”的验证场景，但本轮真实启动后，场景实际状态并不符合该描述。

## 实测结果

### 1. 启动后并不存在文档宣称的预置中后期建筑

实测 `summary` 与 `scene`：

- `active_planet_id = planet-1-2`
- `planet-1-2.kind = gas_giant`
- 但 `planet-1-2` 初始只有 `battlefield_analysis_base`
- `planet-1-1` 初始也只有 `battlefield_analysis_base`

没有出现文档里宣称应可直接验证的预置建筑，例如：

- `orbital_collector`
- `vertical_launching_silo`
- `em_rail_ejector`
- `ray_receiver`
- `jammer_tower`
- `sr_plasma_turret`
- `planetary_shield_generator`
- `self_evolution_lab`

### 2. 初始 `system runtime` 也没有戴森运行态

实际启动后：

- `/world/systems/sys-1/runtime.available = false`
- `active_planet_context` 中：
  - `em_rail_ejector_count = 0`
  - `vertical_launching_silo_count = 0`
  - `ray_receiver_count = 0`

这与文档中“启动后可直接在 system 页看到戴森态势和支撑建筑”的口径不一致。

### 3. 玩法本体并非缺失，而是官方验证场景没有预铺

本轮继续手工补建后，以下内容已被确认可用：

- 可建：
  - `orbital_collector`
  - `vertical_launching_silo`
  - `em_rail_ejector`
  - `ray_receiver`
  - `jammer_tower`
  - `sr_plasma_turret`
  - `planetary_shield_generator`
  - `self_evolution_lab`
  - `advanced_mining_machine`
  - `pile_sorter`
  - `recomposing_assembler`
- 可执行：
  - `build_dyson_node`
  - `build_dyson_frame`
  - `build_dyson_shell`
  - `launch_solar_sail`
  - `launch_rocket`
  - `set_ray_receiver_mode`
- 在手工建完脚手架并发射后：
  - `/world/systems/sys-1/runtime.available = true`
  - `dyson_sphere.layers[0].energy_output = 612`
  - `solar_sail_orbit.total_energy = 10`

也就是说，问题不在“中后期功能不存在”，而在“官方 midgame 场景没有按文档预置成可直接验证的状态”。

## 影响

- 玩家或测试者按照 `docs/player/上手与验证.md`、`docs/player/玩法指南.md`、`docs/dev/client-web.md` 进入官方 midgame 后，会误以为戴森链路没有实现
- Web `/system/sys-1` 首次打开时看不到文档宣称的戴森态势，验证路径被打断
- 智能体或人工测试都必须先手工铺电网、补建筑、补戴森脚手架，才能进入真正的中后期回归
- 文档、场景和真实实现口径已经分叉

## 改动需求

1. 二选一明确收敛，不要继续让“文档说已预置、实际未预置”的状态并存。
2. 如果保留“官方 midgame 为可直接验证戴森链路”的定位：
   - 在启动阶段真实预置以下最小闭环：
     - 供电
     - `vertical_launching_silo`
     - `em_rail_ejector`
     - `ray_receiver`
     - 至少一个可见戴森层脚手架
   - 确保 `/world/systems/sys-1/runtime.available = true`
3. 如果决定不再预置：
   - 全面回写玩家文档与 Web 联调文档
   - 明确写成“这是高科技解锁场景，不是预铺完成的戴森验证场景”
   - 给出真实的手工补建步骤和最小资源/电力要求
4. 补自动化回归：
   - midgame 启动后校验预期场景是否存在
   - 校验 `system runtime` 是否与文档定位一致

## 验收标准

- 官方 midgame 的实际状态与文档口径完全一致
- 测试者不需要先靠读源码才能知道“这局到底是不是预置戴森验证场景”
- `/system/sys-1` 首屏展示与文档描述一致，不再出现“文档说能看，实际全空”的情况
