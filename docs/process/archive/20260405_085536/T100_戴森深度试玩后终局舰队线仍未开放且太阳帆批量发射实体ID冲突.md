## 当前仍未实现 / 新发现的问题

### 问题 1：终局高阶舰队线仍未对玩家开放，当前版本不能宣称 DSP 相关建筑、科技树、玩法“都已实现”

#### 复现

1. 在官方 midgame 局查询 `/catalog`
2. 观察以下科技：
   - `prototype`
   - `precision_drone`
   - `corvette`
   - `destroyer`
3. 在 CLI 执行：
   - `help produce`
   - `produce u-2 corvette`
4. 代码侧复核：
   - `client-cli/src/commands/action.ts`
   - `shared-client/src/api.ts`
   - `server/internal/model/entity.go`

#### 实际现象

- `/catalog.techs` 中：
  - `prototype.hidden = true`
  - `precision_drone.hidden = true`
  - `corvette.hidden = true`
  - `destroyer.hidden = true`
- CLI `help produce` 仍只暴露：
  - `Produce unit (worker/soldier)`
- CLI 实测：
  - `produce u-2 corvette`
  - 返回 `unit_type 必须是 worker 或 soldier`
- 代码侧当前公开单位面也仍停留在：
  - `shared-client/src/api.ts`：`UnitTypeName = 'worker' | 'soldier'`
  - `client-cli/src/commands/action.ts`：`UNIT_TYPES` 只有 `worker`、`soldier`
  - `server/internal/model/entity.go`：仅定义 `worker`、`soldier`、`executor`
- 当前没有公开的：
  - 高阶单位生产入口
  - 高阶单位部署/编队入口
  - 轨道/太空单位查询入口
  - 对应玩家可观察战斗闭环

#### 影响

- 当前版本虽然已经覆盖了大量 DSP 中后期建筑与戴森主链，但终局高阶舰队线仍然没有进入玩家可用玩法。
- 因此不能继续把当前实现表述成“戴森球计划相关建筑、科技树、玩法都已经实现”。
- 这会直接误导后续试玩结论、能力盘点和玩家预期。

#### 改动要求

- 对以下终局科技 / 玩法线做明确收口：
  - `prototype`
  - `precision_drone`
  - `corvette`
  - `destroyer`
- 必须二选一并彻底落到代码、CLI、API、文档一致：
  - 方案 A：真正实现公开玩法线
    - 公开科技树显示
    - 玩家可用的生产入口
    - 部署 / 编队 / 查询入口
    - 对应事件、战斗结算与可观察结果
  - 方案 B：继续隐藏
    - 但同步修改所有“当前已经完整覆盖 DSP 科技树 / 终局玩法”的文档口径
    - 不再对外宣称“都已实现”

### 问题 2：太阳帆批量发射时，多个太阳帆仍共用同一个 `entity_id`

#### 复现

1. 在官方 midgame 局完成以下链路：
   - `build 3 2 em_rail_ejector`
   - `transfer b-30 solar_sail 4`
2. 执行：
   - `launch_solar_sail b-30 --count 2 --orbit-radius 1.2`
3. 查询：
   - `/events/snapshot?event_types=entity_created&limit=20`
4. 代码侧复核：
   - `server/internal/gamecore/solar_sail_settlement.go`

#### 实际现象

- CLI SSE 在同一 tick `5326` 连续收到两条 `entity_created`
- 两条事件都使用相同的：
  - `entity_id = sail-p1-5326`
- 事件快照里可直接看到：
  - `evt-5326-13 -> sail-p1-5326`
  - `evt-5326-14 -> sail-p1-5326`
- 代码侧当前实现为：
  - `LaunchSolarSail(...)`
  - `ID: "sail-" + playerID + "-" + strconv.FormatInt(launchTick, 10)`
- 因此只要同一玩家在同一 tick 内发射多张帆，就会天然冲突

#### 影响

- 事件流无法稳定区分同批次发射的多张太阳帆
- 后续回放、销毁事件、可视化和调试都会出现对象歧义
- 这会污染戴森中后期回归测试与 AI/脚本驱动试玩结论

#### 改动要求

- 太阳帆实体 ID 必须保证同 tick 批量发射时仍然全局唯一
- 至少需要补齐：
  - 太阳帆创建 ID 生成策略
  - 轨道状态内的成员 ID 一致性
  - `entity_created` / `entity_destroyed` / 回放链路的一致性
- 增加自动化测试，覆盖：
  - 同一玩家
  - 同一 tick
  - `launch_solar_sail --count >= 2`
  - 返回的每个太阳帆 `entity_id` 都唯一

## 验收标准

1. 对终局高阶舰队线，仓库口径必须统一为以下二者之一：
   - 玩家已经能通过公开科技树 + 公开命令真实体验该玩法
   - 当前仍未实现，且所有玩家侧文档明确写清，不再宣称“都已实现”
2. `produce`、CLI 帮助、API 类型定义、服务端能力模型不能再彼此矛盾：
   - 要么高阶单位真正开放
   - 要么所有玩家入口明确不支持，并与文档口径一致
3. 同一玩家在同一 tick 内批量发射 2 张及以上太阳帆时：
   - 每张帆都有唯一 `entity_id`
   - `entity_created` 事件不再冲突
   - 后续生命周期事件与回放也能稳定定位到具体太阳帆
4. 本轮已确认可用的链路不能回退：
   - 默认新局真实矩阵科研门禁
   - `switch_active_planet`
   - `build_dyson_*`
   - `launch_solar_sail`
   - `launch_rocket`
   - `set_ray_receiver_mode power`
   - `photon` 模式的 `dirac_inversion` 门禁
