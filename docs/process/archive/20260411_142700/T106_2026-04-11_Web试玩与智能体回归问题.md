## 当前问题列表

### 1. Web 行星页对“建造距离 / 供电约束”的引导不够优雅，导致中后期建造链路很容易误操作

- 真实现象：
  - midgame Web 首轮尝试 `orbital_collector` 时直接收到 authoritative 失败：`executor out of range: 7 > 6`
  - 同一轮 Web 新建的 `jammer_tower`、`recomposing_assembler`、`advanced_mining_machine` 都真实落库了，但处于 `no_power / power_out_of_range`
  - CLI 继续补测后，`planetary_shield_generator` 虽然可以建成且 `running`，`self_evolution_lab` 也能带 `electromagnetic_matrix` 配方建出，但仍可能进入 `under_power`
- 影响：
  - 功能本身并没有缺失，但 Web 工作流没有把“先移动执行体”“先补电网”“当前坐标是否在建造范围内”表达清楚，用户很容易误以为建筑没实现或 UI 坏了
  - 对中后期验证尤其不友好，因为这类建筑的电力/距离约束比前期更敏感
- 改动要求：
  - 行星页建造工作流需要显式展示当前执行体可达范围，或至少在提交前给出距离预检提示
  - 建造失败时，UI 应把 authoritative 原因直接翻译成可操作提示，而不是只留在命令结果账本里
  - 对新建建筑的 `power_out_of_range / under_power` 需要有更直接的工作台提示，告诉玩家下一步该补塔还是补发电

### 2. 默认新局 Web 建造下拉仍直接暴露大量未解锁中后期建筑，页面可发现性与操作收敛度不够

- 真实现象：
  - 默认新局 `client-web` 行星页建造下拉本轮仍有 `60` 个选项
  - 证据里明确包含 `advanced_mining_machine`、`ray_receiver` 等开局显然不该直接呈现的中后期建筑
- 影响：
  - 用户在前期页面上看到大量不可用或不该优先关注的建筑，会显著稀释操作焦点
  - 这会让 Web 操作台看起来像调试面板，而不是收敛后的玩家工作流
- 改动要求：
  - 建造列表应按当前已解锁科技收口，至少默认只突出当前可建/推荐建内容
  - 如需保留“全量建筑”调试能力，也应改为显式切换的高级模式，不应默认暴露给普通玩家工作流

### 3. 研究页真实点击仍存在 pointer interception，Web 交互不稳定

- 真实现象：
  - 默认新局真实浏览器中，点击研究卡片会被其它区域拦截
  - 证据见 `.run/manual-playtest/evidence/default-web/report.json` 中 `research.selectIssue`
- 影响：
  - 虽然当前因为卡片默认选中，仍可继续点“开始研究”绕过去，但这不是稳定可用的交互
  - 会直接降低 Web 端“科技树可正常游玩”的可信度
- 改动要求：
  - 修正研究卡片层级、滚动容器和遮挡关系，确保真实点击命中卡片本身
  - Playwright 应增加真实点击回归，而不是只覆盖最终按钮能否点击

### 4. 智能体“观察”链路能执行动作，但最终回复不闭环

- 真实现象：
  - 在 `/agents` 私聊里要求李斯“先观察 planet-1-1 当前局势，只回复一句话总结”
  - turn 状态为 `succeeded`
  - 但最终回复仍是“已提交扫描请求，待结果返回后再总结”，没有交付用户要求的一句话总结
- 影响：
  - 这会让前端看起来像“agent 已成功”，但玩家拿不到真正任务结果
  - 协作体验上属于半完成态冒充完成态
- 改动要求：
  - 对 observe 类任务，agent runtime 必须要求 provider 回到最终总结，而不是只回“已提交”
  - 若 provider 只完成中间动作但没给最终答复，应继续收口而不是直接判 turn succeeded

### 5. 智能体创建下级成员链路未闭环

- 真实现象：
  - 在 `/agents` 中让李斯创建下级智能体“胡景”失败
  - turn `failed`
  - `errorCode = provider_incomplete_execution`
  - `rawError = 这轮只有规划，没有执行所需动作。`
  - authoritative 结果确认没有创建出 `胡景`
- 影响：
  - “创建成员 -> 委派成员”是智能体工作台最核心的协作场景之一，当前仍不可稳定使用
- 改动要求：
  - provider repair 与 runtime 完成判定需要继续收口，确保 `agent.create` 不会停留在“只规划不执行”
  - 前端需要把失败原因清楚暴露给用户，而不是让用户只看到 turn 失败

### 6. 智能体科研任务仍会构造错误参数，前端错误提示也不够可操作

- 真实现象：
  - 要求李斯“把 10 个 electromagnetic_matrix 装入 b-9，然后启动 basic_logistics_system 研究”
  - `/agents` 页面只显示泛化错误：`执行失败，请稍后重试。`
  - `agent-gateway` 实际错误是：`transfer_item requires buildingId`
- 影响：
  - 这不是单纯模型答非所问，而是 agent 动作参数组装仍不稳定
  - 前端把真实错误吞成泛化提示，用户无法判断是权限问题、模型问题还是参数问题
- 改动要求：
  - `agent-gateway` 需要修正 research + transfer 组合任务里的参数构造
  - `/agents` 页面应尽量展示真实 `errorCode / errorMessage`，至少对参数缺失类错误给出可操作文案

## 改动目标

- 保持当前已确认可用的戴森主链、Web 命令入口和 midgame 建筑实现不回退
- 把 Web 行星页从“调试表单堆叠”继续收口成真正可操作的玩家工作流
- 把智能体工作台从“能显示 turn”推进到“能可靠完成观察 / 建造 / 创建下级 / 科研委派”

## 验收标准

1. 默认新局 Web 建造列表不再默认暴露未解锁的大量中后期建筑；前期页面的可操作项应明显收敛。
2. 研究页科技卡片在真实浏览器中可稳定点击，不再出现 pointer interception。
3. midgame Web 中，当玩家尝试超出执行体建造范围或新建建筑缺电时，页面能直接给出下一步引导，而不是让用户自行猜测。
4. `/agents` 中 observe 任务只有在给出最终总结后才算成功完成。
5. `/agents` 中创建下级成员链路可真实创建成功，并能在 UI 中立即看到结果。
6. `/agents` 中科研委派能真实完成 `transfer_item + start_research`，或在失败时把具体错误暴露给用户。
