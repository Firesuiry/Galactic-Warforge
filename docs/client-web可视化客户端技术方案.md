# SiliconWorld Web 可视化客户端技术方案

## 1. 目标与背景

当前 `server` 已经具备较完整的查询接口、事件流、迷雾与回放能力，但缺少一个适合人类与 AI 共同观察局势的可视化客户端。现有 `client-cli` 更适合命令行操作和接口调试，不适合以下场景：

- 快速查看整颗行星的建筑布局、资源点分布、单位位置与迷雾变化。
- 观察一段时间内建筑状态、战斗、资源变化、施工推进的时序关系。
- 为 AI 提供更稳定的“当前局势画面”，便于判断扩张、防御、供电、物流是否正常。
- 通过截图、回放、事件时间线复现显示问题与玩法问题。

本方案的目标是新增一个基于 Web 的 `client-web`，作为当前 `server` 的专用可视化观察端，并为后续逐步扩展为“可视化操作端”打基础。

## 2. 当前服务端可直接复用的能力

### 2.1 已存在的查询与流式能力

当前 `server` 已经提供以下能力，可直接支撑 Web 端第一阶段实现：

- 世界摘要：`GET /state/summary`
- 玩家统计：`GET /state/stats`
- 银河总览：`GET /world/galaxy`
- 恒星系详情：`GET /world/systems/{system_id}`
- 行星详情：`GET /world/planets/{planet_id}`
- 行星迷雾：`GET /world/planets/{planet_id}/fog`
- 实时事件流：`GET /events/stream`
- 事件补拉：`GET /events/snapshot`
- 产线告警快照：`GET /alerts/production/snapshot`
- 回放：`POST /replay`
- 回滚：`POST /rollback`

这些接口定义集中在 `server/internal/gateway/server.go`，查询视图主要定义在 `server/internal/query/query.go`。

### 2.2 当前前端可直接显示的数据

行星详情视图已经包含：

- `terrain`
- `buildings`
- `units`
- `resources`
- `environment`
- `tick`

迷雾视图已经包含：

- `visible`
- `explored`

建筑、单位、资源节点已经具备足够的原始字段，可支撑地图渲染与详情面板：

- 建筑：`id`、`type`、`owner_id`、`position`、`runtime`、`storage`、`production`、`job`
- 单位：`id`、`type`、`owner_id`、`position`、`hp`、`attack`、`defense`
- 资源：`kind`、`behavior`、`position`、`remaining`、`current_yield`

### 2.3 当前服务端的可视化缺口

`WorldState` 中已经维护但尚未完整对外暴露的运行时数据包括：

- `LogisticsDrones`
- `LogisticsShips`
- `PowerGrid`
- `Pipelines`
- `Construction`
- `EnemyForces`
- `Detections`

这意味着第一阶段 Web 客户端可以稳定实现：

- 地形查看
- 资源点查看
- 建筑/单位查看
- 迷雾查看
- 事件/告警查看
- 玩家资源与统计查看

但以下高级图层暂时不能完整可视化：

- 物流无人机与货船实时轨迹
- 电网拓扑与供电覆盖
- 管线网络
- 施工队列区域状态
- 敌对势力范围与检测圈

因此本方案采用“两阶段数据策略”：

1. **第一阶段优先使用现有接口完成可用 MVP。**
2. **第二阶段补齐面向可视化的只读 read model 接口，而不是继续在前端硬拼接零散数据。**

## 3. 总体设计原则

### 3.1 先做观察端，再做操作端

第一阶段只解决“看得清”：

- 当前局势
- 建筑状态
- 迷雾变化
- 战斗与事件
- 告警与异常

第二阶段再补“点得动”：

- 命令面板
- 建造辅助
- 调试回放控制
- 未来 AI 操作入口

### 3.2 前端围绕“场景视图”组织，而不是围绕单接口堆页面

Web 客户端的核心页面不是简单罗列接口，而是围绕三个观察场景：

- 星图场景：银河 / 星系 / 行星发现状态
- 行星场景：地图、建筑、单位、资源、迷雾
- 调试场景：事件、告警、回放、快照比对

### 3.3 同步模型采用“全量快照 + 事件增量 + 周期纠偏”

只靠 SSE 不够，因为部分事件并不是完整实体快照；只靠轮询也不够，因为局势变化有时效要求。最佳策略是：

- 首次进入页面：拉全量快照
- 运行中：订阅 SSE 增量
- 断线重连：用 `events/snapshot` 补事件
- 周期性：重新拉取整页快照，修正潜在漂移

### 3.4 优先保证“AI 可读性”

地图渲染与调试视图不仅服务于人，也服务于后续 AI 观察。因此 UI 设计需要满足：

- 颜色高对比
- 图层可开关
- 坐标明确
- 选中对象有稳定 JSON
- 截图结果稳定可复现
- URL 能表达当前视角与图层状态

## 4. 推荐技术选型

## 4.1 核心栈

- `React + TypeScript`
- `Vite`
- `React Router`
- `PixiJS`
- `TanStack Query`
- `Zustand`
- `Vitest`
- `Playwright`

### 4.2 选型理由

#### React + TypeScript

用于承载页面结构、状态面板、调试面板、表格、侧栏与配置界面。当前项目已有 `client-cli` 的 TypeScript 基础，继续使用 TS 成本最低。

#### Vite

适合本地高频调试，启动快，热更新快，并且开发期可以通过 proxy 把 `/state`、`/world`、`/events` 等请求转发到 Go 服务端，避免 CORS 与 SSE 跨域问题。

#### React Router

用于承载以下页面层级：

- `/login`
- `/overview`
- `/galaxy`
- `/system/:systemId`
- `/planet/:planetId`
- `/replay`

#### PixiJS

地图主视图推荐使用 PixiJS，而不是纯 DOM 或 SVG。原因：

- 地图本质是网格、多图层、缩放平移、批量渲染。
- 后续要叠加迷雾、资源点、建筑、单位、告警描边、变化闪烁。
- 将来扩展电网、物流路径、管线、敌情圈时，Canvas/WebGL 路线更稳定。

第一阶段不强求美术资源，可先使用几何图元、文本与颜色编码渲染。

#### TanStack Query

适合管理以下读模型：

- 行星详情快照
- 迷雾快照
- 统计面板
- 事件补拉
- 告警补拉
- 回放结果

它可以天然处理缓存、失效、重拉、错误恢复。

#### Zustand

适合保存不应混入服务端缓存的数据：

- 当前相机位置与缩放
- 选中实体
- 图层显隐
- 调试开关
- 最近事件 ID
- 当前 Hover 坐标

## 5. 项目结构方案

推荐直接新增两个目录，而不是继续复制粘贴 `client-cli` 的逻辑：

```text
client-web/
shared-client/
```

职责划分如下：

### 5.1 `shared-client/`

抽取当前 CLI 中可复用的前后端通信逻辑，供 CLI 与 Web 共用：

- `types/`
- `api/`
- `sse/`
- `event-reducer/`
- `auth/`

优先抽取这些文件中的共性：

- `client-cli/src/api.ts`
- `client-cli/src/types.ts`
- `client-cli/src/sse.ts`

这样后续 CLI 与 Web 使用同一套接口类型，不再维护两份相似代码。

### 5.2 `client-web/`

仅保留浏览器端特有逻辑：

- 页面路由
- Pixi 地图渲染
- 详情侧栏
- 时间线与告警面板
- 调试工具栏
- 截图与回放视图

推荐目录结构：

```text
client-web/
  src/
    app/
    pages/
    widgets/
    features/
      auth/
      galaxy/
      planet-map/
      entity-panel/
      events/
      alerts/
      replay/
      debug-tools/
    stores/
    hooks/
    styles/
```

## 6. 页面与功能设计

### 6.1 登录页

输入项：

- 服务端地址
- `player_id`
- `player_key`

特性：

- 默认提供开发环境快捷填充：`p1 / key_player_1`、`p2 / key_player_2`
- 支持把地址与玩家信息缓存到本地
- 连接成功后进入总览页

### 6.2 总览页

展示：

- 当前 tick
- 当前玩家资源
- 当前研究状态
- 当前 active planet
- 产线、电力、物流、战斗统计
- 最近关键事件
- 最近告警

用途：

- 作为所有页面的统一入口
- 作为 AI 查看“全局摘要”的第一屏

### 6.3 星图页

展示：

- 银河总览
- 已发现 / 未发现恒星系
- 恒星系坐标与距离关系
- 某星系内已发现行星

交互：

- 点击星系进入恒星系详情
- 点击行星进入行星地图页
- 未发现内容以占位形式显示

### 6.4 行星地图页

这是第一阶段的核心页面。

布局建议：

- 中央：Pixi 地图
- 左侧：图层与过滤器
- 右侧：实体详情 / 调试信息
- 下方：事件时间线 / 告警 / 命令结果

#### 地图图层

第一阶段：

- 地形层
- 资源层
- 建筑层
- 单位层
- 迷雾层
- 坐标网格层
- 选中高亮层

第二阶段：

- 电网层
- 物流轨迹层
- 管网层
- 施工层
- 敌情与探测层

#### 实体交互

- Hover：显示 ID、类型、所属玩家、坐标
- Click：固定选中并打开详情侧栏
- 双击：将视角居中到该实体
- 支持按类型过滤显示

#### 详情侧栏

按选中对象类型显示不同面板：

- 建筑：状态、停机原因、产线、库存、施工、升级、视野
- 单位：血量、攻击、防御、移动状态、目标
- 资源点：剩余量、行为类型、当前产率、是否稀有

### 6.5 回放页

围绕现有 `/replay` 能力构建调试视图：

- 输入 `from_tick` / `to_tick`
- 选择 `step` / `speed` / `verify`
- 查看 replay 执行结果
- 对 replay digest 与 snapshot digest 进行比对
- 支持跳转回某个 tick 的地图视角

这不是完整视频回放，而是“调试型回放控制页”。

## 7. 地图渲染设计

### 7.1 坐标体系

采用统一网格世界坐标：

- 服务端坐标：`x / y / z`
- 前端地图坐标：`tileX / tileY`
- 屏幕坐标：由相机变换计算

第一阶段统一把 `z` 当作保留字段，不参与平面渲染。

### 7.2 Tile 尺寸与缩放

建议默认 tile 尺寸：

- 迷你视图：`12 px`
- 标准视图：`24 px`
- 检查视图：`32 px`

允许用户滚轮缩放，缩放等级固定为若干档，避免任意缩放导致 AI 截图不稳定。

### 7.3 渲染样式建议

第一阶段先走“高可读调试风格”，不要一开始就做复杂美术：

- `buildable`：深灰底
- `blocked`：黑灰底
- `water`：蓝色
- `lava`：橙红色
- 资源点：按资源种类上色
- 我方建筑：青色描边
- 敌方建筑：红色描边
- 我方单位：绿色
- 敌方单位：红色
- 已探索未可见区域：半透明暗罩
- 未探索区域：纯黑罩

### 7.4 性能策略

第一阶段先实现正确性，再做优化，但基础性能策略应一开始就纳入：

- 地形与静态资源点分层缓存
- 建筑/单位/迷雾独立 layer
- 尽量局部刷新，不每帧重绘全部图层
- 事件驱动更新动态 layer
- 全量快照重建只在初始化或纠偏时触发

## 8. 数据同步与状态管理方案

### 8.1 初次加载流程

进入行星地图页时执行：

1. 拉 `summary`
2. 拉 `stats`
3. 拉 `planet`
4. 拉 `fog`
5. 拉 `alerts snapshot`
6. 拉最近一批 `events snapshot`
7. 建立 SSE 连接

### 8.2 SSE 订阅策略

浏览器端不直接使用原生 `EventSource`，而采用 `fetch + ReadableStream`：

- 原因一：当前服务端使用 `Authorization: Bearer` 鉴权
- 原因二：现有 CLI 已经验证了这套方式可行
- 原因三：后续更容易在浏览器端复用重连、补拉和调试逻辑

建议默认订阅事件：

- `command_result`
- `entity_created`
- `entity_moved`
- `entity_destroyed`
- `building_state_changed`
- `resource_changed`
- `production_alert`
- `research_completed`
- `threat_level_changed`
- `entity_updated`
- `tick_completed`

### 8.3 事件合并策略

前端不直接把 SSE 事件当“显示文本”，而要进入统一 reducer。

Reducer 负责：

- 合并实体创建 / 删除 / 移动
- 合并建筑状态变化
- 合并资源变化提示
- 维护最近事件窗口
- 维护最近告警窗口
- 记录 `last_event_id`

### 8.4 漂移修正策略

以下情况触发全量重拉：

- SSE 重连成功后
- `events/snapshot` 返回的 `available_from_tick` 超过前端本地缓存范围
- 切换行星
- 用户手动点击“重新同步”
- 每隔固定 tick 自动纠偏一次

### 8.5 状态分层

建议明确区分三类状态：

#### 服务端快照状态

由 TanStack Query 管理：

- 行星快照
- 迷雾快照
- 玩家统计
- 银河 / 恒星系 / 行星树

#### 本地派生运行态

由 Zustand 管理：

- 图层开关
- 选中实体
- 相机
- 最近事件
- 最近告警
- 视图过滤器

#### 渲染态

由 Pixi 场景对象管理：

- 缓存的 layer
- 当前高亮对象
- hover hit-test 结果

## 9. 建议新增的服务端只读接口

第一阶段不强依赖新增接口，但为了后续优雅实现，建议在第二阶段补齐以下 read model。

### 9.1 `GET /world/planets/{planet_id}/runtime`

用于返回当前行星的动态运行态：

- `logistics_drones`
- `logistics_ships`
- `construction_tasks`
- `enemy_forces`
- `detections`

### 9.2 `GET /world/planets/{planet_id}/networks`

用于返回网络型覆盖信息：

- `power_links`
- `power_coverage`
- `pipelines`

### 9.3 `GET /catalog/buildings`

用于返回前端展示所需元数据：

- 建筑名称
- 分类
- footprint
- 是否需要资源点
- icon key
- 默认颜色

### 9.4 `GET /catalog/items`

用于返回物品展示元数据：

- 物品名称
- 类别
- 图标 key
- 默认颜色

### 9.5 为什么不建议继续堆很多零碎接口

后续如果前端为了一个图层就加一个小接口，会导致：

- 查询层越来越碎
- 前端拼装逻辑越来越复杂
- CLI、Web、未来 AI 三套读模型继续分叉

因此推荐路线是：

- 第一阶段使用现有接口完成 MVP
- 第二阶段把“地图页真正需要的只读数据”收敛成少量 read model 接口

## 10. 与现有 CLI 的复用方案

当前 `client-cli` 已有三块非常适合抽取：

- API 调用
- SSE 连接与重连
- 通用类型定义

后续实现时推荐直接把这三块迁移到 `shared-client/`，然后：

- CLI 改为依赖 `shared-client`
- Web 改为依赖 `shared-client`

这样能避免：

- CLI 与 Web 类型不一致
- 一个接口改了要改两份 TS
- SSE 重连逻辑出现两套行为差异

## 11. 调试与显示验证方案

这是本项目里非常关键的一部分。

### 11.1 开发期联调方式

推荐开发模式：

- `server` 使用 `config-dev.yaml` 启动在 `18080`
- `client-web` 使用 Vite 启动在 `5173`
- 通过 Vite proxy 把 API 与 SSE 转发到 `18080`

优点：

- 浏览器端与服务端同机调试
- 不需要额外处理 CORS
- SSE 鉴权逻辑保持不变
- 能直接打开浏览器 DevTools 看网络与渲染

### 11.2 页面内置调试工具栏

建议客户端右上角常驻一个可折叠调试面板，显示：

- 当前服务器地址
- 当前玩家 ID
- 当前 tick
- SSE 连接状态
- 最后事件 ID
- 最近一次全量同步时间
- 当前相机坐标与缩放
- 当前 hover tile 坐标
- 当前选中实体 ID

同时提供以下按钮：

- 重新拉取当前行星快照
- 重新拉取迷雾
- 手动补拉事件
- 导出当前视图截图
- 导出当前选中对象 JSON

### 11.3 事件调试面板

事件面板除了显示文本，还要支持：

- 按事件类型过滤
- 点击事件跳转到对应实体
- 高亮该事件影响到的地图对象
- 查看事件原始 payload

这样可以快速定位：

- 为什么某建筑停机
- 为什么某单位消失
- 为什么某资源点变化
- 某次战斗命中的对象是谁

### 11.4 Storybook + Mock 数据

为了脱离服务端快速调显示问题，建议为以下组件单独做 Storybook：

- 地图画布
- 建筑详情卡
- 单位详情卡
- 资源点详情卡
- 迷雾图层
- 事件时间线
- 告警卡片

Mock 数据来源建议：

- 从真实服务端抓取一组 `planet` / `fog` / `events snapshot` JSON
- 固化为 `fixtures`
- 使用 MSW 在浏览器中模拟接口返回

这样显示 bug 可以不依赖实时跑服复现。

### 11.5 Playwright 截图回归

为了保证后续 UI 迭代不把地图显示弄坏，建议加 Playwright：

- 打开固定页面
- 切换固定图层组合
- 移动到固定坐标
- 截图
- 做视觉回归比对

重点覆盖：

- 行星地图基础显示
- 迷雾显示
- 建筑选中高亮
- 事件触发后的局部变化
- 告警面板

### 11.6 基于回放的显示调试

已有 `POST /replay`，因此可以建立“指定 tick 的可复现显示场景”：

- 先让服务端 replay 到指定 tick
- 客户端刷新并进入对应视角
- 截图并保存

这样可以把“偶发显示问题”转成“固定 tick 的重现场景”。

### 11.7 AI 友好的调试增强

为了方便 AI 使用客户端观察局势，建议额外支持：

- URL 中带 `planetId`、`x`、`y`、`zoom`、`layers`
- 一键复制当前视角链接
- 一键导出当前画面 PNG
- 一键导出当前画面对应的结构化摘要 JSON

结构化摘要 JSON 建议至少包含：

- 当前 tick
- 当前行星 ID
- 当前屏幕范围内建筑列表
- 当前屏幕范围内单位列表
- 当前图层开关
- 当前选中对象

## 12. 部署方案

### 12.1 开发环境

- `server` 独立运行
- `client-web` 用 Vite dev server
- 通过 proxy 访问服务端

### 12.2 生产 / 日常使用环境

推荐最终把 `client-web` 构建产物作为静态资源由 Go 服务端同源托管：

- 避免 CORS
- 避免 SSE 跨域鉴权差异
- 用户只启动一个服务
- AI 自动化只需面对一个入口地址

建议最终挂载路径：

- UI：`/ui/`
- API：保留现有路径

## 13. 分阶段落地策略

### 13.1 第一阶段：可用 MVP

目标：在不改服务端 read model 的前提下，把核心观察能力做出来。

完成后应具备：

- 登录
- 总览
- 星图查看
- 行星地图
- 建筑 / 单位 / 资源详情
- 迷雾显示
- SSE 实时更新
- 事件 / 告警面板
- 截图导出

### 13.2 第二阶段：运行态高级图层

目标：补齐物流、电网、管网、施工、敌情等运行态信息。

需要：

- 新增服务端 read model 接口
- 新增地图 overlay
- 新增过滤与着色规则

### 13.3 第三阶段：操作与回放增强

目标：让 Web 客户端不只可看，还能可控、可复现、可回归测试。

增加：

- 命令面板
- 常用操作按钮
- 回放控制台
- Playwright 视觉基线
- Storybook 场景库

## 14. 风险与对策

### 14.1 风险：前端事件合并逻辑过早复杂化

对策：

- 第一阶段只处理必要事件类型
- 无法可靠增量合并时直接触发局部或全量重拉

### 14.2 风险：服务端未暴露的运行态导致地图图层实现不完整

对策：

- 第一阶段只交付已有数据能支撑的图层
- 第二阶段集中补 read model，不走前端猜测

### 14.3 风险：地图渲染过度追求美术表现导致进度变慢

对策：

- 第一阶段全部使用几何图元与文字
- 把重点放在正确性、可读性、可调试性

### 14.4 风险：CLI 与 Web 双端类型漂移

对策：

- 第一个开发任务之一就是抽公共 `shared-client`
