# 上手与验证

## 1. 这份文档解决什么问题

如果你现在想真正进游戏，不管你是玩家、测试者，还是想先快速联调，这份文档给你最短路径：

- 怎么启动服务端
- 怎么进入 CLI 或 Web
- 第一轮应该验证什么
- 遇到问题后该看哪份文档

## 2. 启动本地环境

### 2.1 启动服务端

普通新局：

```bash
cd /home/firesuiry/develop/siliconWorld/server
env PATH=/home/firesuiry/sdk/go1.25.0/bin:$PATH \
  go run ./cmd/server -config config-dev.yaml -map-config map.yaml
```

官方中后期场景：

```bash

env PATH=/home/firesuiry/sdk/go1.25.0/bin:$PATH \
```

## 3. 先用哪种客户端

- 想快速推进科技、建造和排查命令链路：优先 `client-cli`
- 想看地图、详情、图层和操作回显：优先 `client-web`
- 想确认 Web 页面能不能正常显示操作结果：一定要进浏览器实看

最常见的组合是：

- CLI 推进命令和玩法
- Web 看局势、物流站配置和页面回显

## 4. 第一轮建议验证路径

### 4.1 最小可玩路径

1. 登录后执行 `summary`
2. 扫描银河 / 恒星系 / 行星
3. 先在基地正交相邻格建一台 `wind_turbine`
4. 再在基地另一侧建一台空 `matrix_lab`
5. 先直接执行一次 `start_research electromagnetism`，确认“有研究站但没装矩阵”时仍不会凭空开研
6. 把 `10` 个 `electromagnetic_matrix` 装入研究站本地存储
7. 再执行 `start_research electromagnetism`，确认研究成功后真实解锁的是 `tesla_tower + mining_machine`
8. 继续推进供电、采矿、物流与制造链
9. 再进入冶炼、组装、第二台 `matrix_lab --recipe electromagnetic_matrix` 与基础防御

当前默认图可复现实例：

```text
build 3 2 wind_turbine
build 2 3 matrix_lab
start_research electromagnetism
transfer <matrix_lab_id> electromagnetic_matrix 10
start_research electromagnetism
build 4 2 tesla_tower
build 5 1 mining_machine
```

说明：

- `build 3 2` / `build 2 3` 只是当前默认图的可复现实例，不是永远固定的世界规则
- `<matrix_lab_id>` 必须替换成这次运行里实际建出的研究站 ID，不要硬编码某次试玩的临时实体号
- 当前默认新局启动包已经是 `minerals = 240`；按这条路线走完后，首台研究站会保留，且还剩 `20 minerals`

### 4.2 官方 midgame 快速验证路径

1. 用 `config-midgame.yaml + map-midgame.yaml` 启动服务端
2. 登录后执行 `summary`，确认 `active_planet_id = planet-1-2`
3. 执行 `system sys-1`，确认 `planet-1-2.kind = gas_giant`
4. 执行 `switch_active_planet planet-1-1`，再切回 `switch_active_planet planet-1-2`，确认已加载且有 foothold 的星球可切换
5. 先在基地附近补足供电：
   先建 `tesla_tower`，再连续补 `wind_turbine`，直到 `stats.energy_stats.generation >= 84`
   当前官方 seed 的实测最小值是：基地原有发电 + `1` 座 `tesla_tower` + `8` 台 `wind_turbine`
6. 在已通电空地执行 `build <x> <y> orbital_collector`
7. 再执行 `build <x> <y> vertical_launching_silo` 与 `build <x> <y> em_rail_ejector`
8. 用 `inspect planet-1-2 building <building_id>` 找到发射井和电磁弹射器 ID，并确认它们进入 `running`
9. 用公开装填命令把背包装到建筑本地存储：
   `transfer <silo_id> small_carrier_rocket 1`
   `transfer <ejector_id> solar_sail 3`
10. 用 `build_dyson_node/frame/shell` 先建至少一个戴森层脚手架
11. 执行 `launch_solar_sail <ejector_id> --count 1`
12. 执行 `launch_rocket <silo_id> sys-1 --layer 0`
13. 在已通电空地分别执行 `build <x> <y> jammer_tower`、`build <x> <y> sr_plasma_turret`、`build <x> <y> planetary_shield_generator`
14. 再建一台空配方 `self_evolution_lab`，以及一台 `build <x> <y> self_evolution_lab --recipe electromagnetic_matrix`
15. 把执行体移动到 `(7,4)` 附近后，再试一轮官方 midgame 直测建筑：
   `build 8 6 recomposing_assembler`
   `build 8 7 pile_sorter`
   `build 10 7 advanced_mining_machine`
16. 用 `inspect planet-1-2 building <shield_id>` 确认 `runtime.functions.shield.current_charge` 会持续上升；如同场验证全部新增建筑，继续补风机保证它们进入 `running`
17. 对射线接收站执行 `set_ray_receiver_mode <receiver_id> power` 或 `hybrid`
18. 如需核对非当前 active 星球的运行态，直接调用服务端 `/world/planets/{planet_id}/runtime` 与 `/world/planets/{planet_id}/networks`
19. 观察 SSE / `event_snapshot` 中出现 `rocket_launched`，并留意 `damage_applied` 事件中的 `shield_absorbed` / `shield_remaining` 字段；随后用 `summary` / `stats` / `/world/planets/{planet_id}/networks` 一起确认 `ray_receiver power/hybrid` 已让玩家 `energy`、`generation` 与网络 `supply` 同步上涨

### 4.3 Web 最小回归项

- 能否正常登录
- 总览页是否能看到玩家与世界摘要
- 行星页能否显示地形、建筑、资源和迷雾
- 详情侧栏是否能正确回显实体状态、建筑建造结果和兵力调配结果
- 命令面板提交后，运行态、active planet 和当前局势是否同步更新

### 4.4 官方战争验证路径

这是一套给战争系统回归和试玩复用的 authoritative 官方验证局，不代表普通新局已经自然推进到这里。

1. 用 `config-war.yaml + map-war.yaml` 启动服务端
2. 登录 `p1 / key_player_1`
3. 先执行 `summary`，确认 `active_planet_id = planet-1-1`
4. 执行 `war_industry`，确认至少能看到：
   - `Orbital Supply Port`
   - `Planetary Logistics Station`
   - `Interstellar Logistics Station`
5. 执行蓝图与军工最小链路：
   - `blueprint_variant corvette corvette_verify utility --name 官方回归舰`
   - `blueprint_validate corvette_verify`
   - `blueprint_finalize corvette_verify --target-state prototype`
   - `queue_military_production <recomposing_assembler_id> <battlefield_analysis_base_id> corvette_verify --count 1`
   - 再次执行 `war_industry`，确认部署枢纽 `ready_payloads` 中出现 `corvette_verify`
   - `<recomposing_assembler_id>` 与 `<battlefield_analysis_base_id>` 可先用 `scene` / `inspect` 查，或直接在 Web 行星页 / `/war` 工作台里查看当前选中建筑 ID
6. 执行舰队、任务群与战区最小链路：
   - `commission_fleet <battlefield_analysis_base_id> corvette_verify sys-1 --fleet-id fleet-verify`
   - `task_force_create tf-verify --name 官方验证群 --stance escort`
   - `task_force_assign tf-verify fleet fleet-verify --system sys-1 --planet planet-1-1`
   - `task_force_deploy tf-verify --system sys-1 --planet planet-1-1`
   - `theater_create theater-verify --name 官方战区`
   - `theater_define_zone theater-verify primary --system sys-1 --planet planet-1-1 --radius 8`
   - `theater_set_objective theater-verify secure_planet --system sys-1 --planet planet-1-1 --description 官方战争闭环`
7. 用查询命令核对 authoritative 运行态：
   - `fleet_status fleet-verify`
   - `task_forces`
   - `theaters`
   - `system_runtime sys-1`
   - `planet_runtime planet-1-1`
8. 再验证轨道封锁与登陆：
   - `blockade_planet tf-verify planet-1-1`
   - `landing_start tf-verify planet-1-1 --operation-id landing-verify`
   - 随后重新执行 `system_runtime sys-1`，确认 `planet_blockades` 与 `landing_operations` 已进入 authoritative 运行态
9. 进入 `/war` 页面时，应该能直接看到：
   - 蓝图工作台
   - 军工总览
   - 战区面板
   - 战报与情报
10. 当前边界也要按现在的真实实现理解：
   - 官方战争验证局会预置科技、军工底座和补给节点，但不会替你自动创建舰队、任务群和战区
   - `/war` 当前更适合做蓝图查看/编辑、部署尝试、姿态切换、封锁/登陆和战报观察；`task_force_create|assign`、`theater_*` 与军工量产/翻修下单仍以 CLI 为最稳入口
   - `blockade_planet` / `landing_start` 的同步返回只代表命令入队，最终要看 `system_runtime` 或事件流对账

### 4.5 AI 军事委派最小验证路径

这条链路要求你已经启动 `agent-gateway`，并且手上已经有可委派的 `task_force` / `theater`。

1. 先按 4.4 建好一个任务群和战区
2. 准备一个已有的军事 agent，或先创建一个可用 agent
3. 用 `agent_update` 显式下发权限，例如：

```bash
agent_update agent-war-director \
  --command-categories observe,combat,management \
  --theater-ids theater-verify \
  --task-force-ids tf-verify \
  --military-command-ids system_runtime,task_force_set_stance,task_force_deploy \
  --allow-blockade false \
  --allow-landing false \
  --allow-military-production false \
  --military-production-limit 0
```

4. 再用 `agent_message` 给目标，例如：

```bash
agent_message agent-war-director 接管 theater-verify，并让 tf-verify 在战区内维持巡逻，汇报当前局势。
```

5. 用 `agent_thread agent-war-director` 查看它真实执行了什么、为什么这么做、当前是否需要玩家批准更高风险动作
6. 用 `task_forces`、`theaters`、`system_runtime` 对账 agent 的结果
