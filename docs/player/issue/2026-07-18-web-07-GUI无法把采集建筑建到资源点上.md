# GUI 地图点击放置无法把采集建筑建到资源点上

- 状态：已修复
- 首次记录：2026-07-18
- 来源：2026-07-18 Web 试玩（无头 Chromium + Playwright 全程 GUI 操作）
- 类别：Web / 玩法主线 / 建造

## 问题描述

`mining_machine` / `water_pump` / `oil_extractor` 必须建在资源点上，但 client-web 的本地建造预检把"资源点占用"一律当作不可建造，地图点击放置被前端直接拦截，命令根本不会发给服务端。纯 GUI 下采矿闭环走不通。

## 复现步骤

1. `p1 / key_player_1` 登录 client-web，打开 `planet-1-1`
2. 完成 `electromagnetism` 研究，解锁 `采矿机`
3. 底部建造栏点选 `采矿机`，在地图上点击硅矿资源格（如 `(5,1)`）

## 实际现象

- 命令回执：`该位置无法建造：(5, 1) 被资源点占用`
- 服务端从未收到该命令（前端 `reportLocalBlock` 本地拦截）
- 同一坐标直接调服务端 API `build mining_machine` 可以成功，说明是前端校验错误

## 根因

- `client-web/src/features/planet-map/build-workflow.ts` 的 `buildTileAssessment()` 对所有建筑把资源格计入 `blockedTiles`，`findBlockingResource()` 没有按建筑类型区分"必须压资源点"的采集建筑
- `use-planet-interactions.ts:99` 把 `resource` 类 blocked 渲染为"被资源点占用"并拦截提交
- 幽灵预览（PlanetMapPixi 也用同一评估）在资源格上同样显示不可放置，误导玩家

## 影响

- 纯 Web 无法完成"研究 → 解锁矿机 → 压矿"这条开局主线，发展在第 3 步就断了
- 与 `docs/player/玩法指南.md` 阶段 B 描述的公开验证顺序直接矛盾

## 临时绕过

- 改用 `client-cli build <x> <y> mining_machine` 或直接调服务端 API

## 推荐改进

1. `buildTileAssessment` 按建筑定义区分：对 `RequiresResourceNode` 的建筑（catalog 里有该定义），资源格不计入 blocked，反而应校验"目标格必须有匹配资源"，资源种类不匹配才拦截（如把水泵放到铁矿上）
2. 幽灵预览对采集建筑在正确资源格上显示绿色，在错误资源/空格上显示红色并给出原因
3. 补一条 e2e：GUI 点选矿机卡片 → 点击资源格 → 命令受理（现有 `planet-build-workflow.spec.ts` 只覆盖了空地放风机，没覆盖资源格）

## 证据

- 截图：`docs/player/assets/2026-07-18-web-playtest/`（试玩全程截图目录）
- 回执文本：`该位置无法建造：(5, 1) 被资源点占用`
- 对照：API 直发同坐标 `build mining_machine` 返回 `accepted` 且建筑进入 `running`

## 修复记录（2026-07-18，提交：f54bd9e）

- `client-web/src/features/planet-map/build-workflow.ts`：`buildTileAssessment()` 读取 catalog 条目的 `requires_resource_node`，采集建筑（采矿机/水泵/抽油机）资源格不再计入 `blockedTiles`；改为校验锚点格必须命中资源点，未命中则记新原因 `missing_resource`（不可建造、幽灵预览走红），与服务端 `rules.go` 校验对齐。
- `use-planet-interactions.ts`：本地拦截文案新增 `missing_resource` 分支（"需要建在资源点上"）。
- 测试：`build-workflow.test.ts` 新增 3 例（采矿机压资源格可建/空地 missing_resource/普通建筑仍不可压资源格）；`use-planet-interactions.test.ts` 新增 2 例（资源格真正下发 cmdBuild、空地本地拦截文案）；e2e `planet-build-workflow.spec.ts` 新增"采集建筑可直接放置在资源格上"用例（断言回执成功、无"被资源点占用"）。
