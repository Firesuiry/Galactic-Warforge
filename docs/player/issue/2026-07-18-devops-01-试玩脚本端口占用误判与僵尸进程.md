# 试玩脚本端口占用误判 + go run 退出残留僵尸服务端

- 状态：已修复（提交 hash：f54bd9e）
- 首次记录：2026-07-18
- 来源：2026-07-18 启动本地试玩环境时实测
- 类别：开发工具 / scripts/start-local-playtest.sh

## 问题描述

`scripts/start-local-playtest.sh` 有两个健壮性问题，都会让"脚本显示就绪"和"实际能玩"脱节：

1. **健康检查只看 HTTP 200**：`wait_for_http` 对 `http://127.0.0.1:18080/health` 只要 curl 通就算就绪。本次试玩时 18080 被机器上另一个无关服务（Sub2API）占用，它的 /health 也返回 200，脚本直接判定"服务端已就绪"，而游戏服务端其实 bind 失败退出了（日志里 `address already in use`），Web 代理被转发到陌生服务，页面所有 API 请求拿到 HTML 报 `Unexpected token '<'`。
2. **pid 文件杀不掉真正的服务端**：pid 文件记录的是 `go run` 包装进程；`go run` 会再 fork 编译产物。kill 包装进程后二进制孤儿继续跑。实测发现一个 7 月 14 日残留的 server 进程：占用约 6.4GB 内存、25% CPU，持续向同一 data dir / 日志文件写入（tick 已到 1490 万），和新开的服务端互相污染存档与日志。

## 推荐改进

1. 健康检查加内容校验：`/health` 返回体里校验游戏标识字段（或换用游戏特有的探活路径），不只是 HTTP 200
2. 启动前先检测端口占用者是不是本游戏进程；被异己占用时直接报错并提示换 `SERVER_PORT`，不要静默继续
3. 停止逻辑改为按端口杀监听进程（`lsof -ti:<port>`，脚本里 cleanup_ports 已有，可复用到 stop 路径），或启动时 `exec go run` 换成直接编译后 `exec` 二进制，保证 pid 文件指向真实监听进程
4. 建议在脚本里加一条 `stop` 子命令，试玩结束一键清理三个端口的进程

## 证据

- `server.log`：`HTTP listening on :18080` 后紧跟 `bind: address already in use`
- `curl 127.0.0.1:18080/world/summary` 返回 Sub2API 的 HTML 登录页
- `ps`：PID 2653663（7 月 14 日启动）RSS 8.0GB、CPU 25%

## 修复记录（2026-07-18，提交 hash：f54bd9e）

仅改 `scripts/start-local-playtest.sh`：

1. `wait_for_http` 增加响应体校验：服务端要求含 `"tick"`，智能体平台/Web 代理要求含 `"status":"ok"`，外部服务占用端口时不再误判就绪
2. 启动前逐端口检查占用者：`/proc/<pid>/cmdline` 含仓库路径的判定为本仓库残留进程才 kill；外部进程（含 lsof 看不到的 root/容器监听者，用 `/dev/tcp` 探测兜底）一律报错并提示用 `SERVER_PORT/GATEWAY_PORT/WEB_PORT` 换端口，不再静默继续
3. 服务端改为先 `go build -o .run/local-playtest/server-bin` 再 `exec` 二进制，`server.pid` 指向真实监听进程，消除 `go run` 孤儿僵尸
4. 新增 `stop` 子命令：按端口清理三个服务并删除 pid 文件；`print_summary` 末尾提示一键清理
5. kill 后轮询确认端口已释放，未释放则报错退出

验证：默认端口 18080 被 root 进程占用时秒级报错；`SERVER_PORT=28080` 全流程启动成功、重复运行自动清理旧进程无误判、`stop` 后三端口全部释放且无 server-bin/npm/node 残留。
