# SiliconWorld 玩法指南

> 本文基于当前 `server/` 已实现代码与 `docs/` 现有文档整理，目标不是描述“理想中的完整设计”，而是回答：**现在这版服务端，玩家实际上能玩什么，应该怎么一步一步向外扩张。**
>
> 如果你已经看过 `docs/player/上手与验证.md` 或 `docs/dev/服务端API.md`，这份文档可以理解为它们的“玩法版总纲”。

## 0. 如何开始新游戏 / 继续上一局 / 存档在哪里

- 服务端现在把 `server.data_dir` 当成“一局游戏的工作目录”。
- 想开始新游戏：把 `server.data_dir` 指向一个不存在或空目录后启动服务端。
- 想继续上一局：下次启动时继续使用同一个目录；只要里面同时有 `meta.json` 和 `save.json`，服务端就会直接续档。
- 如果目录里只有一个文件，或者目录非空但不是完整存档，服务端会拒绝启动；这类 partial 目录需要你手动清理或重建，避免把坏状态当成新局继续跑。
- 当前一局游戏的持久化文件固定就是：`<data_dir>/meta.json` 和 `<data_dir>/save.json`。
- 自动保存默认每 `60` 秒刷新一次 `save.json`；也可以用 CLI `save` 或 Web 顶栏的“保存”按钮手动落盘。
- 第一版不持久化 RNG 状态，所以续档后未来随机事件不保证和“不停服一直玩下去”完全一致；保证的是当前世界状态能继续玩。
- 续档时，目录内部保存的 `battlefield`、玩家列表、地图配置优先于外部配置；端口、限流、历史窗口、自动保存周期等纯运行参数仍按本次启动配置覆盖。
- 当前不做多槽位或命名存档点。如果你想并行保留多局游戏，做法是准备多个不同的 `data_dir`。

当前推荐有两条启动路线：

- 普通新局：`config-dev.yaml + map.yaml`，适合从开局一路推进到中期工业化。
- 官方中后期场景：`config-midgame.yaml + map-midgame.yaml`，会直接把 active planet 固定到气态行星，并预置 `vertical_launching`、`gas_giants`、`solar_sail_orbit`、`ray_receiver`、`signal_tower`、`plasma_turret`、`gravity_matrix`、`planetary_shield`、`self_evolution`、`integrated_logistics`、`photon_mining`、`annihilation` 等中后期科技与测试物资；因此 `jammer_tower`、`sr_plasma_turret`、`planetary_shield_generator`、`self_evolution_lab`、`advanced_mining_machine`、`pile_sorter`、`recomposing_assembler` 都能在官方 midgame 直接验证。`dirac_inversion` 则故意不预置，用来保留 `photon` 模式的门禁验证。

## 1. 当前版本是什么类型的游戏

当前 `server` 最适合被理解成一款：

- 以 **Tick 驱动** 为核心的工业经营游戏；
- 以 **active planet 为当前操作焦点，但允许在多个已加载行星之间切换经营** 的经营/建设游戏；
- 带有 **银河 -> 恒星系 -> 行星** 逐层发现机制的外层探索游戏；
- 带有 **战争迷雾、基础单位战斗、防御塔、敌对势力、太阳帆、戴森球框架** 的中后期扩展游戏。

已经打通的主线闭环大致是：

**研究 -> 解锁建筑/配方 -> 建造 -> 发电 -> 采矿 -> 物流/仓储 -> 冶炼/生产 -> 防御/战斗**

中后期还有：

- 行星物流 / 星际物流调度
- 同星系已加载行星之间的最小跨星球物流
- 轨道采集器
- 太阳帆发射
- 戴森球节点 / 框架 / 壳面
- 射线接收

但其中一部分目前更像“系统已存在、玩家入口还不够完整”的阶段，本文会在后面明确指出。

## 2. 开局你拥有什么

每个玩家开局默认拥有：

- 一座基地：`battlefield_analysis_base`
- 一个执行体单位：`executor`
- 基础资源：
  - `minerals = 240`
  - `energy = 100`
- 初始背包：
  - `electromagnetic_matrix = 50`
- 执行体默认能力：
  - 建造效率 `1.0`
  - 操作范围 `6`
  - 并发任务数 `2`
- 已发现内容：
  - 主银河
  - 出生恒星系
  - 出生行星
- 初始已完成科技：
  - `dyson_sphere_program`
    - 现在会直接提供 `matrix_lab + wind_turbine` 建造权限

这意味着默认新局的第一段科研入口已经固定下来：

- 基地本身不发电，所以第一步要先在基地正交相邻格补一台 `wind_turbine`
- 再把第一台空 `matrix_lab` 贴着基地摆下去，让研究站进入有电状态
- 再把启动包里的 `electromagnetic_matrix` 装进研究站，才能合法启动第一门 `electromagnetism`

当前默认胜利条件是 `hybrid`：

- **摧毁对手的 `battlefield_analysis_base`**
- 或者率先完成终局科技 `mission_complete`
- 当 `mission_complete` 研究完成后，会立即触发 `game_win`；`summary` 会返回 `winner` / `victory_reason` / `victory_rule`，事件流里会追加 `victory_declared`

## 3. 先懂这 6 个核心概念

### 3.1 你不是机甲，而是“上帝模式指挥官”

玩家不是手动开机甲采矿，而是向服务端下命令，由自己的执行体去完成：

- 建造
- 升级
- 拆除
- 生产单位
- 研究推进辅助

执行体有操作范围和并发上限，所以你不能无限同时施工。

### 3.2 资源有两层：`minerals/energy` 和 `inventory`

当前服务端里，玩家资源至少分成两层：

- `minerals` / `energy`：更像全局基础资源池，许多建造和单位生产直接消耗它
- `inventory`：玩家物品库存，用于部分物品型消耗与返还

同时，建筑本身还有自己的存储/输出缓存。也就是说：

- **矿机产矿** 主要先变成玩家的 `minerals`
- **生产建筑做出来的物品** 主要进入建筑自己的存储
- 像 `launch_solar_sail` 这样的中后期命令，会直接读取目标建筑本地已经装载的物品，而不是默认读玩家背包
- 现在玩家已经有公开装填入口：`transfer <building_id> <item_id> <quantity>`，可把背包物品直接装到建筑本地存储

这点会直接影响部分高级玩法，后文会专门提醒。

### 3.3 研究是按 Tick 自动推进的

当前研究系统已经可用，但实现方式要特别注意：

- 研究可以排队
- 研究会随 Tick 自动推进
- 至少需要 1 个处于 `running` 的研究站
- `matrix_lab` 不设置 `recipe_id` 时会作为研究站；设置了 `recipe_id` 时则按普通生产建筑运行
- `start_research` 时，所需每种矩阵都必须已经出现在研究站本地库存中
- **研究推进会真实消耗研究站本地库存中的矩阵，不再是抽象研究点**
- 缺研究站或缺矩阵时，`summary` 里的 `tech.current_research.blocked_reason` 会显示 `waiting_lab` / `waiting_matrix`

如果你走 `client-web` 的行星页，`研究与装料` 页签现在已经按这套真实规则做成阶段化研究工作台：

- 顶部直接展示当前研究进度，以及 `waiting_lab` / `waiting_matrix` 这类阻塞原因
- 中部会分成 `当前可研究` / `已完成` / `尚未满足前置`
- 默认新局在 `electromagnetism` 完成前，会额外给出 `风机 -> 空研究站 -> 装 10 电磁矩阵 -> 研究 electromagnetism` 的开局提示

所以当前版本的研究体验更接近：

**满足前置 -> 摆研究站并保持通电 -> 把矩阵装进研究站 -> 开始研究 -> 随 Tick 消耗真实矩阵 -> 获得解锁**

### 3.4 “发现世界” 和 “实际经营” 不是一回事

当前支持扫描：

- `scan_galaxy`
- `scan_system`
- `scan_planet`

扫描后你能逐步看到更多星图信息，但这不等于你已经能像经营出生星球那样去完整建设所有星球。

当前要区分三层状态：

- **已发现**：可以在星图里看到
- **已加载 runtime**：可以查询该星球的 `runtime / networks`
- **当前 active planet**：新命令默认落到这颗星球上执行

也就是说：

- 外层探索可用
- 跨星图信息发现可用
- 已加载且你已有 foothold 的星球，可以通过 `switch_active_planet` 切过去继续经营
- 同一恒星系、已加载行星之间的星际物流最小闭环已经接通

### 3.5 战争迷雾是真实存在的

玩家只能看到自己视野内的内容，主要来自：

- 自己的建筑视野
- 自己单位的视野

你可以查行星迷雾图，也可以利用扩建、防御塔、单位移动去拓视野。

### 3.6 所有操作本质上都是“命令驱动”

当前玩家可用操作，本质都走命令系统。比较核心的有：

- 建造 / 升级 / 拆除
- 配置物流站 / 物流槽位
- 生产单位
- 移动 / 攻击
- 扫描银河 / 星系 / 行星
- 开始 / 取消研究
- 取消 / 恢复施工
- 切换当前 active planet
- 把玩家背包物品装填到建筑本地存储
- 切换 `ray_receiver` 模式
- 发射太阳帆
- 建造 / 拆除戴森球结构

命令进入队列后，会在后续 Tick 执行。

## 4. 推荐的扩张路线：从出生点一路推到外层世界

这一节是最重要的“玩法顺序”。

### 阶段 A：先看清你在哪，别急着乱造

开局第一件事，不是立刻铺建筑，而是先确认：

- 当前 Tick
- 当前 active planet 是哪颗星球
- 基地附近有哪些可建地块
- 附近矿点、水域、油井、危险区域分布
- 当前迷雾覆盖范围

建议先做的查询：

- 世界摘要
- 当前行星详情
- 当前行星迷雾
- 银河 / 恒星系总览

如果你想把“外层世界”也点亮，推荐顺序是：

1. `scan_galaxy`
2. `scan_system`
3. `scan_planet`

这一步的意义是：先建立全局地图观，再决定后续目标。

### 阶段 B：第一优先级永远是“拿到稳定供电 + 采矿”

当前真正建议的第一项关键科技是：

- `electromagnetism`

但要先明确一件事：

- `wind_turbine` 已由开局预完成的 `dyson_sphere_program` 直接提供

`electromagnetism` 当前真正补齐的一组早期能力是：

- `tesla_tower`
- `mining_machine`

这意味着你能开始真正进入工业化。

推荐顺序：

1. 先在基地正交相邻格建第一台 `wind_turbine`
2. 再在基地另一侧建一台空 `matrix_lab`
3. 把 `10` 个 `electromagnetic_matrix` 装进这台研究站，再研究 `electromagnetism`
4. **先确认执行体到目标矿点是否在操作范围内；超出范围就先移动执行体**
5. 视距离补 `tesla_tower`，把电网接出去
6. 把 `mining_machine` 直接压到资源点上

当前默认图里，基地附近已经有一处能直接接通的 starter 矿点，公开验证顺序就是：

1. `build 4 2 tesla_tower`
2. `build 5 1 mining_machine`
3. 完成这条链后，首台 `matrix_lab` 仍保留，且还剩 `20 minerals`

如果你后续想接更远的矿点，再去考虑移动执行体和继续延长电网；但默认新局的第一条采矿闭环本身已经不需要拆研究站，也不需要先走位绕路。

这里有两个硬规则：

- `mining_machine` **必须建在资源点上**
- 建造、拆除、生产等执行体操作都受 **操作范围** 限制；超范围命令会在后续 tick 以 `OUT_OF_RANGE` 失败
- 建筑没电会进入 `no_power`，电力是所有后续系统的前提

你可以把这一阶段的目标理解成：

**先把“资源是负增长”变成“资源开始自己涨”**

如果你走 Web，这一段对应的最短操作路径是：

1. 打开行星页 `研究与装料`
2. 先按开局推荐路径选中 `electromagnetism`
3. 给 `matrix_lab` 装入 `10` 个 `electromagnetic_matrix`
4. 点击 `开始研究`

### 阶段 C：拿到基础物流，把“采出来”变成“送得动”

第二批非常关键的早期科技是：

- `basic_logistics_system`

它会打开最基本的产线骨架：

- `conveyor_belt_mk1`
- `sorter_mk1`
- `depot_mk1`

这阶段你要做的，不是堆高级建筑，而是把物料流真正串起来：

- 传送带负责连续运输
- 分拣器负责建筑和传送带之间的装卸
- 仓库负责缓冲

这一阶段一旦稳住，你的基地就会从“点状建筑”变成“有流向的产线”。

### 阶段 D：从采矿走向冶炼和基础制造

推荐继续推进以下早期核心科技：

- `automatic_metallurgy`：打开基础冶炼
- `basic_assembling_processes`：打开基础制造台
- `electromagnetic_matrix`：把 `matrix_lab` 从“研究站入口”推进到“蓝矩阵产线入口”

对应的关键建筑是：

- `arc_smelter`
- `matrix_lab`
- `assembling_machine_mk1`

这一阶段的目标是：

1. 用熔炉把原料变成基础材料
2. 用制造台开始做基础零件
3. 保留第一台空 `matrix_lab` 继续滚科技，再补第二台 `matrix_lab --recipe electromagnetic_matrix` 进入矩阵生产

这里有一个当前版本非常重要的实现细节：

- **生产建筑的 `recipe_id` 主要是在建造时一起指定**
- 当前没有单独暴露一个“后期再改配方”的公共命令
- **研究站要留空 `recipe_id` 才会进入科研模式；如果你给 `matrix_lab` 指定了配方，它就不会再当研究站**

所以你在造生产建筑时，最好就想清楚它要做什么。

换句话说，当前版本更适合：

- “一台建筑做一个固定工种”
- 而不是“先乱造，后面再随时重配方”

### 阶段 E：把电力、液体、化工链拉起来

当你已经有了稳定的早期采矿 + 冶炼 + 制造之后，就该进入能源升级和液体化工阶段。

建议重点关注这些方向：

- `solar_panel` 对应的太阳能路线
- `thermal_power_plant` 对应的燃料发电路线
- `water_pump` / `storage_tank` 对应的液体基础设施
- `oil_extractor` / `oil_refinery` 对应的石油链入口
- `chemical_plant` / `quantum_chemical_plant` 对应的化工升级
- `fractionator` 对应的氢/氘相关链条
- `spray_coater` 对应的喷涂增益路线

这一阶段的核心目标是：

- 把“只靠矿和简单电力”的基地，升级成“有油、有化工、有高级能源”的工业基地

这里有一个当前版本已经验证过的硬约束：

- `water_pump` 必须直接建在 `water` 资源点上
- `oil_extractor` 必须直接建在 `crude_oil` 资源点上
- 这两类采集建筑会先把产物写进自己的本地存储，再由相邻传送带/仓储继续外运
- 当前实现里的 `oil_fractionation` 只产 `refined_oil`，不再额外吐出 `hydrogen`；搭 `sulfuric_acid` 主线时不需要先处理副产分流
- 生产建筑当前只会为配方输入保留有限批次缓冲，不会再无限吸料把库存塞满；如果高级链路停住，优先检查缺料、缺电、输出/副产物去向
- 当前传送带混线在下游半满时会优先补入不同物料，不再因为固定源顺序把后一种原料长期饿死；像 `water + stone_ore` 这种两路合流现在可以稳定驱动 `sulfuric_acid`
- 真实地图里资源点旁边经常夹着 `water` 或其他不可建地，晚期液体/化工链不要只按“资源坐标差一格”纸面排布，先确认中间带位和建筑位真的可建
- 化工/制造建筑最好从一开始就给成品预留一条可出货边；如果四周全封成输入，虽然本地存储还能积货，但下一段产线会很难自然接上
- 晚期酸线如果要持续往后喂石墨烯，别只让成品停在一格输出带上；至少补一段北侧传送带或一座相邻 `depot_mk1`，否则 `sulfuric_acid` 很快会被出货侧反压

燃料型发电建筑的当前 authoritative 口径也已经固定：

- `thermal_power_plant` 消耗 `coal`
- `mini_fusion_power_plant` 消耗 `hydrogen_fuel_rod`
- `artificial_star` 消耗 `antimatter_fuel_rod`
- 这三类建筑都按 `consume_per_tick = 1` 逐 tick 消耗燃料
- 如果最后一根燃料在当前 tick 被消耗完，`inspect` / `scene` 在该 tick 结束时仍会显示 `running`，而 `/world/planets/{planet_id}/networks` 与 `stats.energy_stats` 也会继续保留这个 tick 的真实供电收益
- 只有下一 tick 仍然没有新燃料时，它们才会回到 `no_power / no_fuel`

你会开始真正感受到：

- 不同建筑需要更稳定的供电
- 物流拥堵会影响产线
- 仓储、流量监控、喷涂都会开始有意义

### 阶段 F：从“能生产”走向“能守住”

当前服务端已经有较明确的战斗与防御框架，所以中期一定要补防御。

可建造的防御/战斗相关建筑包括：

- `gauss_turret`
- `missile_turret`
- `implosion_cannon`
- `laser_turret`
- `plasma_turret`
- `signal_tower`

这一阶段建议做两件事：

1. 在基地、矿区、关键产线附近补炮塔
2. 用单位做侦查和压制

单位生产方面，当前最重要的现实细节是：

- 玩家可以用 `produce` 生产单位
- 当前公开单位边界以 `/catalog.units` 为准；当前版本实际只公开：
  - `worker`
  - `soldier`
- 当前能直接生产单位的建筑，实际上主要是：
  - `assembling_machine_mk1`
- `battlefield_analysis_base` 现在也是部署枢纽：
  - 先用配方拿到载荷 item
  - 再用 `transfer` 装进基地本地存储
  - 最后用部署命令把载荷变成 runtime 实体
- 当前公开高阶单位线已经打通：
  - `prototype` / `precision_drone`
    - 通过 `deploy_squad` 进入 `planet runtime` 的 `combat_squad`
  - `corvette` / `destroyer`
    - 通过 `commission_fleet` 进入 `system runtime` 的 `fleet`
- 高阶舰队线的后续控制也已经公开：
  - `fleet_assign`
  - `fleet_attack`
  - `fleet_disband`
  - `fleet_status`
  - `system_runtime`

也就是说，中后期的部队体系现在分成两层：

- 出门侦查
- 与敌方单位/建筑交火
- 配合炮塔进行区域控制
- 基地载荷部署
- 同星系舰队编队与轨道打击

### 阶段 G：把基地做成“更大的工业网络”

继续扩张时，你会逐渐进入“大产线”和“广域物流”的思路。

当前已经能摆出来的相关基础设施包括：

- 更高阶传送带：`conveyor_belt_mk2`、`conveyor_belt_mk3`
- 更高阶分拣器：`sorter_mk2`、`sorter_mk3`
- `splitter`
- `traffic_monitor`
- `depot_mk2`
- `planetary_logistics_station`
- `interstellar_logistics_station`
- `logistics_distributor`

科技树这一区间还有一个需要知道的公开语义变化：

- 当前玩家可见科技树会保留“桥接科技”
- 如果某个科技暂时没有直接 `unlock` / `effect`，但它仍然通向后续真实可玩的公开科技，`/catalog.techs[].leads_to` 会明确告诉你它把路线引向哪里
- 真正研究后没有直接收益、也不会导向公开收益的死胡同科技，当前版本已经从公开科技目录里移除

这一阶段现在已经可以做收敛版的物流闭环，不再只是看骨架：

- `planetary_logistics_station` / `interstellar_logistics_station` 完工后会自动补齐默认容量对应的物流单位
- 行星站会自动补无人机；星际站还会自动补货船
- 玩家可以直接用 `configure_logistics_station` 调整无人机容量、输入/输出优先级，以及星际开关 / 曲速开关 / 货船槽位
- 玩家可以直接用 `configure_logistics_slot` 按 `planetary` 或 `interstellar` 作用域，为某个 `item_id` 设置 `none|supply|demand|both` 与 `local_storage`
- 当源站设为 `supply`、目标站设为 `demand` 后，后续 tick 会自动派出无人机或货船完成配送

所以当前更实际的玩法是：

1. 研究并建造物流站
2. 找到源站和目标站的 `building_id`
3. 给源站配置供给槽位
4. 给目标站配置需求槽位
5. 等自动配送跑起来

如果你用 Web：

- 打开对应行星页
- 选中己方物流站
- 在右侧“详情”页签看结构化物流状态
- 切到右侧“命令”页签，用“物流站配置 / 物流槽位配置”直接完成站点与槽位配置

但要知道当前边界：

- 当前最成熟的是“同星系、已加载行星”的物流闭环
- 当前开放的是收敛版物流配置，不是 DSP 完整参数页
- 更广义的殖民、跨恒星系长期调度仍未扩展成完整版玩法

### 阶段 H：从行星经营走向星图扩张

当你基地变稳以后，就该做两件外扩动作：

1. 扫描更远的星系和行星
2. 评估中后期目标星体

当前探索链路非常清晰：

- 先看银河
- 再看系统
- 再看行星

你可以利用它做：

- 提前筛选资源型星球
- 判断哪里有气态行星
- 规划后续的戴森与星际物流目标

但是要记住：

- 当前服务端仍然把 active planet 当成主要操作焦点
- 但只要目标星球 runtime 已加载、你也已经在那里建立 foothold，它就不再只是“只可侦察”的信息点，而是可以切过去继续经营

### 阶段 I：进入太阳帆、射线接收和戴森球阶段

当前代码里，这部分系统已经有真实入口：

- `em_rail_ejector`
- `vertical_launching_silo`
- `ray_receiver`
- `launch_solar_sail`
- `launch_rocket`
- `build_dyson_node`
- `build_dyson_frame`
- `build_dyson_shell`
- `demolish_dyson`

玩家视角下可以这样理解：

- `EM-Rail Ejector`：装载并发射太阳帆
- `Vertical Launching Silo`：默认配方就是 `small_carrier_rocket`
- `launch_solar_sail`：把电磁发射器中已装载的太阳帆送入轨道
- `launch_rocket`：把垂直发射井中已装载的 `small_carrier_rocket` 送进指定戴森层
- `ray_receiver`：接戴森能量，甚至产出临界光子
- `build_dyson_*`：直接搭戴森球结构

当前这套玩法的关键前提是：

1. `launch_solar_sail` 当前只接受 `em_rail_ejector`
2. `launch_rocket` 当前只接受 `vertical_launching_silo`
3. 发射前要先把对应载荷通过传送带/建筑 IO 装进发射器本地存储
4. `launch_rocket` 目标层必须至少已有一个 `build_dyson_*` 脚手架
5. `set_ray_receiver_mode` 可把 `ray_receiver` 切到 `power` / `photon` / `hybrid`；其中 `photon` 需要先解锁 `dirac_inversion`

这意味着：

- **太阳帆发射链路已经接入正常建筑存储/装载流程**
- **火箭发射链路也已经可玩，但仍需要你自己把原料/火箭喂进 silo**
- **玩家不需要再把太阳帆转进个人 `inventory` 才能发射**
- **`ray_receiver` 本身不会白送基础能量；没有太阳帆/戴森供能时，它不会凭空产电或产临界光子**
- **火箭成功发射后会直接提高目标戴森层的 `construction_bonus`，后续 `ray_receiver` 能吃到更多可用输出**
- **当前真实供能会同时计入太阳帆轨道能和已建戴森球结构总产能，`ray_receiver` 最终吃到的仍受自身 `input_per_tick` 上限约束**
- **当 `ray_receiver` 切到 `power` / `hybrid` 且已存在太阳帆或戴森结构产能时，同一轮真实 tick 会同时抬高 `summary.players[pid].resources.energy`、`stats.energy_stats.generation` 与 `/world/planets/{planet_id}/networks.power_networks[].supply`**
- **当前公开 `inspect planet-1-2 building <ray_receiver_id>` 主要用于确认 `runtime.functions.ray_receiver.mode` 与建筑运行态；真正的 authoritative 电力回灌结果请看 `summary`、`stats` 与 `GET /world/planets/{planet_id}/networks`**
- **`power` 模式的验收口径是“切换后不再新增 `critical_photon`”而不是“把旧光子库存清零”；切模式前已经存在于 `inventory / output_buffer` 的历史光子会保留**
- **如果 `/world/planets/{planet_id}/networks.power_coverage` 已显示 `connected = true` 且网络 `shortage = true`，那么 `inspect` / `scene` / SSE 里的缺电原因应读作 `under_power`，不再是 `power_out_of_range`**

如果你走 Web，这一阶段现在也有对应的上下文提示：

- 把 `solar_sail` 装进 `em_rail_ejector` 后，命令账本会直接提示“下一步可发射太阳帆扩展戴森云”
- 把 `small_carrier_rocket` 装进 `vertical_launching_silo` 后，会提示“下一步可发射火箭构建戴森球结构”
- `set_ray_receiver_mode` 成功后，会按 `power / photon / hybrid` 三种模式分别提示应该观察电网回灌、光子产出或两者并行结果

同时，`build_dyson_node/frame/shell` 当前更偏“系统验证型入口”：

- 命令可用
- 结构会创建
- 但它不像传统主线玩法那样，已经把完整科技/材料门槛全部做成硬约束

所以这一阶段的正确认知是：

- 戴森球系统已经能体验
- 但更偏中后期实验性玩法，而不是完全成熟的最终主线
- 官方 midgame 现在也已经预置 `integrated_logistics`、`photon_mining`、`annihilation`，所以 `advanced_mining_machine`、`pile_sorter`、`recomposing_assembler` 不再需要玩家先手动补科研才能验证
- `recomposing_assembler` 现在已经补齐两条终局弹药配方：
  - `build <x> <y> recomposing_assembler --recipe antimatter_capsule`
  - `build <x> <y> recomposing_assembler --recipe gravity_missile`
- `prototype`、`precision_drone`、`corvette`、`destroyer` 这条高阶舰队线现在已经进入公开主线，但语义明确分流：
  - `produce` 仍只负责 `worker` / `soldier`
  - 高阶单位必须先做成载荷 item，再通过 `deploy_squad` / `commission_fleet` 进入 runtime
  - `fleet_attack` 当前只支持同一恒星系目标
  - 部署枢纽必须通电，否则会被 authoritative 校验以 `deployment hub is not operational: ...` 拒绝

## 6. 当前版本最值得玩家注意的现实边界

这一节非常重要，直接决定你玩的时候会不会“以为自己操作错了”。

### 6.1 当前主经营舞台基本还是 active planet

你可以扫描很多星球，而当前命令默认仍然会落到 active planet 上执行。

所以正确玩法是：

- 把 active planet 当当前施工与操作焦点
- 把已加载且有 foothold 的其他星球当成可切换的分基地
- 把还没加载或还没 foothold 的外层星球当作后续规划信息

### 6.2 研究已经是“矩阵实物消耗版”

当前研究会真实消耗研究站本地库存中的矩阵。

所以：

- 研究站必须处于 `running`
- 所需矩阵必须先进入研究站本地库存
- 如果 `start_research` 成功后又缺矩阵，研究会卡在 `blocked_reason = waiting_matrix`

### 6.3 生产建筑最好建造时就带上配方

当前公共玩法里，没有成熟的“随时改配方”体验。

所以建议：

- 一条线一个用途
- 一台机一个配方
- 少做频繁切换

### 6.4 物流系统已经能玩，并且接入了最小多星球闭环

现在玩家已经可以直接使用：

- `configure_logistics_station`
- `configure_logistics_slot`

这意味着当前已经能形成 `造站 -> 配槽位 -> 自动配送 -> 跨已加载行星派船` 的玩法闭环。CLI 和 Web 行星页现在都已经能直接触达这套收敛版物流配置。

但它仍有明确边界：

- 当前支持的是同一恒星系、已加载行星之间的最小星际物流
- 开放给玩家的物流参数还是收敛版，不是 DSP 全量面板
- 更广义的殖民、跨恒星系长期调度仍未打通

### 6.5 太阳帆 / 戴森球可体验，但部分链路偏实验性

特别是：

- `launch_solar_sail` 依赖 `em_rail_ejector` 本地已装载的 `solar_sail`
- `vertical_launching_silo` 现在能默认生产 `small_carrier_rocket`，但你仍需要自行完成供料或直接用 `transfer` 装填成品火箭
- `launch_rocket` 需要目标层已有 `build_dyson_*` 脚手架
- `build_dyson_*` 命令本身仍更偏系统直连入口

如果你是普通玩家，建议把戴森部分理解为：

- **能体验**
- **能验证系统**
- **但脚手架规划入口仍然比中前期工业闭环更偏实验性**

### 6.6 `orbital_collector` 已有官方验证路线，但不属于普通新局主线

它依赖：

- 当前 active planet 是气态行星

因此：

- 在普通新局里，它仍然未必会自然进入主线
- 但现在已经可以通过官方 midgame 场景稳定验证：启动后 active planet 会直接落在 `planet-1-2` 这颗气态行星上

## 7. 一条最实用的新手流程

如果你只想知道“第一局大概怎么玩”，可以直接按下面走：

1. 先查当前星球、迷雾、资源点分布
2. `build 3 2 wind_turbine`
3. `build 2 3 matrix_lab`
4. `transfer <matrix_lab_id> electromagnetic_matrix 10`，然后研究 `electromagnetism`
5. `build 4 2 tesla_tower`
6. `build 5 1 mining_machine`
7. 研究 `basic_logistics_system`
8. 铺 `conveyor_belt_mk1` + `sorter_mk1` + `depot_mk1`
9. 研究 `automatic_metallurgy`
10. 造 `arc_smelter`
11. 研究 `basic_assembling_processes`
12. 到这里默认 starter 闭环已经成立：首台研究站保留、首台矿机 `running`、资源重新开始增长
13. 造 `assembling_machine_mk1`，建造时直接带 `recipe_id`
14. 研究 `electromagnetic_matrix`，再补一台 `matrix_lab --recipe electromagnetic_matrix`
15. 补 `solar_panel` / `thermal_power_plant`，稳住电力
16. 补 `gauss_turret` 和 `signal_tower`，开始防区化
17. 再去考虑物流站配置、油化工、星际规划、戴森球

如果你是来验证中后期闭环，而不是体验开荒，直接走官方 midgame 场景会更高效：

1. 用 `config-midgame.yaml + map-midgame.yaml` 启动服务端
2. `summary` 确认 `active_planet_id = planet-1-2`
3. `system sys-1` 确认 `planet-1-2.kind = gas_giant`
4. 先补 `tesla_tower + wind_turbine`，把 `stats.energy_stats.generation` 堆到至少 `84`，再开始建中后期建筑
5. 先建 `orbital_collector` 验证气态行星入口
6. 再建 `vertical_launching_silo` 与 `em_rail_ejector`
7. 用 `transfer` 把背包里的 `small_carrier_rocket` / `solar_sail` 装进发射建筑
8. 用 `switch_active_planet` 在 `planet-1-1` / `planet-1-2` 间切换，确认 loaded 行星可继续经营
9. 用 `build_dyson_node/frame/shell` 先搭一个目标层脚手架
10. 先执行 `launch_solar_sail`，再执行 `launch_rocket`
11. 在已通电空地依次建 `jammer_tower`、`sr_plasma_turret`、`planetary_shield_generator`、`self_evolution_lab`
12. 如需顺手验证矩阵生产，再额外建一台 `self_evolution_lab --recipe electromagnetic_matrix`
13. 用 `inspect` 观察护盾发生器的 `runtime.functions.shield.current_charge` 持续增长，并确认 `jammer_tower` / `sr_plasma_turret` 进入 `running`
14. 对射线接收站执行 `set_ray_receiver_mode <receiver_id> power|hybrid`
15. 先记下接收站当前 `critical_photon` 基线，再观察 `rocket_launched` 事件，以及后续 `summary` / `stats` / `/world/planets/planet-1-2/networks` 中的收益变化；正确结果是玩家 `energy`、`generation`、网络 `supply` 继续上涨，而旧光子库存不被清零、也不再继续增长
