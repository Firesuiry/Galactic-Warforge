# SiliconWorld 项目概览

## 1. 项目定位

SiliconWorld 不是《戴森球计划》的原样复刻，而是一个更偏服务器化的工业战争游戏：

- 服务端权威
- Tick 驱动结算
- HTTP + SSE + 命令接口驱动
- 多玩家 / 阵营对抗
- AI 可通过客户端侧能力参与操作

它的体验核心是：工业经营、外层探索、防御与战争、以及逐步进入太阳帆和戴森结构等中后期系统。

## 2. 系统边界

### 2.1 服务端负责什么

- 世界状态与规则计算
- 命令鉴权、入队与结算
- 资源、物流、研究、战斗、迷雾、事件
- 快照、回放、审计、保存

### 2.2 客户端负责什么

- 人类操作界面
- Web / CLI 等不同交互形式
- AI 的提示词、记忆、模型调用和多轮规划
- 将 AI 或用户意图转换成标准命令后提交给服务端

### 2.3 agent-gateway 的位置

`agent-gateway` 不属于 `server/`。它是本地 AI 运行时，负责连接模型、CLI 和 Web 前端，是客户端侧能力而不是服务端核心模块。

## 3. 世界与交互模型

- 世界分层为银河 -> 恒星系 -> 行星。
- 当前实现仍以 `active planet` 作为命令默认落点，但已加载行星会并行保留 runtime，并可在有 foothold 的目标间切换 active planet。
- 行星地图是 2D 网格，不是球面铺设。
- 玩家是“上帝视角指挥者”，开局默认带执行体，而不是直接扮演机甲。
- 所有操作统一走命令接口，服务端不区分人类命令和 AI 生成命令。

## 4. 核心设计原则

- 服务端权威：状态只能由服务端 Tick 结算改变。
- 逻辑与表现解耦：服务端不绑定具体渲染表现。
- 命令驱动：客户端提交意图，服务端做鉴权、校验和结算。
- 可回放与可审计：关键状态和命令可追踪、可复盘。
- AI 职责外置：AI 行为编排放在客户端或本地网关，不放在服务端内核。

## 5. 与《戴森球计划》的关键差异

- 不是机甲视角，而是多人命令驱动服务器。
- 地图是 2D 平面，不是球面建造。
- 当前完整模拟仍以 `active planet` 为焦点，但已加载行星的 runtime / networks 以及同星系最小星际物流闭环已经接通。
- 战斗和敌对势力权重更高，工业与战争耦合更紧。
- 交互天然面向 API、CLI、Web 和 AI，而不是单一图形前端。

## 6. 当前戴森中后期落点

当前实现已经不再只是“有戴森相关数据结构”，而是形成了一条可验证的中后期到终局最小闭环：

- 官方 `config-midgame.yaml + map-midgame.yaml` 场景会把新局直接落到气态行星 `planet-1-2`
- `players[].bootstrap` 会预置中后期科技、基础资源和戴森链路关键物资
- `vertical_launching_silo` 现在自带默认配方 `small_carrier_rocket`
- 玩家可以通过 `transfer_item` 把背包物品直接装入建筑本地存储，再配合 `build_dyson_*` 脚手架与 `launch_solar_sail` / `launch_rocket` 走完当前这条中后期发射链路
- `launch_rocket` 会把火箭转成目标戴森层的 `rocket_launches` / `construction_bonus`；当前实现按每枚火箭 `+0.02`、上限 `0.5` 的简化公式推进，若层上已有壳面还会继续抬升壳面覆盖率
- `GET /world/systems/{system_id}/runtime` 现在已经成为中后期戴森态势的公开 authoritative 事实源，会返回太阳帆轨道、戴森层总产能/火箭累计发射次数，以及当前 active planet 上的弹射器 / 发射井 / 射线接收站上下文；即使该 system 还没有独立 `space runtime`，只要当前 active planet 属于它，也仍可能拿到 `active_planet_context`
- `ray_receiver` 只会消费真实存在的太阳帆轨道能量和戴森层能量，不再凭空产出
- `battlefield_analysis_base` 现在同时承担最小部署枢纽角色；玩家可把 `prototype` / `precision_drone` / `corvette` / `destroyer` 载荷装入其本地存储，再通过 `deploy_squad` / `commission_fleet` 写入行星 combat runtime 与恒星系 space runtime
- Web 行星工作台也已经切到 `request_id + command_index -> command_result` 的 authoritative 闭环；同步响应里的 `accepted` 只表示命令入队，不再代表最终成功
- 终局边界也已经收口到服务端权威规则：`mission_complete` 会在 `hybrid` 胜利规则下立即转成 `game_win`，而 `thermal_power_plant` / `mini_fusion_power_plant` / `artificial_star` 都会在缺燃料时进入 `no_power/no_fuel`

这条闭环的当前边界也很明确：

- 它优先依赖官方 midgame 场景来验证，不代表普通新局已经自然打通到这里
- 整个运行态仍然以 `active planet` 为主要操作焦点
- `build_dyson_*` 仍是偏实验性的脚手架入口，不是完整版材料化戴森规划系统；它们当前主要做 runtime 结构写入与最小校验，不会真实扣除戴森建筑材料
- `demolish_dyson` 当前也只是拆掉 runtime 结构，并在事件里带回简化退款提示，不是完整的资源返还链路
- 高阶单位与舰队现在属于“公开最小闭环”，不是完整版太空战争系统；`fleet_attack` 当前仍只支持同一恒星系内、以目标行星 `enemy_forces[].id` 为靶标的攻击
