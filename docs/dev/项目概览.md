# SiliconWorld 项目概览

## 1. 项目定位

SiliconWorld 不是《戴森球计划》的原样复刻，而是一个更偏服务器化的工业战争游戏：

- 服务端权威
- Tick 驱动结算
- HTTP + SSE + 命令接口驱动
- 多玩家 / 阵营对抗
- AI 可通过客户端侧能力参与操作

它的体验核心是：工业经营、外层探索、防御与战争、以及逐步进入太阳帆和戴森结构等中后期系统。

## 2. 系统边界

### 2.1 服务端负责什么

- 世界状态与规则计算
- 命令鉴权、入队与结算
- 资源、物流、研究、战斗、迷雾、事件
- 快照、回放、审计、保存

### 2.2 客户端负责什么

- 人类操作界面
- Web / CLI 等不同交互形式
- AI 的提示词、记忆、模型调用和多轮规划
- 将 AI 或用户意图转换成标准命令后提交给服务端

### 2.3 agent-gateway 的位置

`agent-gateway` 不属于 `server/`。它是本地 AI 运行时，负责连接模型、CLI 和 Web 前端，是客户端侧能力而不是服务端核心模块。

## 3. 世界与交互模型

- 世界分层为银河 -> 恒星系 -> 行星。
- 当前实现的完整运行态主要围绕单个 `active planet` 展开。
- 行星地图是 2D 网格，不是球面铺设。
- 玩家是“上帝视角指挥者”，开局默认带执行体，而不是直接扮演机甲。
- 所有操作统一走命令接口，服务端不区分人类命令和 AI 生成命令。

## 4. 核心设计原则

- 服务端权威：状态只能由服务端 Tick 结算改变。
- 逻辑与表现解耦：服务端不绑定具体渲染表现。
- 命令驱动：客户端提交意图，服务端做鉴权、校验和结算。
- 可回放与可审计：关键状态和命令可追踪、可复盘。
- AI 职责外置：AI 行为编排放在客户端或本地网关，不放在服务端内核。

## 5. 与《戴森球计划》的关键差异

- 不是机甲视角，而是多人命令驱动服务器。
- 地图是 2D 平面，不是球面建造。
- 当前完整模拟集中在单个 `active planet`，不是多星球同时深度经营。
- 战斗和敌对势力权重更高，工业与战争耦合更紧。
- 交互天然面向 API、CLI、Web 和 AI，而不是单一图形前端。
