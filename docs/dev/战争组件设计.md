# 战争系统设计

## 1. 文档定位

本文档定义 SiliconWorld 下一阶段的完整战争系统设计，目标是把当前已有的：

- 行星层地面单位与防御建筑
- `prototype / precision_drone / corvette / destroyer` 的最小部署闭环
- 三层宇宙地图（银河 / 恒星系 / 行星）
- Tick 驱动、服务端权威结算、AI Agent 协作

演进成一套真正可玩的“工业战争”系统。

设计目标不是简单复制某一款现成游戏，而是融合：

- `红警 / C&C` 的基地压制、前线推进、单位可读性
- `星际争霸` 的节奏、侦察、时机波次、宏观与微操张力
- `钢铁雄心` 的编制、战区、补给、模板设计
- `Terra Invicta` 的舰船工程约束
- `NEBULOUS: Fleet Command` 的导弹、点防御、电子战与编队火力

形成适合 SiliconWorld 当前架构的战争玩法。

## 2. 当前实现与关键缺口

当前仓库已经具备一些战争雏形：

- 地表单位：
  - `worker`
  - `soldier`
  - `prototype`
  - `precision_drone`
- 太空单位：
  - `corvette`
  - `destroyer`
- 公开命令：
  - `deploy_squad`
  - `commission_fleet`
  - `fleet_assign`
  - `fleet_attack`
  - `fleet_disband`
- 已有运行态：
  - 行星 `CombatRuntime`
  - 恒星系 `SpaceRuntime`
- 已有相关防御能力：
  - 炮塔
  - 护盾
  - 信号塔
  - 战争迷雾

但目前系统仍停在“可部署、可查询、可做简单攻击”的阶段，还缺：

- 单位设计器
- 舰船 / 机甲模块化
- 成体系的兵种克制
- 编制、舰队、战区、战线
- 补给、燃料、弹药、维修
- 轨道封锁、登陆、制空 / 制轨
- 设计定型、改型、量产、翻修
- 电子战、隐形、侦察、情报可信度
- 战报、损耗、撤退、士气 / 凝聚度
- AI Agent 可代理的军事工作流

因此本轮设计不应继续做“多加几个单位”的线性扩展，而应直接定义完整战争框架。

## 3. 外部调研结论

### 3.1 红警 / C&C 提供的启发

可借鉴点：

- 战争节奏必须短反馈、强可读。
- 生产与前线压制是同一条循环，不是割裂系统。
- 基地、矿区、路口、防御塔构成前线空间控制。
- 单位辨识度必须高于美术细节堆叠。

设计转译：

- SiliconWorld 的战争单位需要在大地图和局部视图下都能一眼识别类型与威胁。
- 关键战术对象要少而硬：矿区、物流枢纽、轨道窗口、登陆点、前哨、防空区、护盾区。
- 前线交战必须持续牵动工业后方，而不是只看一次性爆兵。

### 3.2 星际争霸提供的启发

可借鉴点：

- 宏观运营与微观控制必须同时重要，但不要求全局都高 APM。
- 侦察、骚扰、压制、时机波是战争节奏核心。
- 科技攀升不是单纯数值变大，而是解锁新战术维度。
- 视野与战争迷雾直接决定主动权。

设计转译：

- SiliconWorld 需要把“侦察成功”和“压经济成功”做成真实收益。
- 战争命令必须支持：骚扰、牵制、佯攻、切后勤、断补给、速攻窗口。
- 高级科技应优先解锁新组件、新战术、新部署层级，而不是只加伤害。

### 3.3 钢铁雄心提供的启发

可借鉴点：

- 玩家不应只控制单兵，而应设计模板、组织编制、划战区、定优先级。
- 速度、火力、装甲、成本、补给是强耦合约束。
- 超宽、缺补给、燃料不足会直接击穿“纸面最优模板”。
- 改型与翻修比“推倒重练”更符合大战略节奏。

设计转译：

- SiliconWorld 需要“蓝图 -> 编制 -> 战区 -> 命令姿态”的链路。
- 强度不能只看 DPS，需要同时受补给吞吐、指挥带宽、部署空间、燃料线限制。
- 高阶单位必须支持改型与旧型号翻修。

### 3.4 Terra Invicta 提供的启发

可借鉴点：

- 舰船设计必须是工程问题，而非纯菜单选项。
- 驱动、反应堆、散热、装甲、推进剂、挂点、角色彼此制约。
- 质量会反过来影响加速度、转向与航程。
- 不同角色不是简单“大船打小船”，而是由任务、航程、硬点和成本共同定义。

设计转译：

- SiliconWorld 的舰船与机甲必须引入：
  - 功率预算
  - 体积预算
  - 质量预算
  - 热负荷
  - 结构刚性
  - 指挥 / 计算需求
- 船体和机体要先有“底盘角色”，再谈挂件。

### 3.5 NEBULOUS 提供的启发

可借鉴点：

- 导弹、点防御、电子战、诱饵、齐射规划能形成高质量中远程博弈。
- 弹药不是无限的，火力计划需要受编程通道、装填时间、舰内配置限制。
- 防空 / 点防御协同是舰队体系的一部分，不是单一数值。
- 信息战和导弹战天然适合 AI 辅助执行。

设计转译：

- SiliconWorld 的太空战应支持：
  - 导弹 / 鱼雷 / 打击机
  - 点防御
  - 干扰 / 锁定 / 诱饵
  - 齐射窗口
  - 电子压制
- 高级舰队的战术差异应主要来自“组件组合 + 编队角色 + 火控策略”。

## 4. 设计总原则

### 4.1 战争是工业系统的延伸

玩家先有工业，后有战争；战争反过来保护、掠夺、切断工业。

核心闭环：

`科研 -> 设计 -> 定型 -> 量产 -> 编制 -> 部署 -> 补给 -> 交战 -> 战损 / 战报 -> 改型 -> 再量产`

### 4.2 三层地图都必须有军事意义

- 银河层：远征方向、主战场选择、战略通道争夺
- 恒星系层：轨道封锁、舰队拦截、制轨权、跨星球投送
- 行星层：基地建设、地面前线、据点、防空、登陆与占领

### 4.3 设计器必须先于单位表膨胀

不采用“持续手写新增 50 个单位 ID”的方案。

采用：

- 少量底盘 / 船体
- 多类模块
- 设计验证规则
- 蓝图定型
- 蓝图改型

这样才符合项目“激进式演进、减少耦合”的原则。

### 4.4 默认支持 AI Agent 接管

所有战争操作都应可拆成 AI 可代理任务：

- 设计军备
- 维护产能
- 组织舰队
- 设定战区
- 侦察敌情
- 自动补给
- 自动修复与补员
- 按姿态执行战斗

### 4.5 战争必须可解释

玩家需要明确知道：

- 为什么这个设计非法
- 为什么这支舰队打不过
- 为什么这条战线断补给
- 为什么隐形失效
- 为什么点防没拦住这一轮齐射

因此系统必须内置诊断和战报，而不是只给最终血量结果。

## 5. 战争层级模型

## 5.1 行星层

行星层负责：

- 机甲、地面载具、士兵、无人机
- 基地火力网
- 防空与轨道打击抗性
- 地面资源点争夺
- 登陆区、前线推进、据点巩固
- 行星物流与地面补给

玩家感受应更接近：

- 红警 / 星际式的局部实时战斗
- 但通过命令与姿态控制，而不是要求全图高强度手操

### 5.2 恒星系层

恒星系层负责：

- 舰队部署与巡航
- 轨道封锁
- 护航与袭扰
- 星际运输线拦截
- 轨道轰炸
- 制轨权与登陆掩护

玩家感受应更接近：

- 太空海军 + 作战任务群调度

### 5.3 银河层

银河层负责：

- 战区划分
- 主攻 / 佯攻方向
- 远征补给线
- 关键恒星系争夺
- 战略目标与胜利条件推进

银河层不直接做细战术，而负责“把哪场仗变成决定性战役”。

## 6. 单位与组织体系

## 6.1 单位域

战争单位分为五大域：

1. `ground_unit`
   - 士兵
   - 轻型机甲
   - 重型机甲
   - 履带 / 悬浮载具
2. `air_unit`
   - 无人机
   - 近地航空支援机
   - 轰炸机
3. `orbital_unit`
   - 轨道平台
   - 轨道打击器
   - 登陆投送舱
4. `space_unit`
   - 护卫舰
   - 驱逐舰
   - 巡洋舰
   - 航母 / 母舰
5. `support_unit`
   - 补给舰
   - 维修舰
   - 指挥舰
   - 电子战舰
   - 工程舰

## 6.2 组织层级

推荐采用以下层级：

- 地面：
  - `element` 单体
  - `squad` 小队
  - `company` 连队
  - `task_group` 地面任务群
- 舰队：
  - `ship`
  - `division` 分队
  - `fleet` 舰队
  - `armada` 战区舰群

## 6.3 指挥带宽

引入 `command_capacity`：

- 指挥中心
- 指挥舰
- 战场分析基站
- 军事 AI 核心

共同提供指挥带宽。

超编后惩罚：

- 响应延迟上升
- 编队维持变差
- 命中率下降
- 协同火力衰减
- 自动反应失效

这样可以防止无脑堆单位。

## 7. 蓝图设计器

## 7.1 核心概念

战争单位不再直接等于固定 ID，而是：

- `底盘 / 船体（chassis / hull）`
- `组件（components）`
- `蓝图（blueprint）`
- `改型（variant）`

关系如下：

`底盘` 定义容纳边界，`组件` 定义能力，`蓝图` 定义定型方案，`改型` 定义基于已定型蓝图的演进。

## 7.2 底盘类别

### 机甲底盘

- `light_frame`
  - 高机动、低载荷、侦察 / 骚扰
- `medium_frame`
  - 通用前线主力
- `heavy_frame`
  - 高装甲、慢速、突破 / 守点
- `assault_frame`
  - 极高武装与体积，依赖后勤

### 舰船船体

- `corvette_hull`
  - 侦察、骚扰、拦截
- `destroyer_hull`
  - 主力护航、反导、对空
- `cruiser_hull`
  - 远程火力、旗舰候选
- `carrier_hull`
  - 舰载机与导弹平台
- `siege_hull`
  - 重炮、轨道压制、低机动

## 7.3 组件大类

每个单位设计由以下组件组成：

- 动力：
  - 反应堆 / 能源核心
  - 电容 / 电池组
- 推进：
  - 主引擎
  - 姿态推进器
  - 地面关节 / 伺服 / 悬浮模块
- 防御：
  - 装甲
  - 护盾
  - 点防御
  - ECM / 干扰器
  - 诱饵 / 烟幕 / 热诱饵
- 感知：
  - 光学
  - 雷达
  - 红外
  - 主动扫描
  - 战场数据链
- 武器：
  - 实弹
  - 激光
  - 电浆
  - 导弹
  - 轨道炮
  - 近防炮
  - 反舰鱼雷
- 功能：
  - 隐形模块
  - 维修模块
  - 登陆舱
  - 货舱
  - 指挥模块
  - AI 战术核心
  - 机库 / 无人机舱

## 7.4 强制约束

每个设计必须通过以下约束校验。

### 1. 功率约束

定义：

- `reactor_output`
- `peak_draw`
- `sustained_draw`
- `capacitor_buffer`

规则：

- 峰值功耗不能长期高于反应堆输出
- 高于持续输出但低于峰值缓冲时，可短时爆发
- 功率不足时优先级低的模块自动降载或离线

结果：

- 同一船体无法无限堆主炮、护盾和隐形

### 2. 体积约束

定义：

- `internal_volume`
- `mount_slots`
- `external_hardpoints`

规则：

- 组件总体积不能超过底盘可用体积
- 特定武器必须占用指定挂点
- 某些模块互斥

### 3. 质量约束

定义：

- `dry_mass`
- `loaded_mass`
- `thrust`
- `ground_pressure`
- `maneuver_mass_limit`

规则：

- 质量越高，速度 / 加速度 / 转向越差
- 行星重力和地形会进一步放大惩罚
- 登陆舱、弹药和装甲会显著推高载重

### 4. 刚性约束

定义：

- `frame_rigidity`
- `weapon_recoil`
- `torque_load`
- `armor_stress`

规则：

- 大型主炮、高速转向、大重量偏置都消耗刚性预算
- 超限设计允许保存为“实验蓝图”，但会有以下缺陷：
  - 精度下降
  - 模块故障率上升
  - 高速机动时自损

### 5. 热负荷约束

定义：

- `heat_generation`
- `heat_dissipation`
- `heat_sink`

规则：

- 武器、护盾、隐形、推进都会产热
- 热量过高时：
  - 隐形失效
  - 射速下降
  - 感测暴露增加
  - 极端时模块熔断

### 6. 隐形与信号约束

定义：

- `radar_signature`
- `thermal_signature`
- `em_signature`
- `stealth_rating`

规则：

- 隐形不是“完全不可见”
- 只是提高敌方探测门槛和锁定时间
- 高功率武器开火、推进加力、主动雷达照射会迅速暴露自身

### 7. 维护约束

定义：

- `maintenance_cost`
- `ammo_drain`
- `fuel_drain`
- `repair_complexity`

规则：

- 强蓝图不应只贵在建造，还要贵在维持
- 高端装备若无后勤支撑，持续战斗能力必须明显崩塌

## 7.5 蓝图状态机

建议蓝图生命周期：

- `draft`
- `validated`
- `prototype`
- `field_tested`
- `adopted`
- `obsolete`

流程：

1. 玩家创建草案
2. 服务端校验约束
3. 投入原型制造
4. 在试验场或实战中收集指标
5. 正式定型
6. 允许基于定型蓝图派生改型

## 7.6 改型系统

一个定型蓝图可以派生多个改型：

- `Mk2 engine refit`
- `AA escort variant`
- `stealth strike variant`
- `planetfall assault variant`

改型只允许在父蓝图基础上修改有限字段，继承：

- 船体
- 大部分结构参数
- 产线适配

这样能降低内容管理成本，并保留“军工谱系感”。

## 8. 生产与军工体系

## 8.1 生产分层

战争生产拆成三层：

1. `组件生产`
   - 引擎
   - 武器
   - 雷达
   - 护盾发生器
   - 导弹弹体
2. `总装`
   - 机甲装配厂
   - 舰船船坞
   - 无人机机库
3. `部署`
   - 行星兵营 / 指挥枢纽
   - 轨道船坞
   - 战场分析基站
   - 舰队集结点

## 8.2 量产效率

同一蓝图持续量产时，应获得：

- 装配熟练度
- 质检稳定度
- 单位工时下降

但切换蓝图会损失产线效率。

这样能鼓励：

- 主力型号定型
- 局部改型，而非每次重做

## 8.3 翻修与升级

已有单位可在维修设施中执行：

- 更换武器
- 更换护盾
- 更换传感器
- 更换引擎
- 加装隐形模块

约束：

- 不能突破底盘上限
- 战场临时翻修能力远弱于后方船坞

## 9. 编制、姿态与战区

## 9.1 编制模板

在蓝图之上，再引入编制模板。

示例：

- `行星突击连`
  - 6 轻机甲
  - 4 中机甲
  - 2 防空车
  - 2 工程车
- `护航驱逐分队`
  - 4 驱逐舰
  - 2 电子战舰
  - 1 补给舰
- `登陆任务群`
  - 2 航母
  - 6 护航舰
  - 8 登陆舱
  - 轨道火力支援舰若干

## 9.2 战斗姿态

单位和编制需支持姿态：

- `hold`
- `patrol`
- `escort`
- `intercept`
- `harass`
- `raid_logistics`
- `siege`
- `bombard`
- `retreat_on_losses`
- `preserve_stealth`
- `aggressive_pursuit`

姿态会影响：

- 目标优先级
- 交战距离
- 弹药使用策略
- 是否追击
- 是否暴露主动探测

## 9.3 战区

玩家可定义：

- 主战区
- 次战区
- 禁入区
- 补给优先区
- 集结区
- 登陆预备区

战区将成为 AI Agent 军事自治的核心边界。

## 10. 侦察、迷雾与电子战

## 10.1 情报分级

敌方目标不再只有“看见 / 没看见”。

改为四级情报：

1. `unknown_signal`
2. `classified_contact`
3. `confirmed_type`
4. `fully_resolved`

例子：

- 只知道该区域有热源
- 推测是舰队，但不确定构成
- 确认是导弹驱逐分队
- 确认具体舰种、伤势、姿态、签名模式

## 10.2 探测来源

- 视野
- 主动雷达
- 被动电磁监听
- 红外
- 信号塔
- 侦察机 / 侦察舰 / 无人机
- 间谍 / 渗透设施

## 10.3 电子战

电子战组件提供：

- 干扰锁定
- 提升己方误导概率
- 压低敌方传感精度
- 诱导敌方导弹偏航
- 伪造虚假目标

电子战不应只做一个减命中 buff，而要实打实影响探测、锁定、导弹和齐射质量。

## 11. 补给、燃料与维修

## 11.1 战争补给资源

建议引入以下军事补给维度：

- `ammo`
- `missiles`
- `fuel`
- `spare_parts`
- `shield_cells`
- `repair_drones`
- `crew_replacement`

地面与太空单位消耗结构不同。

## 11.2 补给网络

补给来源：

- 行星物流站
- 星际物流站
- 轨道补给港
- 补给舰
- 前线补给车 / 空投

补给链断裂后，单位不应立即消失，而是逐级退化：

1. 弹药不足
2. 护盾充能变慢
3. 维修停止
4. 机动下降
5. 士气和凝聚度下降
6. 最终被迫撤退或崩溃

## 11.3 维修与恢复

维修分三级：

- 战场应急修复
- 前线维修站
- 后方完整翻修

不同级别影响：

- 可恢复血量上限
- 可恢复子系统数量
- 可更换组件范围

## 12. 战斗结算模型

## 12.1 基本结算阶段

每 Tick 战斗可按如下顺序处理：

1. 侦察更新
2. 指令决策
3. 移动与编队修正
4. 锁定与火控
5. 武器发射 / 命中判定
6. 护盾 / 装甲 / 结构伤害结算
7. 士气 / 凝聚度 / 退却判定
8. 战报与事件生成

## 12.2 伤害分层

伤害优先落点：

- 护盾
- 外层装甲
- 结构
- 子系统

子系统包括：

- 引擎
- 火控
- 传感器
- 反应堆
- 弹药舱
- 指挥舱
- 点防系统

这样战斗结果会自然出现：

- 失去机动
- 失去雷达
- 失去拦截能力
- 武器哑火
- 反应堆过载

而不是只有血条归零。

## 12.3 武器分工

建议建立明确克制：

- 实弹：
  - 对轻甲高效
  - 对护盾一般
- 激光：
  - 命中快
  - 适合点防和精确削弱
  - 受热限制
- 电浆：
  - 近中程高爆发
  - 对护盾和结构都强
- 导弹 / 鱼雷：
  - 高 alpha
  - 受点防与干扰影响大
- 轨道炮：
  - 重穿透
  - 体积和刚性要求高
- 电子压制武器：
  - 直接削弱锁定、数据链或制导

## 12.4 凝聚度与撤退

单位和舰队增加 `cohesion`：

- 队形被打散
- 指挥节点受损
- 补给断裂
- 旗舰被击毁

都会降低凝聚度。

凝聚度低时：

- 编队失序
- 拦截效率下降
- 导弹齐射协调失败
- 更容易触发自动撤退

## 13. 制空、制轨与登陆

## 13.1 制空 / 制轨

行星作战前应先争夺：

- `air_superiority`
- `orbital_superiority`

无制轨权时：

- 登陆损耗更大
- 轨道建筑暴露
- 行星护盾压力增大
- 前线更容易被轨道火力压制

## 13.2 登陆作战

登陆是恒星系层与行星层的连接器。

需要以下要素同时成立：

- 运输能力
- 轨道窗口安全
- 登陆点侦察
- 至少局部制轨
- 初始补给保证

登陆不是简单“把舰队目标设到星球”。

需要独立流程：

1. 侦察登陆点
2. 压制防空 / 轨道火力
3. 打开登陆窗口
4. 投送前锋
5. 建立滩头阵地
6. 接续重装与补给

## 14. 胜利目标与战争压力

战争不只靠“全歼所有单位”。

支持多种胜利路径：

- 摧毁敌方主基地
- 占领关键工业星球
- 切断关键物流命脉
- 获得某恒星系持续制轨权
- 摧毁戴森关键设施
- 完成特定战役目标

这样能让战争与整个工业宇宙真正耦合。

## 15. 与现有系统的对接方案

## 15.1 公开单位目录

当前 `/catalog.units` 应从“固定单位表”演进为“两层目录”：

- `base_hulls`
- `public_blueprints`

其中：

- `base_hulls` 描述可设计底盘
- `public_blueprints` 描述已经定型、可直接生产的标准型号

当前已有的：

- `prototype`
- `precision_drone`
- `corvette`
- `destroyer`

可作为第一批预置蓝图，而不再继续把它们视为最终单位模型。

## 15.2 运行态

建议把战斗运行态拆成：

- `PlanetCombatRuntime`
- `OrbitRuntime`
- `SystemFleetRuntime`
- `WarTheaterRuntime`

避免把所有战争逻辑继续塞进单一 world runtime。

## 19. 分阶段落地路线

## Phase 1：把现有舰队线改造成“预置蓝图 + 更完整运行态”

目标：

- 把 `corvette / destroyer` 从固定成品升级为预置标准型
- 增加舰队姿态、补给、战报、更多查询
- 加入巡逻、护航、袭扰、撤退

## Phase 2：上线蓝图设计器与机甲底盘

目标：

- 提供底盘 + 组件 + 约束校验
- 先开放少量组件集
- 允许机甲和舰船都走统一蓝图管线

## Phase 3：上线补给与维修体系

目标：

- 军事补给资源
- 前线补给与断补给惩罚
- 翻修与改型

## Phase 4：上线制轨、登陆、战区

目标：

- 恒星系层封锁
- 行星登陆
- 战区自治

## Phase 5：上线电子战、隐形、导弹战

目标：

- 情报可信度
- 诱饵、干扰、点防协同
- 导弹齐射控制

## 20. 平衡基线

为避免系统一上线就被单一最优解垄断，建议预设以下平衡铁律：

1. 高火力必然更贵、更热、更暴露。
2. 高装甲必然更慢、更吃补给。
3. 高隐形必然牺牲火力、感测或持续功率。
4. 高航速必然压缩装甲、载荷或续航。
5. 导弹爆发强，但必须被点防、拦截和电子战克制。
6. 重型旗舰不能脱离护航独自成为万能解。
7. 地面与太空必须互相支援，不能让某一层级完全决定胜负。

## 21. 示例蓝图

## 21.1 `Falcon Mk1` 轻护卫

- 底盘：`corvette_hull`
- 角色：侦察 / 骚扰 / 护航
- 组件：
  - 中型引擎
  - 轻型反应堆
  - 双联近防炮
  - 小型导弹舱
  - 轻护盾
  - 被动传感器
- 特征：
  - 速度高
  - 点防强
  - 正面火力弱
  - 适合切运输线

## 21.2 `Bastion Mk2` 驱逐护航舰

- 底盘：`destroyer_hull`
- 角色：编队中枢 / 反导 / 中程火力
- 组件：
  - 中大型反应堆
  - 相控阵雷达
  - 双层点防
  - 中型护盾

  - 护航优秀

- 底盘：`heavy_frame`

  - 重甲
  - 电浆近战炮

  - 强化伺服
  - 小型护盾
- 缺点：
  - 速度慢
  - 地形适应差
  - 对维修与弹药需求高

## 22. 明确不采用的方案

以下方案不采用：

- 为了兼容现有固定单位接口而新增多层 wrapper
- 继续只用手写单位 ID 线性膨胀战争内容
- 把战争简化成“双方单位进场后自动互秒”的黑盒
- 把舰船设计器做成纯外观换皮
- 把蓝图与编制绑死成一个概念

## 23. 结论

SiliconWorld 的战争系统应该被定义为：

一套由工业体系供养、以三层地图为战场、以蓝图设计器为军备核心、以补给与战区为长期约束、以 Tick 权威战斗和 AI Agent 协作为执行框架的星际工业战争系统。

它既不能退化成只会“造几个舰队单位”的功能补丁，也不该照搬传统 RTS 的全手操形式。

最合理的方向是：

- 行星层保留 RTS 式局部推进与前线争夺
- 恒星系层承载舰队与封锁战
- 银河层承载战区与战略目标
- 单位强度由蓝图设计、产能、补给、姿态共同决定

这条路线与项目现有架构最一致，也最能支撑未来长线扩展。
