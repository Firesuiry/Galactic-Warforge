# SiliconWorld 优化方案（2026-07）

> 基于对 server / client-web / client-cli / agent-gateway / docs 五个模块的全面调研，针对当前"项目脱轨"的四个核心问题给出分阶段方案。
> 本文档是路线图，每个任务按现有 T 编号流程（docs/process/task）拆解执行。

## 一、现状诊断（TL;DR）

| 领域 | 现状 | 结论 |
|---|---|---|
| 服务端 | 45 种指令、114 个测试文件、T122 战争回归基线已建 | **最健康的部分**，问题在架构热点与鉴权 |
| Web GUI | 能看大部分状态，但战争战术指挥闭环断裂；巨型组件；9 路 1s 轮询 | **可玩性最大瓶颈** |
| CLI | 约 80 条命令，是指令面的超集 | 健康，作为 GUI 的对照基准 |
| Agent 接入 | agent-gateway 成熟（T123 委派闭环），但只支持"gateway 拉起 LLM"单向模式 | 缺"外部 AI 主动接入"的反向契约 |
| 测试 | 单元/回归充分，但**无平衡性测试、无人类试玩验收方案** | 测的是"代码对不对"，没测"游戏好不好玩" |

### 各问题的具体证据

**问题 1：GUI 很烂**
- `PlanetCommandPanel.tsx` 2309 行、`PlanetPanels.tsx` 1261 行、`WarPage.tsx` 956 行，巨型组件 + 大量 useState/useEffect 手工同步选中态。
- WarPage 有 9 个 react-query 各自 1s 轮询，无统一实时层（server 明明有 SSE `/events/stream`），压力大且界面闪烁。
- 星图页"筛选/情报"等按钮是无 onClick 的死按钮。
- 行星地图 Canvas 2D 手绘，点击命中检测靠 getBoundingClientRect 手算，扩展性弱（pixi.js 已装但未用）。
- `docs/player/issue/` 尚有 6 条 web 问题待处理。

**问题 2：测试方案不完善**
- 已有：Go 单测 114 个、T122 官方战争回归（Go + CLI + Playwright 三层）、戴森 midgame 验证局。
- 缺失：平衡性测试方案（战争设计文档第 20 章只有"铁律"没有落地）、人类试玩验收清单、GUI 操作覆盖率检查（GUI 能点的命令 vs 指令全集的差距无人跟踪）、长局稳定性/性能基线。

**问题 3：人无法通过点点点操作游戏**
GUI 未暴露但 API/CLI 已有的命令（= 玩家被迫切 CLI）：
- 战术指挥：`fleet_assign / fleet_attack / fleet_disband`、`task_force_create / assign / deploy`、`theater_create / define_zone / set_objective`
- 军工：`que_military_production`（量产排队）、`refit_unit`（翻修改装）、`blueprint_variant`
- 通用：`attack / produce / upgrade / cancel_construction / restore_construction / cancel_research / demolish_dyson`

即：玩家能造蓝图、列装舰队，**真打起来却必须切 CLI**。此外缺统一战场视图——星系/舰队位置只有文字列表，玩家"看不懂战局"。

**问题 4：Agent 接入系统不完整**
- 已有：agent-gateway 决策循环（loop.ts）、JSON Schema 约束的 turn、军事权限边界（T123 policy.military）、多 provider（claude-cli / codex-cli / openai-compatible / minimax）。
- 缺失：
  1. 没有 `.claude/skills/`，Claude Code 无法作为玩家主动接入（现在只能被 gateway 当 provider 拉起）。
  2. 无聚合状态快照接口（agent 需多轮试探拉状态）。
  3. 错误反馈是自然语言 repair prompt，无机器可读错误码。
  4. `game-command-schema.ts` 的命令目录未导出为 JSON Schema 供外部引用。
  5. 凭证 bootstrap 无外部 AI 的最小示例。

**问题 5：其他（调研中发现）**
- `gamecore/core.go` 巨型 switch 分发 45 种指令，已成热点，每加一条指令都要改它。
- 鉴权简陋：静态 Bearer keyMap，人与 agent 无角色区分，指令体内 `issuer_type/issuer_id` 自报无校验——委派边界实际靠 client-cli/gateway 在客户端侧兜底，server 不设防。
- 军事委派校验逻辑落在 agent-gateway(TS) 而非 server(Go)，跨语言边界的一致性是风险点：任何直接打 HTTP API 的 agent 可绕过全部权限边界。

## 二、总体原则

1. **人与 AI 用同一套操作面**：GUI 的每个按钮、CLI 的每条命令、agent 的每个 action 最终都是 `POST /commands` 里的同一条指令。差距用"命令覆盖率矩阵"跟踪（见 4.2）。
2. **权限收口到 server**：委派边界（战区/任务群/命令白名单）从 agent-gateway 下沉为 server 的一等公民，客户端只做体验层校验。
3. **SSE 为主、轮询为辅**：Web 端建统一实时数据层，替代 9 路轮询。
4. **每个阶段以"一场可打完的仗"为验收**：功能对齐 T122 官方战争局，人类玩家全程只用 GUI 打完；Claude Code 只用 skill 打完。

## 三、分阶段路线图

### P0：人类可玩性闭环（最高优先级，约 2-3 个任务批次）

**目标：人类玩家在 T122 官方战争局中，全程不碰 CLI 打完一场战争。**

- **T-A1 战争指挥 GUI 补全**：在 WarPage/新页面补齐战区创建与目标设定、任务群组建/指派/部署、舰队 assign/attack/disband、量产排队、翻修。表单直接复用 `shared-client` 的命令目录，避免手写重复类型。
- **T-A2 通用操作补全**：行星页补 attack / produce / upgrade / cancel_construction / cancel_research 按钮；修掉星图页死按钮。
- **T-A3 战场态势图**：星系级战场视图（行星、舰队位置、任务群、封锁/登陆状态可视化），点击舰队/战区即出上下文操作菜单。这是"看懂战局"的关键，可先做 2D 示意图不追求华丽。
- **T-A4 统一实时层**：`shared-client` 增加基于 SSE 的状态流 hook，WarPage 等页面的 9 路轮询收敛为 SSE 推送 + 低频兜底轮询。
- **验收**：新增 Playwright 用例"纯 GUI 打完官方战争局"，并入 T122 回归。

### P1：Agent 接入系统 + Claude Code skill（与 P0 可并行）

**目标：Claude Code 通过一个 skill 就能开局、看局势、下指令、打完仗。**

- **T-B1 聚合快照 API**：server 新增 `GET /state/agent-briefing`（或按 scope 参数），一次返回：己方资源/研究/告警 + 可见战区/任务群/舰队/敌情摘要 + 可用命令目录。这是 agent（和 GUI 总览）的"一眼看懂"接口。
- **T-B2 结构化错误**：`/commands` 的失败响应带机器可读 error code + 字段级原因（当前哪里错、合法取值是什么），替代靠 repair prompt 猜。
- **T-B3 命令目录导出**：把 `game-command-schema.ts` 的 typed 命令编译导出为 JSON Schema 文件（挂到 `GET /catalog/commands`），GUI 表单、CLI 校验、agent prompt、skill 文档共用一份源。
- **T-B4 Claude Code 游戏 skill**：新建 `.claude/skills/play-siliconworld/SKILL.md`，内容包括：如何起本地服（复用 `start_official_war_test_server.sh`）、如何拿 player key、briefing → 决策 → `POST /commands` / CLI 的操作循环、常见战术手册（引用玩法指南）。skill 里让 Claude 走 CLI 或直接 HTTP 均可，但推荐走 CLI（复用越界校验）。
- **T-B5 委派权限下沉 server**：把 T123 的 policy.military（战区/任务群/命令白名单/风险动作开关）落到 server 侧：key 关联 issuer 档案，`/commands` 按档案校验。gateway/CLI 保留体验层预校验。**这是安全性收口，也让"任何外部 AI 直连 API"成为受控行为。**
- **验收**：用 skill 驱动 Claude Code 在官方战争局对战内置 agent，完整打一局并产出战报。

### P2：GUI 架构与工程质量（P0 功能补全后进行，避免边重构边加功能）

- **T-C1 组件拆解**：PlanetCommandPanel（2309 行）按命令域拆为建造/物流/科研/戴森/转移等子面板；WarPage 拆为蓝图/军工/战区/情报四个 feature 模块。命令表单统一走 T-B3 的命令目录自动生成骨架。
- **T-C2 地图渲染升级**：行星地图与战场态势图迁到 pixi.js（依赖已在），拿到场景图命中检测、缩放平移、图层管理。
  - > **2026-07-08 落地（行星地图部分）**：未走 pixi，改为**语义分层 DOM+canvas**——实体（建筑/单位/资源/物流/电力/管道/工地/敌情）成为带 `data-entity-*` 的真实 DOM 节点（`PlanetEntityLayer.tsx`/`PlanetEntityNode.tsx`），叠在只画地形/网格/迷雾/热力图的 canvas 底图之上。**理由**：pixi 仍是位图，DevTools/agent 读不到实体（与 P1 的 agent 可调试目标冲突）；DOM 层才能被 `locator('[data-entity-id=...]')` 定位。同时用 `cameraScheduler`（rAF 合帧）修好拖拽卡顿（30 次拖拽从 ~30 次重绘降到 1 次）。PNG 导出走 `renderEntitiesToCanvas` 离屏合成保全保真。pixi 依赖可后续移除（src 内 0 引用）。战场态势图（`BattlefieldMap.tsx`）形态不同（固定径向、无平移缩放），暂不共用原语。
- **T-C3 web issue 清零**：处理 `docs/player/issue/` 中 6 条待办 web 问题与 stats-01 复核。

### P3：测试与平衡（持续进行，P0 后加码）

- **T-D1 GUI 命令覆盖率矩阵**：脚本对比 `model/command.go` 指令全集 vs GUI 暴露命令 vs CLI 命令 vs agent 白名单，生成矩阵文档并进 CI，防再次"封在 API 里"。
- **T-D2 试玩验收清单**：`docs/player/` 下建标准试玩脚本（新局 30 分钟体验路径 + 战争局路径），每次大版本人工过一遍，问题按现有 issue 流程记录。
- **T-D3 平衡性测试方案**：落地战争设计文档第 20 章：用 agent-gateway 跑 AI vs AI 自动对局（固定种子、双方对称/非对称配置），产出胜率/资源曲线/单位交换比报告，作为平衡基线。
- **T-D4 长局稳定性**：官方局挂机 N 万 tick 的性能与内存基线，纳入定期回归。

### P4：服务端架构演进（穿插在上述任务的接触点做，遵循激进式演进原则）

- **T-E1 指令分发注册表**：`core.go` 巨型 switch 改为 handler 注册表，新指令零接触 core.go（按 AGENTS.md 原则直接重构所有调用点）。
- **T-E2 鉴权升级**：与 T-B5 一体：key → issuer 档案（human/agent、所属 player、权限 policy），废弃"指令体自报 issuer"。
- **T-E3 锁粒度评估**：core.go 21 处锁的竞争 profile，长局压测（T-D4）时一并观测。

## 四、执行方式

1. **任务拆解**：每个 T-xx 按 `docs/process/` 现有流程建任务卡，实现→测试→更新对应 API/CLI 文档（AGENTS.md 规则）→合并主分支。
2. **subagent 分工**：调研、批量组件拆解（T-C1）、覆盖率矩阵脚本（T-D1）、issue 清理（T-C3）等琐碎/可并行任务全部交给 subagent；主线只做方案裁决与验收。
3. **顺序建议**：P0 与 P1 并行启动（前端/后端不同人力面），P2 在 P0 验收后启动，P3 的 T-D1 可立即做（纯脚本），P4 随接触点顺带完成。
4. **每阶段的唯一验收问题**："一个从没看过代码的人（或一个只有 skill 的 Claude），能不能打完一场仗？"

## 五、风险与不做的事

- **不做**：视觉美术层面的大改（先解决"能玩"再谈"好看"）；多人联机大厅等新玩法（当前指令面还没被 GUI 消化完）。
- **风险**：T-B5 权限下沉会改动 `/commands` 契约，需同步更新 agent-gateway、CLI、Web 三端与 API 文档，建议单独任务批次并跑全量 T122 回归。
- **风险**：T-C2 渲染迁移工作量可能超预期，若 pixi 迁移受阻，保底方案是仅战场态势图用 pixi、行星地图维持 Canvas 2D。
