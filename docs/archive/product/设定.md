# AI驱动的戴森球计划 + 红警

## 1. 项目概述

### 1.1 项目定位

这是一个以**星际工业建造与战争对抗**为核心的多人在线策略游戏。
游戏整体世界观和资源发展方式类似《戴森球计划》，强调：

- 星际探索
- 工业生产链
- 自动化建设
- 跨星系扩张

同时引入类似《红色警戒》的核心战斗乐趣：

- 多玩家混战
- 基地建设与防御
- 部队生产与调度
- 战争迷雾下的情报侦察与战术博弈

本项目的独特之处在于：

- **服务端进行全部权威计算**
- **客户端初期以 AI Agent 和数据面板为主，而非高质量渲染**
- 玩家不仅可以手动操作，还可以创建和配置自己的 **AI Agent**
- AI Agent 可通过 API 观察世界、理解任务、自动执行建造/生产/战斗等操作

---

## 2. 核心设计目标

### 2.1 目标体验

玩家进入游戏后，体验应当围绕以下几个阶段展开：

1. **立足生存**
   - 在初始星球建立基础资源采集、发电、冶炼和制造能力
   - 建立第一座基地和防御体系

2. **自动化扩张**
   - 建立更复杂的工业链
   - 从单星球扩展到同恒星系中的多个星体
   - 组织物流、能源和军事网络

3. **科技突破**
   - 沿主科技树推进
   - 在宇宙探索中发现并解析特殊科技：
     - 虫族科技：生物化、快速扩张、数量压制、自适应进化
     - 星灵科技：高能、跃迁、护盾、超视距打击

4. **军事博弈**
   - 与其他玩家争夺关键资源点、要塞星球、稀有遗迹和交通节点
   - 在战争迷雾下进行侦查、突袭、防御和决战

5. **AI协同**
   - 玩家可将各类人物与单位交由 AI Agent 全权托管
   - AI Agent 在玩家授予的权限范围内可自主决策并执行任意行动
   - 玩家从“逐条下达命令”升级为“设定目标、划定权限、监督结果”
   - 高阶玩法演化为“玩家战略中枢 + Agent 自主军团”的协同体系

## 3. 世界观与地图结构

---

## 3.1 三层地图结构

游戏包含三个层级，且每一层都以**三维空间**表示。

### 第一层：星系图（Galaxy Layer）
每个节点代表一个大型天体区域或特殊空间结构，例如：

- 恒星系
- 虚空区域
- 黑洞
- 中子星遗迹
- 远古虫巢
- 星灵遗迹
- 超空间裂隙

特点：

- 是全局战略层
- 主要决定长距离航线、殖民顺序、远征目标和战略控制点
- 玩家在此层级进行：
  - 舰队远征
  - 跨恒星系殖民
  - 战略侦察
  - 跃迁/航道控制
  - 超远程战争部署

### 第二层：恒星系图（System Layer）
每个节点代表一个恒星系中的具体天体或空间区域，例如：

- 恒星
- 行星
- 卫星
- 小行星带
- 拉格朗日点空间站
- 虚空/深空区域

特点：

- 是局部战略层
- 主要决定同一恒星系内的资源布局、轨道封锁、运输线路和局部作战
- 玩家在此层级进行：
  - 行星开发
  - 轨道平台建造
  - 行星间物流
  - 舰队巡逻与拦截
  - 恒星系封锁战

### 第三层：星球内地图（Planet Layer）
代表星球或卫星内部/地表/大气层内的三维空间。

可细分为：

- 地表层
- 低空层
- 轨道近地层（可视项目规模决定是否归在本层）

特点：

- 是战术执行层
- 主要进行具体基地建设、采矿、工厂摆放、地面战斗、防御设施建设等
- 玩家在此层级进行：
  - 建筑布局
  - 资源采集
  - 工业制造
  - 地面与空中作战
  - 局部防御和前线推进

---

## 3.2 地图关系

三层地图不是互相独立，而是嵌套关系：

- 星系层中的一个“恒星系节点”可以进入到恒星系层
- 恒星系层中的一个“行星节点”可以进入到星球层
- 星球层的开发结果会反向影响上层：
  - 提供资源产出
  - 增强防御能力
  - 支持舰队补给
  - 构成战略枢纽

---

## 3.3 战争迷雾

所有层级都支持战争迷雾，但规则不同：

### 星系层迷雾
- 未侦察的星系仅显示基础轮廓或未知状态
- 特殊区域可能完全隐藏
- 玩家需依靠远征舰队、探测器、星门信标解锁信息

### 恒星系层迷雾
- 未侦察天体、轨道设施、舰队部署不可见
- 敌方可能进行伏击、封锁、伪装

### 星球层迷雾
- 与传统 RTS 类似
- 视野由雷达、侦察机、卫星、地面部队提供
- 地形、建筑、天气等可影响侦测结果

## 4. 核心玩法设计

---

## 4.1 建造玩法

### 4.1.1 基础建设
玩家从一个初始据点开始，逐步搭建：

- 采矿设施
- 能源设施
- 冶炼设施
- 制造工厂
- 物流系统
- 仓储系统
- 防御建筑
- 指挥中心

### 4.1.2 自动化生产链
参考《戴森球计划》的设计理念，强调生产链拆分和自动化协作。

基础资源示例：

- 铁矿
- 铜矿
- 硅矿
- 钛矿
- 稀有晶体
- 气体资源
- 生物质
- 暗物质碎片

中间产物示例：

- 钢材
- 电路板
- 高能合金
- 推进器
- 晶体矩阵
- 生物酶
- 灵能核心

终端产物示例：

- 建筑模块
- 载具部件
- 战舰组件
- 高级武器平台
- 科研包
- 特种单位胚体/核心

### 4.1.3 多层物流
物流也是核心玩法之一，分层设计：

- 星球内物流：传送带、运输车、无人机、管道
- 行星间物流：轨道运输船、质量投射器、穿梭机
- 星系间物流：跃迁货舰、星门运输、旗舰补给舰

### 4.1.4 当前已落地的戴森中后期与终局闭环（2026-04-04）
结合当前 `server/` 的实际实现，戴森相关玩法已经从“只有概念接口”推进到一条可复现的中后期验证路线，并补上了终局科技胜利与终局弹药产线；但仍带有明确的阶段性边界。

当前已经落地的闭环是：

1. 通过官方中后期场景直接进入气态行星 active planet，并预置中后期科技、基础资源与火箭关键材料；当前该场景还额外预置了 `integrated_logistics`、`photon_mining`、`annihilation`，但仍故意不预置 `dirac_inversion` 与 `antimatter_fuel_rod`。
2. `gas_giants` 解锁后，可以在气态行星上建造 `orbital_collector`，稳定获得 `hydrogen` / `deuterium`。
3. 官方 midgame 场景现在可以直接通过通用 `build` 验证 `advanced_mining_machine`、`pile_sorter`、`recomposing_assembler`、`artificial_star`，不再停留在“运行时有定义但官方验证场景卡科技门禁”的状态。
4. `vertical_launching` 现在会同时解锁 `vertical_launching_silo` 和配方 `small_carrier_rocket`。
5. `vertical_launching_silo` 自带默认配方 `small_carrier_rocket`，玩家不必额外手填 recipe，就能把 `frame_material`、`deuterium_fuel_rod`、`quantum_chip` 送入发射井内生产火箭。
6. `build_dyson_node` / `build_dyson_frame` / `build_dyson_shell` 可先搭出实验性戴森层脚手架。
7. `launch_rocket` 会把发射井本地已装载的 `small_carrier_rocket` 发往指定 `system_id + layer_index`；成功后会增加该层的 `RocketLaunches` 与 `ConstructionBonus`，并在已有壳面时继续推高覆盖率。
8. `ray_receiver` 不再凭空产能，它只会在太阳帆轨道能量和戴森层能量真实存在时输出；而且 Tick 顺序已经调整为先刷新太阳帆/戴森供能，再结算 `ray_receiver`，所以 `power` / `hybrid` 收益会同步反映到玩家能量、能源统计与电网供给。
9. `set_ray_receiver_mode` 已成为公开控制入口，可在 `power` / `hybrid` / `photon` 间切换；其中 `photon` 仍要求先解锁 `dirac_inversion`。
10. `mass_energy_storage` 与 `gravity_missile` 现在分别会解锁 `antimatter_capsule` 与 `gravity_missile` 配方，两者都能通过 `recomposing_assembler` 进入真实产线，终局弹药不再只是 catalog 名词。
11. 当前默认配置统一使用 `battlefield.victory_rule = hybrid`；因此 `mission_complete` 科研一旦完成，就会立即转成 `game_win`，并通过摘要、事件、审计与存档保持同一份胜利态。
12. `thermal_power_plant`、`mini_fusion_power_plant`、`artificial_star` 这类燃料型发电建筑现在统一遵循“无可达燃料就 `no_power/no_fuel`、补燃料后下一 tick 恢复 `running`”的运行态规则，不再把终局电站的空燃料状态误写成正常运转。

这条闭环对应的关键中后期科技链，当前已经明确到以下节点：

- `high_strength_crystal -> titanium_crystal`
- `titanium_alloy -> titanium_alloy`
- `lightweight_structure -> frame_material`
- `quantum_chip -> quantum_chip`
- `vertical_launching -> vertical_launching_silo + small_carrier_rocket`
- `gas_giants -> orbital_collector`
- `integrated_logistics -> pile_sorter`
- `photon_mining -> advanced_mining_machine`
- `annihilation -> recomposing_assembler`
- `dyson_component -> build_dyson_*`
- `ray_receiver -> ray_receiver`
- `dirac_inversion -> ray_receiver photon mode`
- `mass_energy_storage -> antimatter_capsule`
- `gravity_missile -> gravity_missile`

当前边界也需要在设计上明确写清：

- 这条路线目前依赖官方中后期场景来保证“气态行星 + 中后期科技 + 关键物资”同时就位；它还不等于“普通新局必然自然推进到这里”。
- 当前操作焦点仍围绕 `active planet`，但已加载行星的 runtime 会并行保存，并且已经支持在存在 foothold 的行星之间切换 active planet；因此中后期验证不再只是“锁死在一颗气态行星”，而是“以 active planet 为操作焦点的最小多行星闭环”。
- `build_dyson_*` 当前更偏系统验证入口，主要负责建立脚手架和层级结构，不是最终版的材料化、蓝图化戴森规划系统。
- 终局科技胜利和终局弹药产线已经打通，但高阶单位线仍未开放给玩家；`prototype`、`precision_drone`、`corvette`、`destroyer` 当前不应被视为可达玩法。
- 官方 midgame 场景里的 `artificial_star` 当前主要用于终局边界验证：它可建，但默认不带 `antimatter_fuel_rod`，因此不会被伪装成已正常供电。
- 当前局势读数仍以 `active planet` 为观察焦点；`summary / stats` 不会跨所有已加载行星做总汇总。

当前面向玩家的局势读数也已经收口到同一份 authoritative runtime 真相：

- `summary.players[].resources.energy` 读取的是当前 `active planet` 电力结算后的最终玩家能量，不再由查询层单独推导。
- `stats.energy_stats` 对应真实 `power_networks.supply` / `allocated` 与储能建筑当前剩余能量，因此射线接收站 `power` / `hybrid` 回灌会同步体现在摘要、统计和电网视图。
- `stats.production_stats` 只统计当前 tick 真实发生的落库 / 落站产出，覆盖 recipe 生产、资源采集和 `orbital_collector` 轨采，不再把无配方、缺料或空转建筑的理论 `throughput` 计入“总产出”。

### 4.1.5 当前已落地的多行星最小经营闭环（2026-04-03）
这轮实现之后，设计口径不能再把服务端描述成“只有单行星运行态”。当前真实落地的是一个最小但可玩的多行星闭环：

1. 运行态现在会按 `planet_id` 保留多份 world，并随存档一起保存与恢复。
2. `switch_active_planet` 已成为公开命令；玩家只要已发现目标行星，且在目标行星上有 foothold，就能切换默认操作上下文。
3. 官方 midgame 场景默认预置双据点：`planet-1-2` 用于气态采集与戴森链路验证，`planet-1-1` 用于主工厂与消耗侧验证。
4. 同一恒星系内、已加载行星之间的星际物流已经能跨行星派发与结算，可形成 `orbital_collector -> interstellar_logistics_station -> 主工厂星` 的最小资源链。

当前边界同样需要写清：

- 这不是“任意新星球自由殖民”的完整版系统。
- 当前强调的是“已加载行星 + active planet 切换 + 同星系跨行星物流”的最小闭环。
- 多星球长期扩张、自由前哨部署和更复杂的跨恒星系运输仍属于后续范围。

## 4.2 战斗玩法

### 4.2.1 战斗层次
战斗也覆盖三层：

- **星系层**：远征、战略封锁、星门争夺
- **恒星系层**：轨道控制、行星围攻、补给线截断
- **星球层**：基地攻防、资源点争夺、前线推进

### 4.2.2 战斗单位类型

#### 地面单位
- 工程车
- 轻型战车
- 重型坦克
- 导弹平台
- 机甲
- 生物突击单位
- 灵能战士

#### 空中单位
- 侦察机
- 战斗机
- 轰炸机
- 运输机
- 无人机群

#### 轨道/太空单位
- 护卫舰
- 驱逐舰
- 航母
- 运输舰
- 轨道炮平台
- 探测舰

### 4.2.3 战斗风格

## 4.3 科技玩法

---

## 4.3.1 主科技树
主科技树采用类似《戴森球计划》的递进结构，以基础工业和能源体系为主线。

主线科技大类建议：

- 基础工业
- 能源技术
- 自动化技术
- 物流技术
- 轨道建设
- 舰船工程
- 军事科技
- 高级计算与 AI 协议
- 跃迁与超空间技术

在当前实现里，主科技树的戴森中后期部分已经不再只是概念层，而是至少落地为“配方解锁 -> 建筑建造 -> 火箭发射 -> 戴森层收益 -> 射线接收反馈”的可验证链路。设计上需要把“解锁建筑”与“解锁 recipe”同等看待，不能再把 `small_carrier_rocket` 这类关键产物视为纯概念占位。

### 4.3.1.1 当前科研闭环（2026-04-03）
科研系统现在已经不再是“抽象研究点自动增长”的设计口径，而是明确改成了矩阵实物驱动：

- `matrix_lab` 在未设置生产配方时会进入研究实验室语义。
- `start_research` 必须满足两个前提：
  - 至少存在一个处于 `running` 的研究实验室。
  - 目标科技要求的每种矩阵，至少已经出现在研究实验室本地库存中。
- 后续 tick 会真实消耗研究实验室库存里的矩阵来推进科技进度，而不是继续凭空累计研究点。
- 设计上的 canonical matrix ID 已统一为：
  - `electromagnetic_matrix`
  - `energy_matrix`
  - `structure_matrix`
  - `information_matrix`
  - `gravity_matrix`
  - `universe_matrix`
- 因此，科技树、配方、查询和验证场景都必须围绕这 6 种矩阵组织，不再使用 `matrix_blue` 之类的旧颜色别名。

### 4.3.1.2 当前建筑覆盖与运行时边界（2026-04-03）
这轮收口后，下列 8 个此前容易被写成“有定义但不算可玩”的 DSP 建筑，已经全部进入真实主线：

- `advanced_mining_machine`：`photon_mining` 解锁，可作为高级采矿建筑建造。
- `pile_sorter`：`integrated_logistics` 解锁，可作为堆叠分拣建筑建造。
- `recomposing_assembler`：`annihilation` 解锁，具备双配方位高阶生产语义。
- `energy_exchanger`：`interstellar_power` 解锁，具备储能与充放电枢纽语义。
- `jammer_tower`：`signal_tower` 解锁，供电后会对范围内敌军施加减速场。
- `sr_plasma_turret`：`plasma_turret` 解锁，供电后进入高伤害炮塔结算。
- `planetary_shield_generator`：`planetary_shield` 解锁，供电后持续充能并优先吸收敌军伤害。
- `self_evolution_lab`：`self_evolution` 解锁，同时具备研究、矩阵生产和本地存储语义。

设计上需要同步更新认知：

- 这些建筑不再属于“catalog 里有名字，但不算主线已实现”的灰色地带。
- 当前仍未完成的是更完整的后期终局、黑雾深化玩法和更复杂的表现层，不是这 8 个建筑本体的可建造性与基础运行时。

### 4.3.2 外源科技
在宇宙探索中，玩家可获得两类特殊科技：

#### 虫族科技
偏向：

- 生物单位繁殖
- 自我修复
- 菌毯/感染地表
- 生物建筑
- 快速扩张
- 低成本海量兵种
- 吞噬回收

代价：

- 工业精密度较差
- 高端电子/精密生产受限制
- 可能带来感染、失控、生态污染等风险

#### 星灵科技
偏向：

- 护盾系统
- 瞬时跃迁
- 灵能侦测
- 高能武器
- 低数量高质量部队
- 远程投送
- 空间折跃建筑

代价：

- 资源消耗高
- 科技前置复杂
- 产能扩展慢

### 4.3.3 混合科技
玩家可以选择：

- 纯主科技树
- 主科技 + 虫族分支
- 主科技 + 星灵分支
- 高难度混合流派

混合科技不应只是数值叠加，而应体现系统性差异。
例如：

- 虫族物流偏生物传播网络，不依赖传统运输工具
- 星灵前线部署依赖折跃信标，而不是运输舰
- 人类主科技更稳定均衡，适合工业长期战

---

## 5. AI Agent 玩法设计

这是你项目最特别、也最值得重点设计的部分。

---

## 5.1 AI Agent 的定位

AI Agent 不是外挂，也不是简单自动脚本，而是游戏内的一等公民式控制实体。
玩家可以把 Agent 理解为：

- 助理
- 自动化指挥官
- 生产总监
- 侦察官
- 战区司令
- 资源调度器

玩家拥有最终控制权，并可将人物、部队、基地与舰队的经营与作战权限授予 Agent 托管执行。

---

## 5.2 Agent 的权限模型

Agent 默认应被设计为“高权限自主执行者”，边界不靠弱化能力，而由玩家通过权限策略显式划定。

每个 Agent 应有：

- 名称
- 所属玩家
- 模型配置
- 授权对象范围
- 行动许可集
- 可访问的数据范围
- 执行预算
- 资源调度上限
- 指令目标
- 记忆/上下文
- 安全约束与升级阈值

### 例子
一个“基地运营官”Agent 可以被授权：

- 查询本星球资源
- 查询库存、产线、建筑状态
- 建造基础设施
- 调整生产配方
- 补建电力和物流
- 调动本星球防御部队
- 发起区域级应急预案

以下高影响动作需由玩家显式授权后才能执行：

- 跨星系远征与宣战
- 战略级武器调用
- 大规模资源重分配
- 权限策略变更

---

## 5.3 Agent 类型建议

### 5.3.1 经济型 Agent
负责：

- 资源短缺分析
- 自动补建矿机/电厂/仓库
- 产线平衡
- 科研物资调度
- 物流优化

### 5.3.2 军事型 Agent
负责：

- 前线侦察
- 防御补位
- 敌军威胁评估
- 单支部队调度
- 区域守军部署

### 5.3.3 探索型 Agent
负责：

- 规划探测路线
- 探索未知恒星系
- 标注资源富集点
- 搜索遗迹与特殊事件

### 5.3.4 总指挥型 Agent
高阶 Agent，可综合经济、科技、军事做全局决策，并在授权域内持续自主执行。

---

## 5.4 Agent 的工作模式

建议支持三种工作模式，并默认优先全权托管：

### 1. 建议模式
Agent 不直接执行，只给出建议。

例如：
- “建议在北半球建立第二冶炼区”
- “建议生产 20 架侦察机”
- “建议侦查邻近 3 个恒星系”

### 2. 半自动模式
Agent 在授权范围内自动执行；策略升级动作按玩家规则进行确认或抽检。

### 3. 全权托管模式
Agent 在限定预算和权限内自主决策、持续执行并动态重规划。

---

## 5.5 Agent 的输入输出

### 输入
- 游戏世界状态摘要
- 玩家目标
- 当前权限配置
- 历史记忆
- 可用资源和预算
- 事件流

### 输出
- 结构化操作命令
- 决策解释
- 风险提示
- 任务报告

建议 Agent 不直接输出自然语言指令给核心逻辑，而是输出严格结构化命令，例如：

```json
{
  "action": "build_structure",
  "planet_id": "p-001",
  "structure_type": "smelter",
  "position": {"x": 120, "y": 45, "z": 88},
  "priority": "high"
}
```

## 6.1 玩家身份

- 初始出生位置

- `Authorization: Bearer <user_key>`

## 6.2 多人模式

支持：

- 自由混战（FFA）
- 组队对抗

后续再扩展到长期服和大世界。

## 7. 服务端权威架构

这是技术上最关键的部分。

---

## 7.1 基本原则

### 7.1.1 逻辑与渲染分离
游戏逻辑系统必须与任何具体渲染表现彻底解耦。
也就是说：

- 主服务器只负责世界状态和规则计算
- 客户端只是状态查看器和操作入口
- 后续无论接 Web 前端、桌面客户端、3D 客户端，逻辑都无需修改

### 7.1.2 服务端权威
所有实际游戏运算都在主服务器进行，包括：

- 资源结算
- 建造判定
- 单位移动
- 战斗伤害
- 视野计算
- 科技研究
- 物流运输
- Agent 指令执行

客户端和 API 只能提交“意图”，不能直接改状态。

---

## 7.2 核心服务划分建议

虽然你说“一个主服务器”，但为了后续扩展，逻辑上可以先做模块划分。

### 7.2.1 Game Core
核心世界模拟器，负责：

- Tick 驱动
- 地图状态
- 单位和建筑状态
- 战斗与经济规则
- 科技与探索
- Fog of War

### 7.2.2 Command Gateway
对外 API 接入层，负责：

- 用户认证
- 指令校验
- 限流
- 命令入队
- SSE 推送
- 权限令牌解析与策略下发

### 7.2.3 Agent Runtime
负责：

- 管理玩家创建的 Agent
- 拉取游戏状态摘要
- 调用外部大模型
- 将模型输出转换为结构化命令
- 对命令进行安全校验后提交给 Game Core
- 按权限策略驱动 Agent 的自治循环

### 7.2.4 Data Snapshot / Query Layer
负责：

- 世界状态查询
- 面板统计
- 地图摘要
- 历史记录
- 回放数据

### 7.2.5 Persistence
负责：

- 存档
- 日志
- 回放
- 玩家数据
- Agent 配置
- 权限策略快照

---

## 7.3 Tick 模型建议

建议采用固定 Tick 驱动的模拟方式，例如：

- 每秒 5~20 Tick
- 所有命令在 Tick 边界结算
- 便于回放、调试、并发控制、AI 决策

好处：

- 逻辑清晰
- 易做确定性模拟
- 便于记录与重放
- 更适合 RTS/模拟类游戏服务端

---

## 7.4 多核支持

你提到主服务器要支持多核，这非常合理。
建议不要一开始就做“全局共享状态多线程乱写”，而是做 **分区并行**。

### 分区思路
按地图层级或空间区域拆分任务：

- 不同恒星系的经济/战斗并行计算
- 不同行星内部模拟并行计算
- 全局事件在主调度线程汇总

### 每个 Tick 的流程可类似：
1. 收集玩家命令
2. 收集 Agent 命令
3. 校验命令
4. 分发到各分区并行模拟
5. 合并结果
6. 更新全局事件
7. 生成状态增量
8. 推送给订阅者

---

## 8. API 设计建议

---

## 8.1 API 风格

初期基于 HTTP 是合理的。
建议：

- **HTTP REST**：用于查询和提交命令
- **SSE**：用于订阅不断变化的状态、事件、日志
- 后期如有需要再加入 WebSocket，但初版不必复杂化

---

## 8.2 API 分类

### 8.2.1 认证与用户
- `POST /auth/login`
- `GET /me`

### 8.2.2 世界查询
- `GET /games/{game_id}/state/summary`
- `GET /games/{game_id}/galaxy`
- `GET /games/{game_id}/systems/{system_id}`
- `GET /games/{game_id}/planets/{planet_id}`
- `GET /games/{game_id}/fogmap`

### 8.2.3 命令提交
- `POST /games/{game_id}/commands`

命令示例：
- 建造建筑
- 移动单位
- 设置集结点
- 发起研究
- 创建舰队
- 派出探测器

### 8.2.4 Agent 管理
- `POST /games/{game_id}/agents`
- `GET /games/{game_id}/agents`
- `PATCH /games/{game_id}/agents/{agent_id}`
- `POST /games/{game_id}/agents/{agent_id}/run`
- `POST /games/{game_id}/agents/{agent_id}/pause`
- `POST /games/{game_id}/agents/{agent_id}/assume-control`

### 8.2.5 事件订阅
- `GET /games/{game_id}/events/stream` 通过 SSE 推送：
  - Tick 摘要
  - 战斗报告
  - 建造完成
  - 科技完成
  - 资源告警
  - Agent 执行日志

---

## 8.3 命令模型建议

所有玩家和 Agent 操作都应抽象为统一命令格式：

```json
{
  "request_id": "uuid",
  "issuer_type": "player",
  "issuer_id": "user-001",
  "game_id": "game-001",
  "tick_hint": 12345,
  "commands": [
    {
      "type": "build",
      "target": {
        "layer": "planet",
        "planet_id": "planet-77",
        "position": {"x": 12, "y": 4, "z": 88}
      },
      "payload": {
        "building_type": "solar_panel"
      }
    }
  ]
}
```

这样未来：

- 人类玩家命令
- AI Agent 命令
- 脚本测试命令
- 回放重演命令

都可以复用同一套执行管线。

---

## 8.4 API 权限模型建议

权限系统建议采用“玩家定义策略 + 服务端强校验”的双层模型。

### 8.4.1 核心对象
- `PermissionPolicy`：策略主体，定义 Agent 的可执行边界
- `ScopeBinding`：绑定可操作对象，如人物、部队、基地、舰队、星区
- `ActionGrant`：动作授权集，如建造、调度、作战、交易、外交
- `BudgetGuard`：预算阈值，如每 Tick、每分钟、每阶段资源上限
- `EscalationRule`：升级规则，定义何时自动执行、何时抽检、何时强制确认

### 8.4.2 推荐接口
- `GET /games/{game_id}/agents/{agent_id}/permissions`
- `PUT /games/{game_id}/agents/{agent_id}/permissions`
- `POST /games/{game_id}/agents/{agent_id}/permissions/validate`
- `GET /games/{game_id}/agents/{agent_id}/permissions/audit`

### 8.4.3 决策执行原则
- 默认拒绝未授权动作
- 已授权动作默认自动执行
- 超预算或跨域动作触发升级规则
- 所有权限决策写入审计日志

## 9. 客户端设计建议

---

## 9.1 初期客户端定位

你已经明确了：**初期不做复杂渲染，只做 AI Agent 相关功能 + 简单地图 + 数据看板**。
这是非常正确的，建议进一步明确为：

### 初期客户端功能
1. 登录与鉴权
2. 世界摘要查看
3. 三层地图简图浏览
4. 资源/生产/战斗数据看板
5. 事件日志流
6. 命令输入面板
7. Agent 创建与配置界面
8. 与 Agent 的聊天界面
9. 回执和错误查看

### 初期不做
- 高质量 3D 渲染

- 实时复杂特效

## 9.2 客户端交互形态

更像一个：

- 战略指挥终端
- AI 协作控制台
- 可视化数据面板
- 聊天式命令界面

例如玩家可以输入：

- “让 1 号 Agent 检查当前电力缺口并自动补足”

## 10. 大模型集成设计

---

## 10.1 玩家配置模型密钥

用户可在客户端中配置自己的大模型提供商密钥，例如：

- OpenAI
- Anthropic
- Gemini
- 本地模型网关
- 自建推理服务

这意味着游戏服务本身不一定强绑定某一家模型服务。

### 建议方式
客户端保存或代管用户模型配置，然后由 Agent Runtime 在调用时使用。
但从安全角度看，最好不要把第三方模型密钥在服务端明文长期存储。
可以考虑：

- 客户端每次启动会话时临时注入
- 或使用用户自己部署的代理网关

---

## 10.2 Agent 执行链路

建议采用如下流程：

1. Agent 被激活
2. 服务端整理该 Agent 授权范围内的世界状态摘要
3. 构造提示词与结构化上下文
4. 调用大模型
5. 模型返回结构化行动建议
6. 服务端进行：
   - JSON 校验
   - 权限策略校验
   - 资源可用性校验
   - 风险校验
7. 合法命令进入游戏命令队列
8. 执行结果通过事件流返回

---

## 10.3 Agent 安全机制

必须重点设计，否则很容易失控。

### 基础限制
- 每 Tick 最大命令数
- 每分钟预算上限
- 禁止越权查询
- 禁止访问敌方不可见区域信息
- 禁止执行高危操作（除非专门授权）

### 风险控制
- 高风险命令按升级规则自动执行、抽检或确认
- 异常行为自动熔断
- 命令执行失败过多时自动暂停 Agent
- 模型输出必须是严格 schema，不接受自由文本直接执行

## 13. 设计重点与风险

---

## 13.1 最大亮点
这个项目最大的亮点不是“像戴森球计划+红警”，而是：

**一个 AI 原生的 RTS/工业模拟游戏。**

也就是说，游戏设计从一开始就考虑：

- API 操作
- 结构化命令
- Agent 权限
- 服务端权威
- 数据面板式客户端

这和很多“先做游戏，后补 AI”完全不同。

---

## 13.2 最大风险

### 风险 1：范围过大
三层地图 + 工业链 + RTS + 多人 + 迷雾 + Agent + 服务端权威，规模极大。
必须严格分阶段。

### 风险 2：Agent 自主权配置失衡
- 授权过度且无约束：系统易出现失控链式操作
- 授权不足：AI 价值无法释放

### 风险 3：服务端复杂度
服务端权威 + 多核并行 + 可回放 + 迷雾裁剪，本身就是一个大型系统工程。

### 风险 4：信息量过载
玩家如果同时管理三层地图、工业链、战斗、Agent，很容易认知爆炸。
因此一定要做：
- 摘要视图
- 异常告警
- 自动化分工
- 层级化决策界面

## 15. 建议的设计原则总结

1. **先做服务端模拟，再做表现**
2. **先做最小闭环，再做完整宇宙**
3. **所有操作统一为命令**
4. **所有状态由服务端权威维护**
5. **AI Agent 是高权限自主执行者，边界由权限策略定义**
6. **逻辑系统与渲染系统彻底解耦**
7. **玩家体验重点是“指挥与授权”，不是拼手速**
8. **战争迷雾和情报系统必须成为核心博弈点**
9. **工业、科技、战争、Agent 四条线要互相闭环**
10. **MVP 一定聚焦：单星球建造+战斗+Agent 自动化**
