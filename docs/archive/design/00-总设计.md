# 硅基世界（戴森球后端向）总设计

## 1. 目标与范围
- 目标：在当前 `server` 中实现“戴森球计划风格”的可扩展后端，采用 API 驱动、服务端权威、Tick 结算。
- 范围：优先覆盖行星内工业、星际物流、能源网络、科技解锁、戴森球建造与战斗防御。
- 基础改造约束：
  - 星球内地图采用平面网格，不使用球面经纬度建模。
  - 玩家为“上帝视角指挥者”，系统赠送一个可执行建造/操作任务的机器人执行体。

## 2. 核心设计原则
- 服务端权威：客户端仅提交意图命令，不直接改写状态。
- Tick 结算：所有经济、物流、能源、战斗、戴森球产出在统一 Tick 中推进。
- 命令驱动：统一命令模型入队后执行，返回标准结果并发布事件。
- 跨端同构：API / CLI / Web 共用同一套公开命令语义，客户端必须通过 `request_id -> command_result` 对账最终结果，不能把同步 `accepted` 当成最终成功。
- 可见性裁剪：查询与事件按玩家可见域裁剪，支持战争迷雾与侦测。
- 可审计可回放：保留命令日志、事件、快照摘要，支持复盘与回滚。

## 3. 世界模型
- 世界分层：星系 -> 恒星系 -> 行星。
- 行星层为二维网格：`map_width * map_height`(一般为2000x2000)，每个格子可包含地形、资源点、建筑占用。
- 全局状态核心集合：
  - 玩家状态（权限、库存、科技、执行体参数）。
  - 建筑状态（定义、实例、升级拆除、运行参数）。
  - 物流状态（站点、无人机、运输舰、链路与任务）。
  - 电网状态（拓扑、供需聚合、覆盖与优先级）。
  - 战斗状态（敌对势力、探测信息、防御设施）。
  - 戴森球状态（节点、框架、壳层、太阳帆与能量结算）。

## 4. 玩家与执行体
- 玩家角色：
  - 指挥权限主体，负责下达建造、生产、研究、战斗、戴森球相关命令。
  - 不再作为机甲实体参与底层移动与碰撞建模。
- 执行体机器人：
  - 开局系统赠送并绑定至玩家。
  - 关键参数：建造效率、操作范围、并发任务数、研究加成。
  - 关键约束：建造/升级/拆除/部分生产行为需执行体可达且有并发槽位。

## 6. 实施优先级
- P0：平面地图、玩家与执行体、基础建筑、基础物流、基础能源、基础命令与事件。
- P1：完整生产链、行星/星际物流调度、科技树主干、防御战斗闭环。
- P2：戴森球节点框架壳层全流程、高阶能源、回放分析与运营指标增强。

## 7. 验收标准
- 规则正确：同一命令输入在同一状态下得到确定性结果。
- 回路闭合：采集 -> 运输 -> 生产 -> 科研 -> 解锁 -> 扩产可持续运行。
- 战斗可判定：敌我交互、伤害结算、告警与事件链完整。
- 戴森球可结算：太阳帆与结构建造后有稳定能量产出与消耗去向。
- 多端一致：相同公开命令在 API / CLI / Web 上保持同义，异步命令结果能稳定回写到对应请求。
