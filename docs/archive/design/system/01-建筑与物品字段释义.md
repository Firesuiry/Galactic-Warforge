# 建筑与物品字段释义（对应配置规范 V1）

## 1. 说明范围
- 本文解释 [00-建筑与物品配置规范.md](/home/firesuiry/develop/siliconWorld/docs/archive/design/system/00-建筑与物品配置规范.md) 中“建筑完整字段”和“物品完整字段”的语义。
- 只解释字段含义，不给具体建筑/物品的最终数值。

## 2. 建筑完整字段释义

### 2.1 基础字段
- `spec_version`：配置规范版本号；当前固定为 `v1`。
- `kind`：对象类型；建筑必须为 `building`。
- `id`：建筑唯一标识；全局唯一，且需与文件名一致。
- `name`：建筑显示名称（用于 UI、日志、事件文本）。
- `category`：一级分类（如 `collect`、`transport`、`power`）。
- `subcategory`：二级分类（如 `mining`、`smelting`、`turret`）。
- `tech_tier`：科技层级标记，用于 UI 分层与解锁排序。

### 2.2 `footprint`（占地与朝向）
- `footprint.width`：建筑占地宽度（网格数）。
- `footprint.height`：建筑占地高度（网格数）。
- `footprint.rotatable`：是否允许旋转放置。

### 2.3 `placement`（放置约束）
- `placement.allowed_terrain`：可放置地形集合。
- `placement.require_resource_node`：是否必须贴资源节点放置。
- `placement.resource_node_types`：允许贴合的资源节点类型（矿脉/油井等）。
- `placement.near_equator_only`：是否限制赤道附近放置（用于特定建筑）。
- `placement.orbit_only`：是否只能轨道层部署。
- `placement.gas_giant_only`：是否只能部署于气态行星。

### 2.4 `build`（建造与拆除）
- `build.cost`：建造成本清单（数组）。
- `build.cost[].item_id`：成本物品 ID。
- `build.cost[].count`：该物品需求数量。
- `build.build_ticks`：建造耗时（tick）。
- `build.dismantle_refund_ratio`：拆除返还比例（0~1）。

### 2.5 `unlock`（解锁条件）
- `unlock.required_tech`：前置科技 ID 列表。
- `unlock.required_buildings`：前置建筑要求（通常用于特殊解锁链）。
- `unlock.required_items`：前置物品要求（通常用于一次性解锁门槛）。

### 2.6 `power`（电力模型）
- `power.mode`：电力角色。
- `consumer`：只耗电。
- `producer`：只发电。
- `converter`：消耗某种输入并转化为电力/能量。
- `none`：不参与电网。
- `power.idle_mw`：待机功耗（mw）。
- `power.work_mw`：工作功耗（mw）。
- `power.peak_mw`：峰值功耗（mw）。
- `power.fuel_types`：可接受燃料类型集合（仅发电/转换类常用）。
- `power.generation_mw`：标准发电功率（mw）。

### 2.7 `io`（输入输出端口）
- `io.input_ports`：固体输入端口列表。
- `io.output_ports`：固体输出端口列表。
- `io.fluid_input_ports`：液体输入端口列表。
- `io.fluid_output_ports`：液体输出端口列表。
- 端口对象通用字段：
- `port`：端口名（建筑内唯一）。
- `accepts`：可接收/可输出的物品或物品组。
- `throughput_items_per_tick`：固体吞吐上限（每 tick）。
- `buffer_capacity`：端口缓存容量。
- `priority`：端口优先级（`high`/`normal`/`low`）。

### 2.8 `operation`（运行逻辑）
- `operation.modes`：支持的工作模式列表。
- `operation.default_mode`：默认工作模式。
- `operation.cycle_ticks`：单个工作周期耗时（tick）。
- `operation.recipe_pool`：可用配方池（生产建筑使用）。
- `operation.supports_proliferation`：是否支持喷涂增益。
- `operation.supports_overclock`：是否支持超频/加速模式。

### 2.9 `combat`（战斗属性）
- `combat.enabled`：是否参与战斗系统。
- `combat.hp`：基础生命值。
- `combat.shield`：基础护盾值。
- `combat.resistances`：抗性定义集合（按伤害类型减免）。
- `combat.attack`：攻击参数对象（射程、伤害、间隔、弹种等）；非战斗建筑可为 `null`。

### 2.10 `visibility`（可见性属性）
- `visibility.detectable_level`：被侦测等级（用于迷雾下信息裁剪）。
- `visibility.scout_radius`：主动侦测半径（建筑作为探测源时使用）。

### 2.11 `events`（事件声明）
- `events.emits`：该建筑可能产生的事件类型列表（用于事件字典和订阅校验）。

### 2.12 `upgrade`（升级链）
- `upgrade.next_ids`：可升级目标建筑 ID 列表。
- `upgrade.inherit_runtime_buffer`：升级时是否继承运行期缓存（I/O 缓冲、库存）。
- `upgrade.upgrade_ticks`：升级耗时（tick）。

## 3. 物品完整字段释义

### 3.1 基础字段
- `spec_version`：配置规范版本号；当前固定为 `v1`。
- `kind`：对象类型；物品必须为 `item`。
- `id`：物品唯一标识；全局唯一，且需与文件名一致。
- `name`：物品显示名称。
- `category`：物品分类（如 `raw`、`intermediate`、`component`、`fluid`）。
- `form`：物品形态（`solid`/`liquid`/`gas`）。
- `stack_space`：单件占用的堆叠空间权重（用于容量计算）。

### 3.2 `storage`（存储能力）
- `storage.stack_size`：单堆可叠加上限（通常仅固体使用）。
- `storage.can_store_in_depot`：是否可放入仓储箱。
- `storage.can_store_in_tank`：是否可放入储液罐。
- `storage.can_store_in_station`：是否可放入物流站库存槽。

### 3.3 `transport`（运输能力）
- `transport.belt_transportable`：是否可通过传送带运输。
- `transport.sorter_transportable`：是否可由分拣器搬运。
- `transport.station_transportable`：是否可被物流站处理。
- `transport.drone_transportable`：是否可由无人机运输。
- `transport.vessel_transportable`：是否可由运输舰跨星运输。

### 3.4 `fuel`（燃料属性）
- `fuel.is_fuel`：是否可作为燃料。
- `fuel.fuel_value_mj`：燃料热值（mj）。
- `fuel.fuel_type`：燃料类型（`none`/`chemical`/`hydrogen`/`antimatter`）。

### 3.5 `combat`（弹药属性）
- `combat.is_ammo`：是否可作为弹药。
- `combat.ammo_type`：弹药类型（与炮塔/武器兼容校验关联）。
- `combat.base_damage`：作为弹药时的基础伤害参数。

### 3.6 `production`（生产关联）
- `production.produced_by`：可产出该物品的配方 ID 列表。
- `production.consumed_by`：会消耗该物品的配方 ID 列表。

### 3.7 `tags`（标签）
- `tags`：标签数组，用于筛选、分组匹配、端口 `accepts` 规则复用。

## 4. 字段使用建议
- 单位统一：功率只用 `mw`，能量只用 `mj`，避免混用导致结算误差。
- 字段不可滥用：`category/subcategory/tags` 各司其职，不互相替代。
- 端口先收敛再扩展：V1 保持简单端口模型，复杂逻辑通过新增字段版本化演进。
- 运行态不入静态配置：库存、生产进度、断电状态等必须在运行时状态里维护。
