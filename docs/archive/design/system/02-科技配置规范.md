# 科技配置规范（V1）

## 1. 目标
- 为科技树系统提供统一、可验证、可扩展的静态配置方案。
- 科技采用“每个科技一个配置文件”。
- 配置只描述静态定义，不直接存放玩家研究进度与队列状态。
- 对齐《Dyson Sphere Program》Research 页面核心要素：`Data Volume`、`Items Needed`、`Items/Set`、`Prereqs`、`Unlocks`、`Leads To`、隐藏前置。

## 2. 配置目录结构

```text
config/
  defs/
    manifest.yaml
    techs/
      main/
        electromagnetism.yaml
      logistics/
        logistic_system.yaml
      energy/
        thermal_power.yaml
      combat/
        missile_turret.yaml
      mecha/
        drive_engine_lv4.yaml
      hidden/
        digital_analog_computation.yaml
      ...
```

规则：
- `techs/` 下每个文件只允许定义 1 个科技。
- 文件名必须等于对象 `id`：`<id>.yaml`。
- 目录按分类组织，但分类以文件内字段为准，目录仅用于管理。

## 3. 基础约定
- 格式：YAML（UTF-8，空格缩进 2）。
- 顶层版本：每个文件都必须带 `spec_version`（当前 `v1`）。
- 标识符规范：
  - `id` 只允许小写字母、数字、下划线：`^[a-z0-9_]+$`。
  - 一旦发布不可复用、不可重命名（需新建 id 并迁移引用）。
- 单位规范：
  - 研究哈希：`hash`
  - 研究时间：`tick`
  - 数量：`count`
- 科技图约束：
  - V1 科技依赖图必须为有向无环图（DAG）。
  - 可重复科技使用“同 id 多级”表示，不通过显式环来表达。

## 4. 清单文件（manifest）

`config/defs/manifest.yaml` 管理加载入口与一致性校验。

必填字段：
- `spec_version`: `v1`
- `load_order`:
  - `items_glob`: `defs/items/**/*.yaml`
  - `buildings_glob`: `defs/buildings/**/*.yaml`
  - `techs_glob`: `defs/techs/**/*.yaml`
- `integrity`:
  - `require_unique_ids`: `true`
  - `require_reference_resolved`: `true`
  - `strict_unknown_fields`: `true`
  - `require_acyclic_tech_graph`: `true`
- `runtime_policy`:
  - `hot_reload`: `false`（默认）
  - `fail_fast_on_boot`: `true`

## 5. 科技配置规则

### 5.1 科技文件最小结构

```yaml
spec_version: v1
kind: tech
id: electromagnetism
name: 电磁学
category: main
research:
  data_volume_hash: 360
  items_needed:
    - item_id: magnetic_coil
      count: 10
```

### 5.2 科技完整字段（V1）

```yaml
spec_version: v1
kind: tech
id: basic_chemical_engineering
name: 基础化工
name_en: Basic Chemical Engineering
category: chemical
subcategory: oil
tech_tier: 3
ui:
  icon: tech/basic_chemical_engineering
  position:
    x: 320
    y: 180

research:
  # 对齐 DSP Research 的 Data Volume
  data_volume_hash: 900

  # 对齐 DSP Research 的 Items Needed
  items_needed:
    - item_id: electromagnetic_matrix
      count: 45
    - item_id: energy_matrix
      count: 45

  # 对齐 DSP Research 的 Items/Set
  items_per_set: 2

  # 科技类型：single（一次性）| repeatable（可重复）
  mode: single

  # repeatable 模式使用
  repeatable:
    max_level: 0            # 0=非重复；-1=无限重复；>0=最多重复次数
    data_volume_scale: 1.0  # 每级 data_volume 乘数
    item_cost_scale: 1.0    # 每级 items_needed 乘数

prerequisites:
  required_techs: [plasma_extraction_refining]

  # 隐藏科技前置（对齐 DSP 中特殊掉落条件）
  hidden_requirements:
    - type: item_obtain
      target_id: silicon_based_neuron
      count: 15

unlocks:
  buildings: [oil_refinery]
  recipes: [refined_oil, hydrogen_from_refinery]
  abilities: [chemical_industry_enabled]
  units: []

# 可选：显式声明后继科技；未配置时由系统反向构图得到
leads_to: [xray_cracking]

visibility:
  hidden: false
  show_in_tree: true

metadata:
  source: dsp_reference
  tags: [matrix_research, production_unlock]
```

### 5.3 科技字段约束
- `kind` 必须为 `tech`。
- `id` 全局唯一，且与文件名一致。
- `research.data_volume_hash` 必须大于 0。
- `research.items_needed` 不可为空；每个 `count` 必须大于 0。
- `research.items_per_set` 必须大于 0，且不大于 `research.items_needed` 的条目数量。
- `research.mode=single` 时：`repeatable.max_level` 必须为 `0`。
- `research.mode=repeatable` 时：`repeatable.max_level` 必须为 `-1` 或正整数。
- `prerequisites.required_techs` 中每个科技必须存在。
- `unlocks` 中每个对象引用必须存在（建筑/配方/单位/能力字典）。
- `visibility.hidden=true` 时，`prerequisites.hidden_requirements` 建议非空。

## 6. 与 DSP Research 字段映射

- `Data Volume` -> `research.data_volume_hash`
- `Items Needed` -> `research.items_needed[]`
- `Items/Set` -> `research.items_per_set`
- `Prerequisites` -> `prerequisites.required_techs[]`
- `Unlocks` -> `unlocks.*`
- `Leads To` -> `leads_to[]`（可选）
- 隐藏科技额外条件（如物品拾取）-> `prerequisites.hidden_requirements[]`

## 7. 跨文件引用规则
- `research.items_needed[].item_id` 必须引用物品 `id`。
- `prerequisites.required_techs[]` 必须引用科技 `id`。
- `unlocks.buildings[]` 必须引用建筑 `id`。
- `unlocks.recipes[]` 必须引用配方 `id`。
- `unlocks.units[]` 必须引用单位 `id`。
- `hidden_requirements.target_id` 的引用目标由 `type` 决定（如 `item_obtain` -> 物品 ID）。
- 引用解析失败时，系统启动失败（`fail_fast_on_boot=true`）。

## 8. 加载与校验流程
- 启动阶段按 `manifest.load_order` 加载：先物品，再建筑，后科技。
- 每个文件执行 schema 校验与语义校验。
- 完成全局校验：
  - id 唯一
  - 引用闭合
  - 科技依赖图无环
  - 重复研究参数可计算（倍率与上限合法）
- 校验通过后生成只读内存索引：
  - `tech_by_id`
  - `tech_by_category`
  - `tech_prereq_graph`
  - `tech_unlock_index`

## 9. 版本演进规则
- `spec_version` 不向后兼容时，升级为 `v2` 并保留旧加载器。
- 字段演进优先追加，不直接改语义；删除字段需走迁移期。
- 重大变更需要提供迁移脚本（从旧配置自动转换）。

## 10. 后续可扩展字段（预留）
- `research.lab_constraints`：研究站类型/最小等级约束。
- `research.queue_policy`：是否允许自动入队后继科技。
- `unlocks.parameters`：解锁时附带参数（如倍率、上限值）。
- `faction_limit`：阵营专属科技限制（人类/虫族/星灵）。

## 11. 实现建议
- 服务端实现顺序：
  1. `tech` schema 与加载器
  2. 科技图构建与无环校验
  3. 解锁引用校验器（建筑/配方/单位）
  4. 重复研究成本计算器
- 测试最小集：
  - 文件名与 `id` 不一致应报错
  - 前置科技不存在应报错
  - 科技依赖图存在环应报错
  - 解锁引用不存在应报错
  - `repeatable` 参数非法应报错
  - 合法最小配置应可启动
