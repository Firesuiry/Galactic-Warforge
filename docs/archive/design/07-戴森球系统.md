# 戴森球系统设计

更新时间：2026-04-11

## 1. 设计目标
- 支持一条可通过公开 API / CLI / Web 复现的戴森中后期验证闭环，而不是只保留抽象数据结构。
- 将太阳帆、戴森层、射线接收、气态采集和多行星最小物流链一起纳入 Tick 结算。
- 保证普通新局的第一台研究站与第一段矩阵科研链可以合法起步，不再卡在“初始科技已完成但没有研究入口”的断档状态。
- 将终局科技胜利、终局弹药配方和能源统计纳入同一套 runtime 真相，避免查询层与结算层分叉。
- 保证异步命令结果有 authoritative 回写路径，避免把 `/commands` 的同步 `accepted` 误写成最终成功。
- 明确当前已实现边界，避免把实验性脚手架入口误写成完整版戴森规划系统。

## 2. 核心对象
- 太阳帆：有寿命、轨道、发射与衰减状态。
- 空间 authoritative runtime：top-level `SpaceRuntimeState`，当前负责太阳帆 orbit、空间实体计数，以及后续舰队态的承载边界。
- 戴森层：按恒星组织的多层结构容器。
- 节点 / 框架 / 壳层：构成戴森球结构单元。
- 火箭层增益：每层额外维护 `rocket_launches` 与 `construction_bonus`，用于把已发射火箭转成建造推进收益。
- 气态采集器：`orbital_collector` 只能建在 `gas_giant`，接入电网后稳定产出 `hydrogen` / `deuterium`。
- 射线接收站：`ray_receiver` 支持 `power` / `photon` / `hybrid` 三种模式，其中 `photon` 需要 `dirac_inversion`。
- authoritative 电力快照：每个 world / tick 会生成一份 `PowerSettlementSnapshot`，统一承接 `ws.PowerInputs`、电网覆盖/分配、接收站结算视图，并作为 `summary` / `stats` / `networks` 的单一事实源。
- authoritative 生产快照：每个 world / tick 会生成一份 `ProductionSettlementSnapshot`，只记录真实落库 / 落站成功的产出，用于驱动生产统计。
- 公开命令目录与命令账本：`shared-client/src/command-catalog.ts` 现在是 API / CLI / Web 共用的公开命令真相；Web 侧再用 `request_id -> command_result` 账本承接最终 authoritative 结果。
- 多行星运行态：已加载行星会并行保留 runtime，并允许通过 `switch_active_planet` 切换默认操作上下文。
- 戴森中后期配套建筑：`energy_exchanger` 作为储放能枢纽；`jammer_tower` / `sr_plasma_turret` / `planetary_shield_generator` 作为供电依赖的防御支撑；`self_evolution_lab` 兼作研究站和矩阵工厂。
- 胜利态：支持 `elimination` / `mission_complete` / `hybrid` 三种规则；当前官方配置统一使用 `hybrid`。
- Web 主玩法工作台：行星页右侧固定显示 `PlanetOperationHeader`，把当前路由行星、当前 active planet、最近命令结果与待处理命令数前置成统一上下文，而不是再把命令入口藏在次级标签里。

## 3. 建造链路
当前实际落地的中后期链路如下：

1. 科技与配方前置：
   - 普通新局现在仍只预完成 `dyson_sphere_program`，但这门 0 级科技已直接解锁 `matrix_lab + wind_turbine`；官方 `config-dev.yaml` 还会给每名玩家补一份最小科研启动包：`minerals = 240`、`energy = 100`、`electromagnetic_matrix x50`。由于 `battlefield_analysis_base` 自身不发电，第一段真实入口是先补 `wind_turbine`，再摆空 `matrix_lab` 开研究线。
   - 研究系统已改成矩阵实物驱动，`matrix_lab` 在未设置 `recipe_id` 时承担研究实验室角色。
   - 当前戴森关键链路依赖：
     - `high_strength_crystal -> titanium_crystal`
     - `titanium_alloy -> titanium_alloy`
     - `lightweight_structure -> frame_material`
     - `quantum_chip -> quantum_chip`
     - `vertical_launching -> vertical_launching_silo + small_carrier_rocket`
     - `gas_giants -> orbital_collector`
     - `dyson_component -> build_dyson_*`
     - `mass_energy_storage -> antimatter_capsule`
     - `gravity_missile -> gravity_missile`
   - 当前已补齐一批过去长期停留在“只有定义没有玩家入口”的公开科技接线：
     - `photon_mining -> advanced_mining_machine`
     - `integrated_logistics -> pile_sorter`
     - `interstellar_power -> energy_exchanger`
     - `annihilation -> recomposing_assembler`
     - `signal_tower -> jammer_tower`
     - `plasma_turret -> sr_plasma_turret`
     - `planetary_shield -> planetary_shield_generator`
     - `self_evolution -> self_evolution_lab`
   - 当前 `/catalog.techs` 会先按 runtime 支持能力做 unlock normalize；`engine` 仍保留为前置科技，但 `prototype`、`precision_drone`、`corvette`、`destroyer` 现在已经恢复为公开可研究科技；它们解锁的是载荷 recipe，不再把高阶单位误写成 `produce` 直产单位。
   - 当前 `/catalog.units` 是 `produce`、`deploy_squad`、`commission_fleet`、CLI 帮助与 shared-client 共用的 authoritative 公开单位边界；`worker` / `soldier` 仍走 `world_unit + world_produce`，`prototype` / `precision_drone` / `corvette` / `destroyer` 则走 `factory_recipe + deploy_command`。
   - 当前 `shared-client/src/command-catalog.ts` 是 API / CLI / Web 共用的公开命令真相；其中 `transfer_item`、`switch_active_planet`、`build_dyson_*`、`launch_solar_sail`、`launch_rocket`、`set_ray_receiver_mode` 都已被标记为 Web 必须覆盖的公开命令。
2. 资源准备：
   - 官方中后期验证场景现在通过 `config-midgame.yaml + map-midgame.yaml` 固定落到 `planet-1-2`，并用 `overrides.planets.planet-1-2.kind = gas_giant` 保证气态采集链可稳定复现。
   - 官方 midgame 场景现已额外预置 `integrated_logistics`、`photon_mining`、`annihilation`，可直接验证 `advanced_mining_machine`、`pile_sorter`、`recomposing_assembler`、`artificial_star`；仍故意不预置 `dirac_inversion` 与 `antimatter_fuel_rod`，以同时保留 `ray_receiver photon` 的科技门禁验证和 `artificial_star` 空燃料边界验证。
   - `orbital_collector` 在气态行星、接入电网后提供 `hydrogen` / `deuterium`。
   - 同一恒星系内、已加载行星之间已经支持最小星际物流闭环，可把气态行星资源运回主工厂星。
3. 太阳帆链路：
   - `em_rail_ejector` 先通过本地存储装载 `solar_sail`，再执行 `launch_solar_sail`。
   - `launch_solar_sail` 不再写入包级全局 orbit；太阳帆会进入 top-level `SpaceRuntimeState`，按 `player + system` 分桶，并由空间 runtime 分配唯一 `entity_id`。
   - 单次 `launch_solar_sail` 默认发射 `1` 张、最多 `10` 张；若发射器命中自身成功率失败分支，会消耗已装载太阳帆，但不会生成新的轨道实体。
4. 火箭链路：
   - `vertical_launching_silo` 自带默认配方 `small_carrier_rocket`。
   - 玩家可以通过生产或 `transfer_item` 将 `small_carrier_rocket` 装入发射井本地存储。
   - 目标层必须先由 `build_dyson_node` / `build_dyson_frame` / `build_dyson_shell` 建出至少一个脚手架。
   - `launch_rocket` 会把火箭发往指定 `system_id + layer_index`，并把结果写入该层的 `rocket_launches` / `construction_bonus`。
5. 配套控制、防御与部署：
   - `set_ray_receiver_mode` 已是公开命令，可切换 `power` / `hybrid` / `photon`；`photon` 仍要求 `dirac_inversion`。
   - `jammer_tower` 通过减速场干扰敌军扩散，`sr_plasma_turret` 提供高伤害炮塔结算，`planetary_shield_generator` 充能后优先吸收敌军伤害。
   - `self_evolution_lab` 既可在无配方状态下作为研究站，也可切回矩阵生产配方。
   - `battlefield_analysis_base` 现在就是默认部署枢纽，带 `deployment` module、本地存储，以及对 `prototype` / `precision_drone` / `corvette` / `destroyer` 的公开允许列表；它本身 `generation = 0`、runtime `consume_per_tick = 2`，因此也必须先接入电网才能执行部署命令。
   - `prototype` / `precision_drone` 会先作为载荷 item 进入部署枢纽存储，再通过 `deploy_squad` 生成 `planet runtime` 中的 `combat_squad`。
   - `corvette` / `destroyer` 会先作为载荷 item 进入部署枢纽存储，再通过 `commission_fleet` 生成 `system runtime` 中的 `fleet`；后续可用 `fleet_assign` / `fleet_attack` / `fleet_disband` 控制，并通过 `/world/planets/{planet_id}/runtime`、`/world/systems/{system_id}/runtime`、`/world/fleets` 观察。当前 `fleet_attack` 只支持同一恒星系内、目标行星 `planet runtime.enemy_forces[].id` 对应的敌军目标。
6. Web 公开入口与反馈闭环：
   - `client-web` 行星页现在已把 `PlanetOperationHeader + PlanetCommandCenter` 放到首屏，并提供 `ActivePlanetSwitcher`、研究装料卡与戴森专用 typed form，不再要求玩家退回 CLI 才能补齐中后期关键命令。
   - 窄屏/移动端不会把地图挤下首屏；地图仍保持主视图，右侧工作台会收口成 `工作台 / 选中对象 / 活动流` 三个页签，默认落在 `工作台`，而最新命令反馈和 active planet 上下文仍固定显示在页签上方。
   - 默认新局现在可以纯 Web 完成最小科研起步链：补 `wind_turbine`、摆 `matrix_lab`、向研究站执行 `transfer_item electromagnetic_matrix`、再发起 `start_research`。
   - 官方 midgame 场景现在可以纯 Web 跑通至少一条戴森主链：`switch_active_planet` -> `transfer_item` -> `build_dyson_node/frame/shell` -> `launch_solar_sail` / `launch_rocket` -> `set_ray_receiver_mode`。
   - 建造工作流现在默认只暴露“已解锁且当前主流程需要”的建筑；未解锁项与 `unlock_tech` 缺失的目录异常项只在显式高级模式里展示，避免把调试入口继续当作普通玩家主列表。
   - 建造前检查会直接展示执行体 ID 与 `distance / operateRange`，并按和服务端 `ManhattanDist` 同源的规则提示是否超范围；如果 authoritative 最终返回 `executor out of range: <d> > <r>`，前端会把它翻译成“先移动执行体再建造”。
   - `/commands` 的同步 `accepted` 只表示请求已被受理；Web 当前会先把该请求记为 `pending`，再按 `request_id` 订阅并消费后续 `command_result` authoritative 回写，把结果切到 `succeeded` / `failed` 并给出下一步提示。
   - 对 `build`，`command_result` 只代表施工任务已入队；前端会继续把同坐标 `entity_created` 与后续 `building_state_changed` 收口到同一条账本，并基于 `under_power` / `power_out_of_range` / `power_no_provider` / `power_capacity_full` 给出补供电、补发电或扩容电网的下一步提示。
   - 行星活动流当前默认模式是 `关键反馈`：会折叠 `tick_completed`、`resource_changed`、`threat_level_changed` 这类背景噪声，只保留高信号事件；玩家仍可切到 `全部事件 / 仅命令 / 仅告警`。
   - `/world/systems/{system_id}/runtime` 现在会直接暴露 `dyson_sphere` 与 `solar_sail_orbit`；`/world/planets/{planet_id}`、`/overview`、`/scene` 也都会带上 `system_id`，便于前端在跨星球操作时稳定恢复恒星系上下文。

## 4. 结算规则
- 太阳帆每 Tick 产生轨道能量并进行寿命衰减。
- 太阳帆现已脱离包级全局变量，进入 snapshot-backed 的 `SpaceRuntimeState`；保存、续档、回放、回滚与 digest 校验都会覆盖 `space_entity_counter`、`solar_sail_count`、`solar_sail_systems`、`solar_sail_total_energy`。
- 戴森层总产能由节点 / 框架 / 壳层基础产能叠加火箭推进后的覆盖率与 `construction_bonus` 共同决定。
- 火箭发射成功后：
  - 增加该层 `rocket_launches`
  - 把 `construction_bonus` 提高到最多 `0.5`
  - 若该层已存在壳面，则继续提高壳面 `coverage`
- `ray_receiver` 只消费真实戴森供能；实际可用输入为太阳帆轨道能与已建戴森层总产能之和，再受接收站 `input_per_tick` 限制。
- Tick 顺序现已固定为先刷新 `solar_sail` / `dyson_sphere` 能量，再结算 `ray_receiver`；因此 `power` / `hybrid` 的收益会在同一轮真实反映到玩家 `resources.energy`、`state/stats.energy_stats.generation` 与行星电网 `power_networks[].supply`。
- `settleRayReceivers` 当前只负责生成接收站结算视图和 `ws.PowerInputs`；随后 `finalizePowerSettlement` 会在同一 tick 统一生成 `PowerSettlementSnapshot`，并一次性写回最终 `resources.energy`、`power_networks`、`power_coverage`，避免中途写值被后续结算覆盖。
- 电网 provider 与缺电病因现在统一使用同一套 runtime 真相：`ray_receiver`、储能放电等 `ws.PowerInputs` 动态电源会被识别为有效供电源；建筑若已经 `connected=true` 但当前 tick 因短缺或分配结果为 `0` 拿不到电，状态原因统一收口为 `under_power`，只有真实覆盖/接线失败时才会落到 `power_no_*`。
- `thermal_power_plant`、`mini_fusion_power_plant`、`artificial_star` 这类燃料型发电建筑不再把“可建但没有燃料”误写成 `running`；当 `input_buffer + inventory` 中都没有可达燃料时，会进入 `no_power/no_fuel`，补回燃料后的下一 tick 再恢复 `running`。
- `ray_receiver` 模式语义：
  - `power`：只发电，停止新增 `critical_photon`，但不会清空历史输出缓冲里的光子库存
  - `photon`：只产 `critical_photon`
  - `hybrid`：先发电，再用剩余输入产光子
- 当前公开验收口径以 `summary` / `stats` / `networks` 为主；查询层内部虽然已经统一了 authoritative 的射线接收站结算快照，但 HTTP 网关 `inspect` 目前仍主要暴露建筑 runtime 与模式，不单独公开 `power.*` 明细。
- 系统层面的太阳帆 / 戴森结构验证口径现在以 `/world/systems/{system_id}/runtime` 为主；其中 `solar_sail_orbit` 与 `dyson_sphere` 都直接来自同一份 top-level `SpaceRuntimeState`。
- `settleProduction` 与 `settleOrbitalCollectors` 会在真实落库 / 落站成功的瞬间把产出写入 `ProductionSettlementSnapshot`；`stats.production_stats`、CLI 与 Web 只应消费这份快照，而不是建筑理论 `throughput`。
- `stats.production_stats` 现在也已对齐 authoritative 结算事实源：只统计当前 `active planet`、当前 tick 里真实落库 / 落站的产出，覆盖 recipe、采集和 `orbital_collector` 轨采，不再把空转建筑的理论吞吐量记成“总产出”。
- 产能分配去向：
  - 射线接收站发电
  - 临界光子链路
  - 终局能源与制造链
- 当前默认胜利规则支持 `mission_complete -> game_win`：若该科技在某个 Tick 完成，会先产生 `research_completed`，再产生 `victory_declared`，并把同一份胜利态写入摘要、审计和存档 runtime_state。

## 5. 当前边界
- `build_dyson_*` 仍是偏系统验证的脚手架入口，当前不额外扣除戴森结构材料，也不是完整版蓝图化规划系统。
- 多行星部分当前是“已加载 runtime + foothold 切换 + 同星系跨行星物流”的最小闭环，不是任意新星球自由殖民系统。
- 当前 `client-web` 已能覆盖默认新局科研起步链和官方 midgame 至少一条戴森主链，但它消费的仍然是现有公开命令与 runtime 查询，不存在额外的 Web 专用后门接口。
- 中后期链路目前主要依赖官方 `config-midgame.yaml + map-midgame.yaml` 场景来做稳定验证；普通新局虽然已经能合法建出第一台 `matrix_lab` 并用真实矩阵推进前期科研，但还没有自然推进到同一中后期阶段。
- `mission_complete -> game_win`、`antimatter_capsule`、`gravity_missile` 和高阶单位线都已进入公开最小闭环：玩家现在可以研究 `prototype` / `precision_drone` / `corvette` / `destroyer`，通过 recipe -> `transfer_item` -> `deploy_squad`/`commission_fleet` -> runtime 查询/控制完成整条公开链路验证。
- 当前高阶单位线仍是“最小公开闭环”，不是完整版 DSP 太空战争系统：`fleet_attack` 目前只支持同一恒星系内的 `enemy_force` 目标，尚未覆盖跨星系机动、玩家对玩家太空战、补给/返航等更复杂系统。
- 官方 midgame 场景里的 `artificial_star` 当前主要用于“终局建筑可建造 + 空燃料状态正确”这条边界验证，不是默认已经点亮的终局稳定供电解。
- `summary` / `stats` 当前仍保持 active planet 语义，不做跨所有已加载行星的总汇总。
