# 戴森球系统设计

更新时间：2026-04-03

## 1. 设计目标
- 支持一条可通过公开 API / CLI 复现的戴森中后期验证闭环，而不是只保留抽象数据结构。
- 将太阳帆、戴森层、射线接收、气态采集和多行星最小物流链一起纳入 Tick 结算。
- 保证普通新局的第一台研究站与第一段矩阵科研链可以合法起步，不再卡在“初始科技已完成但没有研究入口”的断档状态。
- 明确当前已实现边界，避免把实验性脚手架入口误写成完整版戴森规划系统。

## 2. 核心对象
- 太阳帆：有寿命、轨道、发射与衰减状态。
- 戴森层：按恒星组织的多层结构容器。
- 节点 / 框架 / 壳层：构成戴森球结构单元。
- 火箭层增益：每层额外维护 `rocket_launches` 与 `construction_bonus`，用于把已发射火箭转成建造推进收益。
- 气态采集器：`orbital_collector` 只能建在 `gas_giant`，接入电网后稳定产出 `hydrogen` / `deuterium`。
- 射线接收站：`ray_receiver` 支持 `power` / `photon` / `hybrid` 三种模式，其中 `photon` 需要 `dirac_inversion`。
- 多行星运行态：已加载行星会并行保留 runtime，并允许通过 `switch_active_planet` 切换默认操作上下文。
- 戴森中后期配套建筑：`energy_exchanger` 作为储放能枢纽；`jammer_tower` / `sr_plasma_turret` / `planetary_shield_generator` 作为供电依赖的防御支撑；`self_evolution_lab` 兼作研究站和矩阵工厂。

## 3. 建造链路
当前实际落地的中后期链路如下：

1. 科技与配方前置：
   - 普通新局现在仍只预完成 `dyson_sphere_program`，但这门 0 级科技已直接解锁 `matrix_lab`；官方 `config-dev.yaml` 还会给每名玩家补一份最小科研启动包：`minerals = 200`、`energy = 100`、`electromagnetic_matrix x50`。
   - 研究系统已改成矩阵实物驱动，`matrix_lab` 在未设置 `recipe_id` 时承担研究实验室角色。
   - 当前戴森关键链路依赖：
     - `high_strength_crystal -> titanium_crystal`
     - `titanium_alloy -> titanium_alloy`
     - `lightweight_structure -> frame_material`
     - `quantum_chip -> quantum_chip`
     - `vertical_launching -> vertical_launching_silo + small_carrier_rocket`
     - `gas_giants -> orbital_collector`
     - `dyson_component -> build_dyson_*`
   - 当前已补齐一批过去长期停留在“只有定义没有玩家入口”的公开科技接线：
     - `photon_mining -> advanced_mining_machine`
     - `integrated_logistics -> pile_sorter`
     - `interstellar_power -> energy_exchanger`
     - `annihilation -> recomposing_assembler`
     - `signal_tower -> jammer_tower`
     - `plasma_turret -> sr_plasma_turret`
     - `planetary_shield -> planetary_shield_generator`
     - `self_evolution -> self_evolution_lab`
2. 资源准备：
   - 官方中后期验证场景现在通过 `config-midgame.yaml + map-midgame.yaml` 固定落到 `planet-1-2`，并用 `overrides.planets.planet-1-2.kind = gas_giant` 保证气态采集链可稳定复现。
   - `orbital_collector` 在气态行星、接入电网后提供 `hydrogen` / `deuterium`。
   - 同一恒星系内、已加载行星之间已经支持最小星际物流闭环，可把气态行星资源运回主工厂星。
3. 太阳帆链路：
   - `em_rail_ejector` 先通过本地存储装载 `solar_sail`，再执行 `launch_solar_sail`。
4. 火箭链路：
   - `vertical_launching_silo` 自带默认配方 `small_carrier_rocket`。
   - 玩家可以通过生产或 `transfer_item` 将 `small_carrier_rocket` 装入发射井本地存储。
   - 目标层必须先由 `build_dyson_node` / `build_dyson_frame` / `build_dyson_shell` 建出至少一个脚手架。
   - `launch_rocket` 会把火箭发往指定 `system_id + layer_index`，并把结果写入该层的 `rocket_launches` / `construction_bonus`。
5. 配套控制与防御：
   - `set_ray_receiver_mode` 已是公开命令，可切换 `power` / `hybrid` / `photon`；`photon` 仍要求 `dirac_inversion`。
   - `jammer_tower` 通过减速场干扰敌军扩散，`sr_plasma_turret` 提供高伤害炮塔结算，`planetary_shield_generator` 充能后优先吸收敌军伤害。
   - `self_evolution_lab` 既可在无配方状态下作为研究站，也可切回矩阵生产配方。

## 4. 结算规则
- 太阳帆每 Tick 产生轨道能量并进行寿命衰减。
- 戴森层总产能由节点 / 框架 / 壳层基础产能叠加火箭推进后的覆盖率与 `construction_bonus` 共同决定。
- 火箭发射成功后：
  - 增加该层 `rocket_launches`
  - 把 `construction_bonus` 提高到最多 `0.5`
  - 若该层已存在壳面，则继续提高壳面 `coverage`
- `ray_receiver` 只消费真实戴森供能；实际可用输入为太阳帆轨道能与已建戴森层总产能之和，再受接收站 `input_per_tick` 限制。
- `ray_receiver` 模式语义：
  - `power`：只发电
  - `photon`：只产 `critical_photon`
  - `hybrid`：先发电，再用剩余输入产光子
- 产能分配去向：
  - 射线接收站发电
  - 临界光子链路
  - 终局能源与制造链

## 5. 当前边界
- `build_dyson_*` 仍是偏系统验证的脚手架入口，当前不额外扣除戴森结构材料，也不是完整版蓝图化规划系统。
- 多行星部分当前是“已加载 runtime + foothold 切换 + 同星系跨行星物流”的最小闭环，不是任意新星球自由殖民系统。
- 中后期链路目前主要依赖官方 `config-midgame.yaml + map-midgame.yaml` 场景来做稳定验证；普通新局虽然已经能合法建出第一台 `matrix_lab` 并用真实矩阵推进前期科研，但还没有自然推进到同一中后期阶段。
- 戴森系统已经能驱动真实能量反馈，中后期配套建筑也不再只是名词定义；当前仍未进入本轮范围的是完整 DSP 式宇宙矩阵终局、黑雾深层链路和更复杂的表现层。
