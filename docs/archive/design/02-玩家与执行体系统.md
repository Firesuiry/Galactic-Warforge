# 玩家与执行体系统设计

## 1. 设计目标
- 玩家作为“上帝视角指挥者”，统一通过命令接口控制世界。
- 执行体机器人作为被系统赠送的行动载体，承担落地执行能力。
- 将“权限”和“执行能力”解耦，支持后续多人协同与角色扩展。

## 2. 玩家模型
- 核心字段：
  - `player_id`、`team_id`、`role`
  - `resources`（基础资源计量）
  - `inventory`（物品库存）
  - `permissions`（命令权限集合）
  - `tech/combat_tech`（科技状态）
  - `stats`（统计信息）
- 权限语义：
  - `*` 表示全命令权限。
  - 细粒度权限按命令类型小写匹配。

## 3. 执行体机器人模型
- 核心字段：
  - `unit_id`
  - `build_efficiency`
  - `operate_range`
  - `concurrent_tasks`
  - `research_boost`
- 初始化规则：
  - 开局自动创建并绑定给玩家。
  - 参数走规范化：效率最小为 1、并发最小为 1、范围不为负。

## 4. 命令执行约束
- 权限约束：玩家无权限则命令拒绝。
- 执行体约束：建造/升级/拆除等操作需执行体存在、可达、并发可用。
- 资源约束：库存或资源不足时命令返回失败结果。
- 生命周期约束：玩家失活后仅允许部分查询或观战行为。

## 5. 玩法层表达
- 玩家不是机甲实体，不参与“玩家角色碰撞/位移”建模。
- 玩家核心行为是调度机器人、组织产线、配置物流、推进科研与战争。
- 可扩展方向：
  - 多机器人协作
  - 执行体模块化装备
  - 角色分工（指挥/后勤/科研）
