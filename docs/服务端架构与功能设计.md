# SiliconWorld 服务端架构与功能设计（不含 AI Agent）

## 1. 文档目标

本文基于现有设定，给出一份可落地的服务端架构方案，重点约束如下：

- 服务端保持权威模拟与规则计算。
- 服务端作为战斗服务器启动，启动时固定玩家数量与每个玩家的 key。
- 服务启动后直接进入战斗，不提供大厅与匹配流程。
- AI Agent 相关能力不在服务端实现。
- Agent 的理解、规划、模型调用、自然语言交互全部放在客户端。
- 客户端仅通过统一命令接口驱动服务端，不可直接修改世界状态。

---

## 2. 设计原则

### 2.1 服务端权威

所有游戏状态变更只能由服务端 Tick 结算产生，包括资源、建造、移动、战斗、科技、迷雾与事件。

### 2.2 逻辑与表现解耦

服务端只维护可验证的世界模型和规则，不依赖具体渲染形式；Web、桌面、3D 客户端均通过同一 API 访问。

### 2.3 命令驱动

客户端提交“意图命令”，服务端做鉴权、校验、排队、结算，返回结果与事件流。

### 2.4 可重放与可审计

服务端以 Tick 为最小结算单位，记录命令与关键快照，支持回放、审计和问题定位。

### 2.5 AI 职责外置

服务端不提供 Agent 实体、Agent Runtime、模型调用链路。客户端可将 AI 产出的动作转换为标准命令后提交。

---

## 3. 架构总览

```text
[Client / AI-Enhanced Client]
      |  HTTP REST + SSE
      v
[Command Gateway]
      |  validated commands
      v
[Command Queue] ---> [Audit Log]
      |
      v
[Game Core (Tick Loop)]
      |         \
      |          \-> [Fog/Visibility Engine]
      |
      +--> [State Snapshot / Query Layer]
      |
      +--> [Persistence (DB + Replay + Event Store)]
```

---

## 4. 核心模块

## 4.1 Command Gateway（接入层）

职责：

- 基于启动参数进行 key 鉴权。
- 命令基础校验（格式、字段、签名、幂等键）。
- 限流与反滥用控制。
- 命令入队与请求追踪。
- SSE 订阅管理与事件推送。

不负责：

- 业务规则结算。
- 直接修改世界状态。
- AI 推理、自然语言解析、Agent 生命周期管理。

## 4.2 Command Queue（命令队列）

职责：

- 缓冲玩家并发命令。
- 按 Tick 边界批处理。
- 支持优先级与过期策略。
- 提供去重和重试保障。

## 4.3 Game Core（权威模拟内核）

职责：

- 固定 Tick 驱动模拟。
- 三层地图状态推进（星系/恒星系/星球）。
- 经济系统结算（资源、产线、物流、能源）。
- 战斗系统结算（移动、攻击、防御、损伤、销毁）。
- 科技系统结算（研究进度、解锁、分支限制）。
- 战争迷雾与可见性计算。
- 命令执行结果与事件产出。

## 4.4 Visibility Engine（迷雾与可见性）

职责：

- 为不同玩家计算可见集合。
- 对查询结果和事件做裁剪。
- 阻断越权信息访问。

## 4.5 Query Layer（查询与快照层）

职责：

- 输出世界摘要、地图节点详情、战报与统计面板数据。
- 提供 Tick 快照与历史区间查询。
- 保障查询接口与模拟内核解耦。

## 4.6 Persistence（持久化层）

职责：

- 玩家数据、世界状态、命令日志持久化。
- 关键 Tick 快照存储。
- 回放数据生成与归档。
- 审计日志持久化。

---

## 5. 客户端与服务端职责边界

### 5.1 客户端负责

- AI Agent 的提示词、记忆、策略编排。
- 第三方模型调用与密钥管理。
- 自然语言到结构化命令的转换。
- 本地 Agent 任务编排与多轮决策。

### 5.2 服务端负责

- 命令鉴权、校验与权威执行。
- 世界状态推进与规则裁决。
- 可见性裁剪与事件分发。
- 回放、审计、故障恢复。

### 5.3 交互契约

客户端不论是手动操作还是 AI 驱动，都必须提交同一命令格式，服务端不区分“人类命令”与“AI 生成命令”的执行通道。

---

## 6. Tick 与并行模型

### 6.1 Tick 机制

- 建议 5~20 Tick/秒。
- 所有命令仅在 Tick 边界结算，客户端不能指定 Tick。
- 命令发出后在最近完成的 Tick 执行结算；校验失败则返回失败且不修改状态。

### 6.2 单 Tick 流程

1. 拉取待处理命令批次。
2. 执行权限与上下文校验。
3. 按空间分区并行计算（恒星系/行星分区）。
4. 合并分区结果并处理全局事件。
5. 生成状态增量、战报、告警。
6. 写入快照与审计日志。
7. 向订阅端推送事件。

### 6.3 并行策略

- 采用分区并行，避免全局共享状态争写。
- 全局一致性步骤（跨区结算、全局资源结算）在主调度阶段完成。

---

## 7. 功能清单（服务端）

### 7.1 启动配置与密钥校验

- 启动参数输入玩家数量、玩家标识与 key 列表。
- 加载后固定玩家身份映射，不支持运行时增删。
- 对每次请求执行 key 校验并映射到玩家身份。

### 7.2 战场运行控制

- 服务启动即创建单个战场实例并进入运行态。
- 按战场配置执行胜利判定、结束结算与归档。
- 不提供创建、加入、退出、大厅、匹配相关能力。

### 7.3 世界模拟

- 三层地图实体管理。
- 资源采集、制造、物流、能源网络。
- 建筑建造、升级、拆除。
- 单位生产、移动、编队、战斗。
- 科技研究与分支解锁。
- 战争迷雾与侦察机制。

### 7.4 命令系统

- 统一命令协议接入。
- 命令校验、排队、执行、回执。
- 幂等与去重处理。

### 7.5 查询与订阅

- 世界摘要查询。
- 分层地图查询。
- 资源/战斗/科技统计查询。
- 事件流订阅（SSE）。

### 7.6 数据与运维

- 存档与恢复。
- 回放导出。
- 审计与操作追踪。
- 健康检查与运行指标输出。

### 7.7 启动参数示例

```yaml
battlefield:
  map_seed: "seed-001"
  max_tick_rate: 10
  victory_rule: "elimination"
players:
  - player_id: "p1"
    key: "key_player_1"
  - player_id: "p2"
    key: "key_player_2"
```

## 8. API 设计（不含 Agent 管理接口）

## 8.1 鉴权方式

- `Authorization: Bearer <player_key>`

## 8.2 世界查询

- `GET /state/summary`
- `GET /galaxy`
- `GET /systems/{system_id}`
- `GET /planets/{planet_id}`
- `GET /fogmap`

## 8.3 命令提交

- `POST /commands`

## 8.4 事件订阅

- `GET /events/stream`

---

## 9. 统一命令模型

```json
{
  "request_id": "uuid",
  "issuer_type": "player_or_client_agent",
  "issuer_id": "user-001",
  "commands": [
    {
      "type": "build",
      "target": {
        "layer": "planet",
        "planet_id": "planet-77",
        "position": {"x": 12, "y": 4, "z": 88}
      },
      "payload": {
        "building_type": "solar_plant"
      }
    }
  ]
}
```

说明：

- `issuer_type` 仅用于审计来源，不改变执行权限模型。
- 权限依据玩家身份与当前战场上下文判定，而不是依据是否 AI 生成。
- 所有命令通过同一执行管线，保证一致性与可回放性。
- 命令模型不支持指定执行 Tick。

---

## 10. 非目标（当前阶段不实现）

- 服务端 Agent Runtime。
- 服务端模型供应商适配与密钥托管。
- 服务端自然语言解析与对话编排。
- 服务端 Agent 权限策略子系统。

以上能力统一由客户端实现，服务端仅暴露标准命令与查询协议。
