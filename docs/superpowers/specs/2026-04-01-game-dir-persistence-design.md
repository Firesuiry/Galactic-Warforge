# Game Directory Persistence Design

## 1. 背景

当前 `server` 已经会把快照写到 `data_dir`，但它只是在运行中向磁盘写文件，并没有在下次启动时从这些文件恢复世界状态。

因此当前实际行为是：

- 服务端启动时总是新建一局世界
- 同一个进程不退出时，客户端可以继续当前局
- 服务端一旦重启，即使磁盘上残留旧快照，也不会续上上一局

这与“每场游戏以一个文件夹为工作目录，重新从该目录启动时继续上一局”的目标不一致。

本次设计要把 `data_dir` 从“快照输出目录”提升为“单局游戏工作目录”，让它具备以下能力：

- 空目录开新局
- 已有目录续旧局
- 自动保存
- CLI 和 Web 手动保存

## 2. 目标

### 2.1 产品目标

本次设计要实现以下用户可见行为：

1. 每场游戏对应一个独立的工作目录。
2. 用空目录启动时，新建一局游戏并立即落盘。
3. 用已有有效存档的目录启动时，直接继续上一局。
4. 游戏运行中自动每分钟保存一次。
5. 玩家可以通过 CLI 和 Web 手动触发保存。

### 2.2 已确认决策

本次已经确认的设计决策如下：

- 采用方案 A：单目录、单权威存档文件，不做多槽位。
- 只刷新 `save.json`，不做命名存档点。
- `gameplay config` 与地图配置保存在目录内部，并在续档时优先于外部配置。
- 纯运行参数允许外部配置覆盖，不随存档锁死。
- 第一版不持久化 RNG 状态。
- 第一版只保证“世界状态恢复后可继续玩”，不保证“续档后随机事件与不停服时完全一致”。
- `client-web` 也要提供手动保存入口，不只做 `server + CLI`。

## 3. 范围

### 3.1 范围内

- 为 `server` 引入可恢复的游戏目录持久化机制。
- 让现有 `server.data_dir` 正式承担“单局游戏目录”职责。
- 在目录中持久化：
  - 游戏初始化配置
  - 地图配置
  - 世界快照
  - 运行所需的调试/历史状态
- 新增手动保存接口 `POST /save`。
- 新增 CLI `save` 命令。
- 在 Web 顶栏增加手动保存按钮。
- 更新相关文档，至少包括：
  - `docs/服务端API.md`
  - `docs/cli.md`
  - `docs/玩家玩法指南.md`
  - `docs/client-web使用说明.md`

### 3.2 范围外

- 不做多槽位存档。
- 不做命名存档点。
- 不做“加载某个历史存档点”界面。
- 不做 RNG 状态持久化。
- 不做跨版本存档迁移器。
- 不做新的独立游戏管理 UI。

## 4. 总体设计

### 4.1 基本思路

将 `server.data_dir` 视为“这一局游戏的根目录”。启动时根据目录内容决定是“新建”还是“恢复”：

- 空目录：创建新局
- 目录内存在合法 `meta.json` 与 `save.json`：恢复旧局
- 目录缺文件或文件损坏：拒绝启动

服务端内存中仍保留现有的 `GameCore`、`CommandLog`、`EventHistory`、`AlertHistory` 等结构。保存时将它们序列化到 `save.json`；恢复时再反序列化回运行时结构。

### 4.2 目录结构

第一版的游戏目录结构固定为：

```text
<game_dir>/
  meta.json
  save.json
```

说明：

- `meta.json` 保存这局游戏的身份、初始化配置、地图配置和保存元信息。
- `save.json` 是唯一权威存档，启动恢复只认它。

第一版不再额外使用目录内 `snapshots/`、`history/`、`manual/` 等子目录，避免范围扩张。

## 5. 文件格式

### 5.1 `meta.json`

`meta.json` 表示“这局游戏是按什么规则创建出来的”。建议字段如下：

```json
{
  "format_version": 1,
  "created_at": "2026-04-01T12:00:00Z",
  "last_saved_at": "2026-04-01T12:10:00Z",
  "gameplay_config": {
    "battlefield": {},
    "players": []
  },
  "map_config": {},
  "config_fingerprint": "sha256:...",
  "map_fingerprint": "sha256:..."
}
```

语义约束：

- `gameplay_config` 只保存会影响这局内容的配置：
  - `battlefield`
  - `players`
- `map_config` 保存这局使用的地图配置快照，而不是只存路径。
- 指纹字段用于诊断与日志，不作为第一版启动是否可恢复的阻断条件。

### 5.2 `save.json`

`save.json` 表示“当前这一刻的可恢复状态”。建议字段如下：

```json
{
  "format_version": 1,
  "saved_at": "2026-04-01T12:10:00Z",
  "tick": 4386,
  "snapshot": {},
  "runtime_state": {
    "active_planet_id": "planet-1-1",
    "winner": ""
  },
  "debug_state": {
    "command_log": [],
    "event_history": [],
    "alert_history": [],
    "audit_log": []
  }
}
```

其中：

- `snapshot` 复用现有 `snapshot.Snapshot` 结构，包含：
  - `world`
  - `discovery`
- `runtime_state` 保存无法直接从世界快照稳定推导出的运行时字段。
- `debug_state` 用于保证重启后这些现有能力仍然可用：
  - `rollback`
  - `replay`
  - `audit`
  - 事件/告警历史查询

### 5.3 不保存 RNG

第一版明确不保存随机数状态。

这意味着：

- 恢复后当前世界状态与保存时一致
- 恢复后未来的随机判定结果不保证与“服务端不停机继续跑”完全一致

该限制会在文档中明确说明。

## 6. 启动与恢复规则

### 6.1 新建游戏

当 `game_dir` 不存在、为空目录，或第一次被指定时：

1. 创建目录。
2. 用外部 `config` 和 `map-config` 初始化新世界。
3. 构建 `meta.json`。
4. 构建首个 `save.json`。
5. 完成启动。

要求：

- 首次启动不允许只写 `meta.json` 不写 `save.json`。
- 首次落盘必须在对外提供服务前完成，避免出现“起服成功但没有初始存档”的窗口。

### 6.2 恢复旧局

当 `game_dir` 中已经存在合法 `meta.json` 和 `save.json` 时：

1. 读取 `meta.json`。
2. 读取 `save.json`。
3. 以 `meta.json` 中的 `gameplay_config` 和 `map_config` 作为这局的真实配置来源。
4. 用 `save.json` 中的 `snapshot` 恢复世界与发现状态。
5. 恢复 `runtime_state` 与 `debug_state`。
6. 再将本次外部传入的纯运行参数覆盖到服务端运行配置中。

### 6.3 配置优先级

配置优先级明确为：

- 游戏内容相关配置：目录内部优先
- 纯运行参数：外部配置优先

第一版中，以下内容视为游戏内容相关配置：

- `battlefield`
- `players`
- `map-config` 全量内容

以下内容视为纯运行参数：

- `server.port`
- `server.rate_limit`
- `server.event_history_limit`
- `server.snapshot_max_events`
- `server.alert_history_limit`
- 自动保存周期配置

### 6.4 运行参数覆盖的后果

恢复旧局时，如果外部传入更小的历史上限：

- 允许覆盖
- 恢复后的 `event_history`、`alert_history` 等内存结构按新的运行上限裁剪

这属于运行策略变化，不视为存档冲突。

## 7. 保存机制

### 7.1 自动保存

第一版自动保存规则固定为：

- 默认每 `60` 秒保存一次

保存触发以 wall-clock 为准，不依赖 tick 数。

自动保存是服务端内部行为，不要求客户端主动轮询。

### 7.2 手动保存

第一版提供两类手动入口：

- CLI：`save`
- Web：顶栏 `保存` 按钮

两者都调用统一的服务端保存逻辑，不在客户端本地做任何持久化决策。

### 7.3 原子写入

`save.json` 的写入必须采用原子替换流程：

1. 生成完整保存载荷
2. 写入 `save.json.tmp`
3. 刷盘
4. 原子重命名覆盖 `save.json`
5. 更新 `meta.json.last_saved_at`

目标是避免进程崩溃后留下半截 `save.json`。

### 7.4 保存失败语义

- 手动保存失败：
  - API 返回错误
  - 当前游戏继续运行
- 自动保存失败：
  - 仅记录错误日志
  - 当前游戏继续运行
  - 下一个自动保存周期继续尝试

## 8. 服务端 API

### 8.1 新增接口

新增：

- `POST /save`

接口语义：

- 认证后立即触发一次保存
- 不做多槽位选择
- 不接受命名存档参数

请求体第一版可为空，或只接受可选调试字段：

```json
{
  "reason": "manual"
}
```

响应建议：

```json
{
  "ok": true,
  "tick": 4386,
  "saved_at": "2026-04-01T12:10:00Z",
  "path": "/path/to/game_dir/save.json",
  "trigger": "manual"
}
```

### 8.2 不新增读取接口

第一版不新增“查询存档列表”“查询存档详情”“加载指定存档点”接口。

恢复行为只发生在服务端启动阶段。

## 9. CLI 设计

CLI 新增命令：

```bash
save
```

行为：

- 调用 `POST /save`
- 输出：
  - 当前 tick
  - 保存时间
  - 保存路径

示例输出：

```text
Saved at tick 4386
Time: 2026-04-01T12:10:00Z
Path: /path/to/game_dir/save.json
```

CLI 不新增：

- `save --slot`
- `save --name`
- `load`

## 10. Web 设计

### 10.1 入口位置

Web 的手动保存入口放在顶栏 `TopNav`。

原因：

- 保存是会话级动作，不属于单个页面
- 顶栏已经承载全局导航和会话级信息，适合增加该入口

### 10.2 交互

在线模式下：

- 顶栏显示 `保存` 按钮
- 点击后调用 `POST /save`
- 成功后显示轻量反馈，例如：
  - `已保存到 tick 4386`
- 失败后显示错误提示

离线 fixture 模式下：

- `保存` 按钮禁用
- 或提示“离线样例不可保存”

第一版不增加：

- 保存历史弹窗
- 存档管理页面
- 自动保存状态面板

## 11. 错误处理

### 11.1 启动期错误

以下情况必须拒绝启动：

- `meta.json` 缺失
- `save.json` 缺失
- `meta.json` JSON 损坏
- `save.json` JSON 损坏
- 存档版本不支持
- `save.json` 中快照恢复失败

错误信息必须直接指出：

- 哪个文件出错
- 是缺失、解析失败还是版本不兼容

### 11.2 运行期错误

运行中保存失败不应终止 tick loop 或 HTTP 服务。

原因：

- 保存失败属于运维故障，不应直接摧毁当前对局
- 玩家应当有机会修复磁盘问题后再次触发保存

## 12. 测试策略

### 12.1 服务端单元测试

必须覆盖：

- 空目录启动会创建 `meta.json` 和 `save.json`
- 已有合法目录启动会恢复到保存时 tick
- 恢复后世界状态与保存时一致
- 恢复时忽略外部 gameplay config 冲突，采用目录内配置
- 恢复时允许外部运行参数覆盖
- 手动保存成功落盘
- 自动保存按周期落盘
- 缺文件、坏 JSON、坏版本会拒绝启动
- 保存失败不会打断游戏运行

### 12.2 API 测试

必须覆盖：

- `POST /save` 成功
- `POST /save` 失败
- 鉴权规则正确

### 12.3 CLI 测试

必须覆盖：

- `save` 帮助文案
- `save` 成功输出
- `save` 错误输出

### 12.4 Web 测试

必须覆盖：

- 在线模式下保存按钮可点击
- 成功提示显示
- 失败提示显示
- fixture 模式下按钮不可用

## 13. 文档更新

本次实现完成后必须同步以下文档：

- `docs/服务端API.md`
  - 新增 `POST /save`
  - 更新 `data_dir` 的真实语义
  - 说明新局/续档行为
- `docs/cli.md`
  - 新增 `save`
- `docs/玩家玩法指南.md`
  - 增加“如何新开一局 / 如何继续上一局”
- `docs/client-web使用说明.md`
  - 说明顶栏手动保存入口

## 14. 非目标与限制

第一版明确限制如下：

- 不支持多槽位
- 不支持命名存档点
- 不支持从历史点位选择加载
- 不保证重启后的随机过程与不停服逐 tick 演化完全一致
- 不提供图形化存档管理界面

## 15. 结论

本次设计将 `server.data_dir` 固化为“单局游戏工作目录”，并通过 `meta.json + save.json` 两文件模型完成：

- 新局创建
- 旧局恢复
- 自动保存
- CLI / Web 手动保存

该方案的重点不是做复杂存档系统，而是把“重启后能继续玩”作为第一优先级，用最小且可靠的结构先打通真实续档能力。
