# Client-Web 智能体协作页重构设计

## 1. 背景

`client-web` 当前的 `/agents` 页面已经从单智能体调试台演进到了 IM 风格工作区，但实际信息架构仍然偏“表单控制台”：

- 创建频道、按星球拉人、定时任务创建都直接平铺在主页面。
- 页面缺少明确的“成员”入口，用户很难找到如何先创建成员，再把成员拉入频道。
- 模板能力已经存在于 `agent-gateway`，但 `client-web` 的主界面没有把模板纳入可发现的用户路径。
- 主聊天区、会话管理、成员管理、任务管理同时暴露，导致主页面不像真正的即时通信工具，而更像一个一次性把所有功能堆出来的运维面板。

用户已明确要求将这一页重构成更接近即时通信工具的结构：

- 左侧应有类似“频道列表 / 联系人列表”的结构。
- 新建频道放在频道列表侧边按钮。
- 拉人进频道放在频道设置里的成员区域。
- 定时任务放在成员配置里，而不是频道页主界面。
- 创建成员与创建模板应是可被发现的显式入口；新建成员后默认不属于任何频道，后续再由频道手动拉入。

## 2. 已确认约束

- 保持在 `/agents` 内完成重构，不拆成新的一级页面。
- 采用双栏 + 内页式管理，而不是三栏常驻详情面板。
- 左侧保留两个一级工作入口：`频道`、`成员`。
- 模板不进入左侧一级导航，只作为成员体系下的二级管理能力出现。
- 成员是工作区级资源，不属于任何频道；频道只负责拉入现有成员。
- 私聊从成员页发起，不在左侧提供单独的“新建私聊”按钮。
- 定时任务归属成员；任务可在成员配置里创建、启停、编辑，并选择投递目标。
- 主界面应以聊天流为核心；频道相关设置通过页内切换进入，而不是常驻首页。

## 3. 目标与非目标

### 3.1 目标

1. 将 `/agents` 重构成双栏 IM 工作区，主路径清晰区分“会话协作”和“成员管理”。
2. 让“创建成员”“绑定模板”“发起私聊”“把成员拉入频道”“配置成员定时任务”都能在界面中自然找到。
3. 将频道管理动作收束到频道上下文内部，将成员管理动作收束到成员上下文内部，避免跨上下文表单堆叠。
4. 复用现有 `agent-gateway` 的会话、成员、模板、任务能力；仅在必要位置补足缺失接口。
5. 为后续继续扩展权限、模板管理、任务编辑留出稳定的页面状态结构。

### 3.2 非目标

- 本轮不把协作系统再改回三栏常驻详情布局。
- 本轮不把模板提升为 `/agents` 的一级导航对象。
- 本轮不做复杂的工作区、多租户、云同步或跨机器协作。
- 本轮不改变 `server/` 的职责边界，协作层仍留在 `client-web` + `agent-gateway`。

## 4. 现状审计

### 4.1 已有能力

`agent-gateway` 目前已经具备以下路由能力：

- `GET /agents`
- `POST /agents`
- `GET /templates`
- `POST /templates`
- `GET /conversations`
- `POST /conversations`
- `GET /conversations/:id/messages`
- `POST /conversations/:id/messages`
- `POST /conversations/:id/members/invite-by-planet`
- `GET /schedules`
- `POST /schedules`

也就是说，后端已经具备“创建成员”“创建模板”“创建频道”“创建定时任务”的基本能力，当前问题主要在 `client-web` 未暴露这些路径。

### 4.2 当前缺口

这次重构需要顺带补齐两个实现缺口：

1. 缺少“把指定成员加入频道”的网关接口。
   当前只有“按星球批量拉人”，无法支撑频道设置中的精确拉人。
2. 定时任务模型还不是“归属成员”。
   当前 `ScheduleJob` 只有 `creatorType`、`creatorId` 和投递目标，没有明确的成员归属字段，前端无法稳定地在成员详情页筛选和管理该成员的任务。

因此，本轮除了前端信息架构调整，还需要对 `agent-gateway` 做最小必要的 API 和模型补强。

## 5. 选定方案

采用“方案 2：双栏 + 内页式管理”。

页面从“左中右三栏同时暴露多个管理表单”改为：

- 左侧：工作入口与实体列表
- 中间：当前实体的主内容区

这不是极致 Slack/Discord 式布局，但能在当前代码基础上较低风险地实现下列目标：

- 频道页保持聊天优先
- 成员页承载模板和任务管理
- 各类配置入口退到页内，不再常驻首页
- 页面状态比现有大表单堆叠更稳定

## 6. 页面信息架构

### 6.1 左侧导航

左侧区域固定分为两个层次：

1. 一级入口切换：`频道`、`成员`
2. 当前入口下的列表视图

选中 `频道` 时：

- 左侧列表上半部分显示频道列表
- 左侧列表下半部分显示私聊列表
- 频道标题右侧显示 `新建频道` 按钮

选中 `成员` 时：

- 左侧显示成员列表
- 成员标题右侧显示 `新建成员` 按钮

模板不占用左侧一级入口。

### 6.2 中间内容区

中间区域一次只承载一个工作上下文：

- 频道聊天页
- 频道设置页
- 成员详情页
- 成员编辑页
- 模板管理页
- 成员定时任务编辑页
- 空状态页

页面不再在同一屏同时展示频道成员、按星球拉人、定时任务创建和主消息区。

### 6.3 顶层状态模型

顶层 UI 状态建议收敛为：

```ts
type AgentsPane = 'channels' | 'members';

type AgentsContentView =
  | 'empty'
  | 'channel-chat'
  | 'channel-settings'
  | 'member-detail'
  | 'member-editor'
  | 'template-manager'
  | 'member-schedule-editor';

interface AgentsWorkspaceState {
  activePane: AgentsPane;
  selectedChannelId: string | null;
  selectedAgentId: string | null;
  contentView: AgentsContentView;
}
```

现有零散的 `showCreateChannel`、`invitePlanetId`、`scheduleMessage` 这类页面级状态，应退回各自的子视图内部维护，避免 `AgentsPage` 继续膨胀。

## 7. 频道工作流设计

### 7.1 频道列表

左侧频道视图包含两段列表：

- `频道`
- `私聊`

`新建频道` 入口放在频道列表标题右侧，交互为在中间区打开轻量新建视图，字段仅包含：

- 频道名称
- 频道主题

创建成功后直接进入该频道聊天页。

私聊不提供独立新建按钮，统一从成员页的“发起私聊”进入。

### 7.2 频道聊天页

选中频道后，中间区默认进入聊天页，结构如下：

- 频道头部：名称、主题、成员数、`频道设置` 按钮
- 主体：消息流
- 底部：消息输入框与发送按钮

聊天页只保留会话本身相关元素，不再常驻展示成员列表、拉人表单或定时任务表单。

### 7.3 频道设置页

点击 `频道设置` 后切入页内设置视图，建议分为三个页签：

- `概览`
- `成员`
- `协作`

其中：

- `概览` 显示频道名称、主题、创建者、最近活动信息。
- `成员` 显示当前成员列表，并提供“从已有成员中拉人进频道”的入口。
- `协作` 仅保留与频道本身强相关的协作说明或限制摘要，不承载成员级定时任务。

### 7.4 频道拉人

频道设置的 `成员` 页签应支持两种加入方式：

- 精确拉人：从现有成员池中勾选一个或多个成员加入当前频道
- 按星球批量拉人：保留现有能力，作为次级入口

如果当前工作区没有可拉入成员，页面直接提示：

- 先去成员页创建成员
- 提供跳转按钮返回成员页

## 8. 成员工作流设计

### 8.1 成员列表

左侧成员视图展示工作区级成员列表。每个列表项至少显示：

- 成员名称
- 运行状态
- 模板名称或模板 ID 摘要

列表标题右侧提供 `新建成员` 按钮。

### 8.2 新建成员

点击 `新建成员` 后，中间区进入成员编辑页。主路径是：

1. 填写成员基础信息
2. 选择已有模板，或进入模板管理页新建模板
3. 保存成员

新成员创建完成后：

- 默认不在任何频道中
- 返回成员详情页
- 等待后续被频道手动拉入

### 8.3 成员详情页

成员详情页建议拆成四个区块或页签：

- `概况`
- `模板`
- `权限`
- `定时任务`

职责划分：

- `概况`：名称、状态、角色、最近活动摘要
- `模板`：当前模板、切换模板、进入模板管理
- `权限`：星球范围、命令类别、可私聊对象、可调度对象、管理权限位
- `定时任务`：该成员拥有的任务列表、新建任务、编辑任务、启停任务

### 8.4 发起私聊

成员详情页头部提供 `发起私聊` 按钮：

- 若与该成员的私聊已存在，则直接跳转到对应私聊会话
- 若不存在，则先创建私聊，再跳转

这样左侧的私聊列表只负责显示已有会话，不负责承担“找人发起会话”的入口职责。

## 9. 模板管理设计

模板不再作为主工作区的一级对象，而是成员体系下的二级资源。

模板入口只出现在两个地方：

1. 新建成员流程中的“选择模板 / 新建模板”
2. 成员详情页中的 `模板` 区块

模板管理视图位于中间内容区，建议采用：

- 左侧模板列表
- 右侧模板详情 / 编辑表单

模板创建成功后可直接回填到当前成员编辑流程。

这能同时解决两个问题：

- 模板入口足够可发现
- 模板不会重新挤占聊天主界面

## 10. 定时任务设计

### 10.1 归属关系

定时任务归属成员，而不是归属频道。

因此任务模型需要新增成员归属字段：

```ts
interface ScheduleJob {
  id: string;
  workspaceId: string;
  name: string;
  ownerAgentId: string;
  creatorType: 'player' | 'agent';
  creatorId: string;
  targetType: 'agent_dm' | 'conversation';
  targetId: string;
  intervalSeconds: number;
  messageTemplate: string;
  enabled: boolean;
  nextRunAt: string;
  lastRunAt?: string;
  createdAt: string;
  updatedAt: string;
}
```

`creator*` 继续表示是谁创建了任务，`ownerAgentId` 表示该任务属于哪个成员配置页。

### 10.2 交互

成员详情页的 `定时任务` 区块应支持：

- 查看该成员的任务列表
- 创建任务
- 修改投递目标、消息内容、间隔秒数
- 启停任务

创建任务时必须选择投递目标：

- 某个频道
- 该成员自己的私聊

任务编辑入口不应出现在频道聊天页或频道设置页。

## 11. 需要补齐的 API 与类型

### 11.1 Client-Web API 补充

`client-web/src/features/agents/api.ts` 需要新增至少以下能力：

- `fetchTemplates`
- `createTemplate`
- `createAgent`
- `addConversationMembers`
- `createAgentSchedule` 或在 `createSchedule` 中显式携带 `ownerAgentId`

`client-web/src/features/agents/types.ts` 也需要补充：

- `AgentTemplateView`
- 更完整的 `ScheduleView`
- 成员详情与模板列表渲染所需字段

### 11.2 Agent-Gateway API 补充

需要新增：

```text
POST /conversations/:id/members
```

请求体建议为：

```ts
{
  actorType: 'player' | 'agent';
  actorId: string;
  memberIds: string[];
}
```

行为：

- 只将明确给定的现有成员加入频道
- 需要遵守 `canManageMembers` 等权限限制
- 自动去重

### 11.3 Schedule API 调整

`POST /schedules` 请求体需要新增：

```ts
{
  ownerAgentId: string;
}
```

`GET /schedules` 返回体也应包含该字段，便于前端在成员页筛选任务。

同时需要新增：

```text
PATCH /schedules/:id
```

用于支持以下成员级任务操作：

- 修改间隔秒数
- 修改消息内容
- 修改投递目标
- 启用 / 停用任务

## 12. 前端组件拆分建议

建议至少拆分出以下组件：

- `AgentsWorkspaceShell`
- `AgentsSidebar`
- `ChannelListPane`
- `MemberListPane`
- `ChannelChatView`
- `ChannelSettingsView`
- `MemberDetailView`
- `MemberEditorView`
- `TemplateManagerView`
- `MemberScheduleEditor`

拆分原则：

- 列表组件只负责导航与入口按钮
- 聊天组件只负责消息流和发送
- 频道设置只负责频道级管理
- 成员详情只负责成员级配置
- 模板和任务各自拥有独立子视图

## 13. 空状态与错误状态

必须显式设计以下空状态：

- 没有频道：提示创建第一个频道
- 没有成员：提示创建第一个成员
- 频道还没有可加入成员：提示先去成员页创建成员
- 成员尚未绑定模板：提示选择模板
- 成员没有定时任务：提示创建任务

错误处理上，页面应尽量把错误限制在当前视图：

- 拉人失败只在频道设置页报错
- 创建成员失败只在成员编辑页报错
- 创建模板失败只在模板管理页报错

避免顶层页面因为单个子操作失败而整体退回错误页。

## 14. 测试策略

本轮至少覆盖以下验证：

### 14.1 单元 / 组件测试

- 频道 / 成员一级入口切换
- 频道聊天页与频道设置页切换
- 成员列表到成员详情页切换
- 新建成员流程与模板回填
- 从成员详情页发起私聊并跳转
- 频道设置里的精确拉人
- 成员详情里的定时任务列表与创建

### 14.2 集成测试

- 顶层数据加载后默认选中第一项频道或成员的逻辑
- 创建频道后自动跳入聊天页
- 创建成员后返回成员详情页
- 频道设置拉人后成员列表刷新
- 成员任务创建后在详情页即时显示

### 14.3 浏览器验证

需要实际打开 `client-web` 页面检查：

- 左侧列表滚动和选中态是否清晰
- 中间视图切换是否符合“频道 / 成员”上下文
- 创建频道、新建成员、模板管理、拉人、定时任务入口是否可发现
- 页面在窄屏下不会把主要操作挤出视口

## 15. 决策摘要

为避免后续实现中再次回到“所有功能堆首页”的旧结构，本设计明确固定以下决策：

- `/agents` 保持单页面，但采用双栏 + 内页式管理
- 左侧只有 `频道` 和 `成员` 两个一级入口
- 模板退到成员体系下的二级入口
- 新成员创建后不自动进入任何频道
- 私聊从成员页发起
- 频道拉人在频道设置的成员区域完成
- 定时任务归属成员，并在成员配置中管理

这组决策共同服务于一个目标：让 `/agents` 看起来像一个真正的即时通信工具，而不是一个把所有智能体能力平铺在首页的控制台。
