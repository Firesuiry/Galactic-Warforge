# Client-Web Grand Strategy Redesign Design

## 1. 背景

当前 `client-web` 已经具备总览、银河、星系、行星、回放等页面，但整体仍然更像“接口观察后台”而不是“大战略观察/指挥端”：

- 行星页主界面被左右两侧大量常驻面板和底部时间线挤压，地图只占很小一部分。
- `overview / galaxy / system / planet` 四个主页面缺少统一的视觉语言与信息分级。
- 当前服务端默认行星尺寸仍是 `32x32`，与设计文档中“行星层一般为 `2000x2000`”的目标不一致。
- `planet` 与 `fog` 查询仍采用整张地图全量返回，这在 `2000x2000` 下不可持续。

本次修改不是单纯换皮，而是一次“世界尺度 + 读模型 + 客户端骨架”的联合重构。

## 2. 目标

### 2.1 产品目标

本次改版要同时达成以下目标：

1. 把 `client-web` 重构为大战略地图优先的观察/指挥端。
2. 把 `overview / galaxy / system / planet` 统一为同一套大战略界面语言。
3. 把服务端默认行星尺寸切换为真实的 `2000x2000`。
4. 用新的行星读模型支撑大地图，不再依赖整张地图全量返回。
5. 把事件时间线与告警统一为默认折叠的情报面板。

### 2.2 设计结论

本次方案的已确认关键决策如下：

- 全站 `client-web` 统一改版，不只重做行星页。
- 行星页默认骨架采用 `A2`：
  - 左侧窄工具栏
  - 中间大地图主视区
  - 右侧稳定指挥栏
  - 底部薄状态条
  - 时间线与告警默认折叠
- 小窗口策略采用 `W2`：
  - 固定骨架
  - 局部弹出窗
  - 不做自由拖拽悬浮桌面
- 服务端默认新生成行星尺寸直接改为 `2000x2000`。
- 不保留整张 `fog` / `terrain` 的全量返回接口形态。

## 3. 范围

### 3.1 范围内

- 修改 `server` 默认行星生成尺寸为 `2000x2000`。
- 重构行星相关读模型，支持摘要、视窗场景和详情检视。
- 重做 `client-web` 的共用导航骨架与 `overview / galaxy / system / planet` 四个主页面。
- 将事件时间线与告警合并为统一情报面板，并默认折叠。
- 将部分次级信息改为局部弹出窗，避免长期挤占主视区。
- 更新相关文档，至少包括：
  - `docs/服务端API.md`
  - `docs/client-web使用说明.md`
  - 任何因 API 或页面结构变化而失效的 Web 相关说明

### 3.2 范围外

- 不做可拖拽、可自由排布的真悬浮窗口系统。
- 不做高成本重美术资源生产。
- 不把 Web 客户端扩展成完整即时操作游戏；重点仍是观察、指挥、查看、下达已有命令。
- 不为了兼容旧读模型保留额外适配层。

## 4. 产品信息架构

### 4.1 全站共用骨架

所有主页面共享同一套四层结构：

1. 顶部战略状态条
2. 中央主视图
3. 左侧窄工具栏
4. 右侧停靠指挥栏

所有次级信息通过局部弹出层进入，不再默认常驻铺满页面。

### 4.2 顶部战略状态条

顶部状态条统一显示：

- 当前玩家
- 当前 tick 或时间推进状态
- 关键资源
- 电力供需
- 当前研究
- 当前活跃行星
- 全局导航入口：`总览 / 星图 / 回放`

约束：

- 保持单行优先
- 视觉厚度压薄
- 不堆大按钮和长说明

### 4.3 左侧窄工具栏

左侧栏只保留高频切换入口，不承载大量文本详情。

行星页建议固定入口：

- 视图模式
- 图层
- 筛选
- 命令
- 情报
- 更多窗口

银河页、星系页、总览页沿用同样的结构，只替换对应场景功能。

### 4.4 右侧停靠指挥栏

右侧栏是稳定工作区，但信息量受控。

行星页固定由三块组成：

1. 当前选中对象详情
2. 当前对象可执行命令
3. 当前局势摘要

约束：

- 默认只展示当前上下文最相关信息
- 深层明细通过局部弹出窗进入
- 右栏宽度固定，不随内容无限扩张

### 4.5 情报面板

事件时间线与告警统一合并为一个 `情报面板`。

默认行为：

- 默认折叠
- 可由左栏按钮或快捷入口打开
- 打开后以局部弹出窗或侧向抽屉显示
- 内部提供 `时间线` 与 `告警` 两个视图
- 显示未读数、关键等级、最近更新时间

约束：

- 默认不占底部整行
- 高危告警允许显示轻量提示，但不自动展开整个面板
- 关闭后主视图恢复完整优先级

## 5. 页面定义

### 5.1 Overview

定位：战役总控页，而不是传统 dashboard 卡片墙。

结构：

- 中间：当前战局摘要主板
- 左侧：前往星图、当前行星、回放、情报面板的快捷入口
- 右侧：玩家状态、研究、电力、物流、活跃行星摘要

原则：

- 先展示“下一步去哪处理问题”
- 摘要优先于明细
- 行动入口优先于原始列表

### 5.2 Galaxy

定位：全局战略地图。

结构：

- 中间：银河地图，恒星系节点是主视觉
- 左侧：筛选、发现状态、视角模式、情报入口
- 右侧：当前选中恒星系情报

交互：

- 点击恒星系仅更新右侧摘要
- 再次确认后进入星系页

### 5.3 System

定位：星系级中间层，用来选择目标行星。

结构：

- 中间：恒星、轨道、行星节点的主视图
- 左侧：视图模式、筛选、情报入口
- 右侧：当前选中行星基础状态与进入入口

原则：

- 重点是“选目标行星”
- 不再采用纯列表页思路

### 5.4 Planet

定位：主战区，是整套客户端的核心页面。

固定采用 `A2 + W2`：

- 左侧窄栏：模式、图层、命令、情报入口、局部弹窗入口
- 中间：大地图主视区
- 右侧：实体详情、命令卡、施工/物流/战况摘要
- 底部：薄状态条，只显示 hover、坐标、选中对象、当前层级
- 时间线/告警：统一进入默认折叠的情报面板

## 6. 视觉与交互层级

### 6.1 视觉方向

统一采用“大战略指挥界面”风格，参考 P 社大战略游戏的层级组织，但不过度追求复杂 HUD。

视觉原则：

- 地图是最高视觉层级
- 状态条和指挥栏是辅助层
- 次级信息默认收起
- 局部弹出窗服务于当前主视图，而不是抢走页面中心

### 6.2 层级定义

- `L0` 主视图层：银河、星系、行星地图或总览主板
- `L1` 战略状态层：顶部状态条、左栏、右栏、底部状态条
- `L2` 局部弹出层：情报、筛选、统计、帮助
- `L3` 阻断层：登录、确认、严重错误、回放确认

约束：

- `L2` 不允许长期霸占大面积主视图
- 调试信息不能默认高于地图优先级

### 6.3 行星页地图交互

行星页地图交互规则：

1. 进入页面先落在战略级默认镜头
2. 滚轮缩放在 `sector` 与 `tile` 层级间切换
3. 拖拽平移地图
4. 点击对象后右侧栏切换到对应详情
5. 双击或快捷操作可聚焦对象
6. 局部弹出窗开启时，地图仍可继续平移和缩放

缩放语义：

- 远景：看分区态势
- 中景：看建筑密度、资源区、施工热点
- 近景：看格子、建筑、单位、网络覆盖

## 7. 服务端大地图设计

### 7.1 默认世界尺寸

`server` 默认行星尺寸从当前实现中的 `32x32` 切换为 `2000x2000`。

该修改必须覆盖：

- 默认地图配置
- 世界生成
- 可见性缓存初始化
- 探索状态缓存
- 相关测试数据与断言

### 7.2 为什么不能保留整张地图全量返回

当前接口形态中：

- `GET /world/planets/{planet_id}` 直接返回整张 `terrain`
- `GET /world/planets/{planet_id}/fog` 直接返回整张 `visible / explored`

在 `2000x2000` 下，这种设计会导致：

- 单次响应体量过大
- 首屏与刷新延迟显著上升
- 前端渲染与内存压力过高
- 截图、测试与回放页面不稳定

因此本次设计明确规定：

- 不保留整张 `terrain` 的全量返回
- 不保留整张 `fog` 的全量返回
- 新版行星页必须改为镜头驱动的读模型

## 8. 行星读模型重构

### 8.1 总体原则

- 首屏只拿骨架和摘要
- 地图视图只拿当前视窗需要的数据
- 详情栏单独获取 inspector 数据
- 不新增兼容旧接口的适配层

### 8.2 新读模型拆分

#### A. Planet Summary

建议接口：`GET /world/planets/{planet_id}`

职责：

- 返回行星元信息
- 返回页面骨架所需摘要
- 不返回整张地图数据

返回字段应包含：

- `planet_id` / `name` / `kind` / `orbit` / `moons`
- `map_width` / `map_height`
- `tick`
- `environment`
- 摘要统计：
  - 建筑数
  - 单位数
  - 资源点数
  - 施工数
  - 威胁等级
  - 活跃网络数
- 当前可用图层标记

#### B. Planet Scene

建议接口：`GET /world/planets/{planet_id}/scene`

职责：

- 返回当前视窗渲染所需数据
- 成为地图主视区的唯一主要数据源

请求参数：

- `x`
- `y`
- `width`
- `height`
- `detail_level=tile|sector`
- `layers=terrain,fog,resources,buildings,units,construction,logistics,power,pipeline,threat`

固定规则：

- `tile` 模式单次最大裁剪范围为 `256x256`
- `sector` 模式按固定战略分区返回，不返回逐格二维数组
- 响应必须带 `tick` 和本次实际裁剪边界

`tile` 模式应返回：

- 裁剪后的 `terrain`
- 裁剪后的 `fog.visible / fog.explored`
- 视窗内资源点
- 视窗内建筑和单位简表
- 视窗内施工、物流、电网、管网、敌情覆盖层

`sector` 模式应返回：

- `sectors[]`
- 每个分区至少包含：
  - 已探索比例
  - 地形占比
  - 建筑密度
  - 资源密度
  - 供电状态
  - 施工活跃度
  - 敌情热度
  - 物流活跃度

#### C. Planet Inspector

建议接口：`GET /world/planets/{planet_id}/inspect`

职责：

- 为右侧详情栏提供稳定的结构化详情
- 让详情与地图场景解耦

请求参数：

- `entity_kind=building|unit|resource|sector`
- `entity_id` 或 `sector_id`

返回内容：

- 标题信息
- 位置
- 所属者
- 运行状态
- 存储、生产、战斗、施工等详细字段
- 关联网络或关联任务摘要
- 可执行命令所需的最小上下文

### 8.3 现有 runtime / networks / events 的位置

- `events/stream`、`events/snapshot`、`alerts` 继续保留，用于情报面板。
- `runtime` / `networks` 不再是地图首屏必拉接口。
- 如有必要，可保留 `runtime` / `networks` 作为调试或次级读模型，但不再主导行星页渲染链路。

## 9. 前端数据流

### 9.1 行星页加载顺序

行星页改为以下顺序：

1. 拉取 `Planet Summary`
2. 根据默认镜头拉取 `Planet Scene`
3. 用户缩放或平移时重新拉取 `Planet Scene`
4. 用户选中对象后拉取 `Planet Inspector`
5. 情报面板独立消费 `events + alerts`

### 9.2 前端结构重构原则

- 地图渲染状态、情报状态、详情状态分层管理
- 地图主视图不直接依赖完整详情对象
- 页面跳转与弹出窗状态采用统一的布局状态模型
- 移动端允许退化为纵向堆叠，但桌面端优先保证主视图面积

## 10. 实施切分

### 10.1 实施包 A：服务端大地图基础

目标：让默认 `2000x2000` 世界真实成立。

内容：

- 修改默认地图配置
- 校验世界生成、可见性、探索缓存
- 补足大地图下基础测试
- 去除不适合大地图的全量返回逻辑

### 10.2 实施包 B：行星读模型重构

目标：让行星页成为镜头驱动页面。

内容：

- 重构行星相关 API
- 增加 `summary / scene / inspect` 型读模型
- 更新 `docs/服务端API.md`

### 10.3 实施包 C：client-web 全站骨架重做

目标：统一 `overview / galaxy / system / planet` 的大战略界面语言。

内容：

- 重做顶栏、左栏、右栏、底部状态条
- 银河页改为全局战略地图页
- 星系页改为轨道与目标选择页
- 总览页改为战役总控页
- 行星页落成 `A2 + W2`

### 10.4 实施包 D：情报与验证收尾

目标：完成情报面板、测试、截图与文档收尾。

内容：

- 合并事件与告警为统一情报面板
- 更新页面测试、视觉测试、fixtures 与截图基线
- 浏览器实测关键显示与交互
- 更新 Web 使用文档

## 11. 风险

### 11.1 数据量风险

最大风险是大地图数据量，而不是样式复杂度。

如果保留整张 `terrain / fog` 全量返回，接口体量、渲染成本和测试稳定性都会先出问题。

### 11.2 测试与 fixtures 失真风险

当前 fixtures、storybook、页面测试和视觉基线大量建立在小地图假设上，需要同步调整，否则测试不能真实反映大地图场景。

### 11.3 伪大战略风险

如果只改配色和边框，不重构信息层级，最终结果仍然会是“后台页外观换皮”，达不到地图主导目标。

### 11.4 过度窗口化风险

第一版如果过度追求复杂窗口系统，会把工作量消耗在窗口管理，而不是大地图与主交互闭环。

本次通过选定 `W2` 明确排除该方向。

## 12. 验收标准

本次完成的判定标准如下：

1. `server` 默认新生成行星尺寸为 `2000x2000`。
2. 行星页不再依赖整张地图全量读取。
3. 整张 `fog` 与整张 `terrain` 的全量返回接口不再保留。
4. `overview / galaxy / system / planet` 四页统一为大战略骨架。
5. 行星页默认实现为 `A2 + W2`。
6. 时间线与告警合并为默认折叠的情报面板。
7. 桌面端主界面明显以地图为中心。
8. 相关服务端测试、前端测试、浏览器实测通过。
9. 相关文档完成同步更新。
