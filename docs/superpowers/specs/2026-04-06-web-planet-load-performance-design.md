# 星球页加载性能优化设计

## 目标

优化 `client-web` 星球页在大地图和低缩放场景下的加载与交互卡顿，允许牺牲远距离细节来换取更稳定的首屏和拖拽/悬停流畅度。

## 现状与根因

当前卡顿主要来自 `PlanetMapCanvas` 的单画布全量重绘：

1. 地形、迷雾、网格在低缩放时仍按大量格子逐个绘制。
2. 鼠标悬停会频繁更新 store，导致整张画布和侧栏一起重渲染。
3. 建筑、单位、资源等实体图层虽然已有视窗裁剪，但仍在每次重绘时遍历全部实体列表。

这意味着“格子太多”本身确实会造成卡顿，但真正的问题是“可见格子密度高时仍保留细粒度绘制”以及“轻交互触发高成本重绘”。

## 方案对比

### 方案 A：分层缓存 + 低缩放降细节

将静态高成本图层与动态轻量图层拆开；在低缩放时主动弱化网格、地形细绘制和逐格迷雾；悬停更新走 `requestAnimationFrame` 节流。

优点：
- 性能收益最大
- 保留现有交互模型和 API
- 改动集中在 `planet-map` 特性内部

缺点：
- `PlanetMapCanvas` 结构要明显重构

### 方案 B：仅做显示裁剪

只在小 tile 时关闭网格、标签和部分图层，不改绘制架构。

优点：
- 改动小

缺点：
- 无法根治 hover/拖拽重绘成本

### 方案 C：迁移到 Pixi/WebGL

优点：
- 长期渲染上限更高

缺点：
- 重构范围过大，不适合这次任务

结论：采用方案 A。

## 设计

### 1. 绘制分层

把星球地图拆成三个职责层：

- `base`：地形、概览热图这类静态背景
- `detail`：资源、建筑、单位、网络、施工、敌情、迷雾
- `overlay`：选中框、hover 框、状态文本

其中 `base` 和低缩放下的部分 `detail` 应支持离屏缓存，仅在依赖输入变化时重算，不在每次 hover 时重复绘制。

### 2. 低缩放降细节

当 `tileSize` 低于阈值时主动降级：

- 极小 tile 时关闭 scene 模式网格线
- 小 tile 时不再逐建筑绘制边框/名称，只保留简化块状标记
- 小 tile 时迷雾从逐格绘制切为块状或直接按较粗粒度覆盖
- 仅当像素尺寸足够时绘制文本标签

目标不是保持每个像素都与高缩放一致，而是让玩家在远景下还能识别分布和重点。

### 3. hover 节流

鼠标移动不再直接把每次 `pointermove` 都写入 store，而是：

- 在组件内保存待处理 hover 坐标
- 用 `requestAnimationFrame` 合并一帧内的多次移动
- 仅当 tile 真的变化时再更新 store

这样可以减少 React 和 canvas 重绘频率。

### 4. 实体可见性裁剪

把建筑、单位、资源及运行时实体按视窗边界提前过滤，绘制阶段只遍历可见集合。建筑过滤需要考虑 footprint，避免大建筑边缘被截断。

### 5. 验证方式

- 单元/页面测试覆盖低缩放时的降细节行为与 hover 节流
- 运行 `client-web` 测试
- 启动浏览器查看星球页，确认地图、建筑、兵力、局势信息仍可显示且交互正常

## 风险

- 分层缓存若依赖集不完整，可能出现图层未刷新
- 过度降细节可能影响可读性，需要把阈值压在低缩放场景
- hover 节流若处理不当，会让状态栏坐标延迟或丢失

## 不做的事

- 不改 server API
- 不切换到新渲染引擎
- 不做无关页面的视觉重做
