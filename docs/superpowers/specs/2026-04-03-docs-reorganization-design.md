# docs 文档体系重组设计

## 1. 背景

当前 `docs/` 存在三个问题：

- 根目录堆积了大量并列文档，读者第一次进入时无法快速判断主入口。
- 面向开发者、面向玩家、面向流程自动化、面向历史归档的内容混放在一起。
- 多份文档在不同层面重复描述同一主题，例如 CLI、玩法、架构、现状分析与 DSP 差异。

本次重组的目标不是只改目录名，而是让“当前有效文档”与“历史资料”分层清晰，并减少重复主文档。

## 2. 目标

### 2.1 面向读者

- 开发者进入 `docs/` 后，应能快速找到当前有效的架构/API/CLI/Web/现状文档。
- 玩家或测试者进入 `docs/` 后，应能快速找到上手方式、玩法主线、已知问题与回归记录。

### 2.2 面向维护

- `docs/` 根目录只保留少量入口级目录，不再堆放大量主题文档。
- 重复主题只保留一份当前主文档，旧版本改为归档。
- 需要被脚本消费的流程文档集中到 `docs/process/`，避免继续和项目主文档混放。

## 3. 新的信息架构

### 3.1 顶层目录

- `docs/README.md`：总入口，先按读者类型分流。
- `docs/dev/`：当前有效的开发者文档。
- `docs/player/`：当前有效的玩家/测试文档。
- `docs/process/`：任务流转、提示词、规则、历史任务记录等流程文档。
- `docs/archive/`：历史设计、旧调研、参考资料、旧方案。
- `docs/superpowers/`：保留现状，作为 agent 工作产物目录，不纳入正式主文档线。
- `docs/remotion/`：保留现状，作为独立多媒体工程目录，不纳入正式主文档线。

### 3.2 当前主文档

开发者主线：

- `docs/dev/项目概览.md`
- `docs/dev/服务端API.md`
- `docs/dev/客户端CLI.md`
- `docs/dev/client-web.md`
- `docs/dev/agent-gateway.md`
- `docs/dev/server现状与差距.md`

玩家主线：

- `docs/player/玩法指南.md`
- `docs/player/上手与验证.md`
- `docs/player/已知问题与回归.md`

### 3.3 归档策略

- 旧架构稿、旧技术方案、旧现状报告、DSP 对标资料进入 `docs/archive/`。
- 原 `design/` 进入 `docs/archive/design/`。
- 原 `detail/` 与 `finished_task/` 进入 `docs/process/`，因为它们属于任务执行流程产物。

## 4. 内容合并策略

### 4.1 开发者文档

- `设定.md` + `服务端架构与功能设计.md` + `server与戴森球计划的一些设计区别.md`
  合并为 `docs/dev/项目概览.md`，只保留当前项目定位、系统边界、核心设计原则和与 DSP 的关键差异。
- `server现状详尽分析报告.md` + `server_dsp_gap_analysis.md`
  合并为 `docs/dev/server现状与差距.md`，以“当前已实现什么、仍缺什么、与 DSP 的核心差异是什么”为主轴。
- `client-web使用说明.md` + `client-web可视化客户端技术方案.md`
  合并为 `docs/dev/client-web.md`，保留当前目标、架构要点、启动方式、调试与回归入口。
- `cli.md` 保留为当前 CLI 参考文档，迁移并更名为 `客户端CLI.md`。
- `agent-gateway.md` 直接迁移为当前开发者文档。

### 4.2 玩家文档

- `玩家玩法指南.md` 保留为主玩法文档，迁移后改名为 `玩法指南.md`。
- `CLI游戏指南.md` 的启动和操作入口信息，和 `client-web使用说明.md` 中与试玩相关的启动方式，合并成 `docs/player/上手与验证.md`。
- `游玩发现的问题.md` 迁移后改名为 `docs/player/已知问题与回归.md`，保留问题和回归记录属性。

## 5. 链接与兼容策略

- 不保留兼容性占位文档，不额外放 wrapper 文件。
- 直接修改仓库内所有可见引用到新路径。
- 对仍然需要保留历史上下文的内容，在新主文档中添加“归档文档”索引。

## 6. 验证方式

- 目录验证：确认根目录只剩入口型目录与特殊工程目录。
- 引用验证：扫描仓库内是否仍残留旧主文档路径引用。
- 内容验证：检查 `docs/README.md`、`docs/dev/README.md`、`docs/player/README.md` 是否能形成清晰导航闭环。
